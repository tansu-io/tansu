## Phase 11 goal

Phase 11 is complete when Tansu can run **multiple brokers**, each with its own local replica log, and safely replicate Kafka partitions using Kafka-compatible leader/follower semantics:

1. Producers write only to the partition leader.
2. Followers pull from the leader using replica fetch.
3. The leader tracks ISR membership and follower progress.
4. The leader advances high watermark only when ISR replicas have caught up.
5. Clients only read committed data up to the high watermark.
6. Leader epochs are persisted and used to detect stale leaders and truncate divergent logs after leadership changes.
7. Failover or manually forced leader change does not corrupt offsets or expose uncommitted records.

This is a major change for Tansu because the current repo still describes the broker as **stateless** with pluggable storage, while Phase 11 requires each broker to own replica-local state and coordinate per-partition roles. ([GitHub][1])

The relevant Kafka behavior to mirror is: each partition has one leader and zero or more followers; writes go to the leader; followers pull from the leader; the leader maintains ISR; a record becomes committed when replicated to all ISR members; the leader advances high watermark and followers learn it through fetch responses. On leader change, followers truncate to the appropriate offset before fetching from the new leader. ([Apache Wiki][2]) Leader epochs are the durable mechanism that makes truncation deterministic: brokers persist `(leader_epoch, start_offset)` and use `OffsetForLeaderEpoch` to decide where a follower should truncate. ([Apache Wiki][3])

# 1. Current Tansu insertion points

Tansu already has the right broad layering:

* `tansu-broker` owns broker startup, TCP accept loop, request routing, cancellation, maintenance, and broker registration. `Broker::serve()` calls `register()` before `listen()`, and `register()` calls `storage.register_broker(...)`. 
* `tansu-broker/src/service/storage.rs` maps Kafka API keys to storage-backed services such as Produce, Fetch, Metadata, ListOffsets, and CreateTopics. 
* `tansu-storage::Storage` is the central abstraction. It already exposes `produce`, `fetch`, `offset_stage`, `metadata`, `brokers`, `maintain`, `node`, and `advertised_listener`. ([GitHub][4])
* `ProduceService` currently calls `Storage::produce(...)` but ignores meaningful replication semantics such as partition leader validation, `acks`, ISR, and `current_leader`. 
* `FetchService` already returns `high_watermark`, `last_stable_offset`, `log_start_offset`, `diverging_epoch`, and `current_leader`, but it does not yet distinguish client fetch from follower fetch. 
* `OffsetForLeaderEpochRequest.json` and `OffsetForLeaderEpochResponse.json` already exist in `tansu-sans-io`, but no storage service/route appears to be wired for it yet.  
* Tansu’s generated `ErrorCode` mapping already includes the errors Phase 11 needs, including `NotLeaderOrFollower`, `NotEnoughReplicas`, `NotEnoughReplicasAfterAppend`, `FencedLeaderEpoch`, and `UnknownLeaderEpoch`. 

The biggest current gap is that storage engines look single-broker or shared-log oriented. For example, `DynoStore::register_broker` currently returns `Ok(())`, `brokers()` returns only the local broker, and topic metadata stores the original `CreatableTopic` rather than durable per-partition replica assignments, leader, ISR, and epochs.

# 2. Definition of “done” for Phase 11

Phase 11 should not be considered finished until the following are true.

## Metadata correctness

For every topic partition, storage can answer:

```rust
topic
partition
replicas: Vec<i32>
leader_id: i32
leader_epoch: i32
isr: Vec<i32>
adding_replicas: Vec<i32>
removing_replicas: Vec<i32>
offline_replicas: Vec<i32>
```

Kafka clients should see correct `leader_id`, `leader_epoch`, `replicas`, and `isr` in `Metadata`, `DescribeTopicPartitions`, and topic description flows.

## Local replica isolation

Each broker must have its own replica log. This is crucial. For Phase 11, shared object-store or shared SQL tables must be namespaced by broker/node ID.

Bad:

```text
clusters/{cluster}/topics/{topic}/partitions/{partition}/records/{offset}.batch
```

Good:

```text
clusters/{cluster}/brokers/{broker_id}/topics/{topic}/partitions/{partition}/records/{offset}.batch
```

For SQL engines, records must similarly include `broker_id` or `replica_id`.

## Produce behavior

A produce request to a follower must return `NotLeaderOrFollower` with `current_leader` populated when the response version supports it.

For `acks`:

|       `acks` | Behavior                                                                                                  |
| -----------: | --------------------------------------------------------------------------------------------------------- |
|          `0` | Append locally on leader and do not wait for response semantics beyond protocol requirements.             |
|          `1` | Success after leader local append.                                                                        |
| `-1` / `all` | Success only after ISR has replicated through the appended offset and `min.insync.replicas` is satisfied. |
|        other | `InvalidRequiredAcks`.                                                                                    |

If ISR size is below `min.insync.replicas` before append, return `NotEnoughReplicas`. If append succeeds locally but replication does not complete before timeout, return `NotEnoughReplicasAfterAppend`.

## Fetch behavior

Client fetch:

* Served by leader only for now.
* Returns records only up to high watermark.
* Returns `NotLeaderOrFollower` when sent to a follower unless preferred-read-replica semantics are later enabled.

Follower fetch:

* Uses `FetchRequest.replica_id >= 0`.
* May read up to leader log end offset, not just high watermark.
* Includes the leader high watermark in response.
* Updates follower progress on the leader.

## ISR behavior

Leader tracks for each follower:

```rust
broker_id
log_end_offset
last_fetch_at
last_caught_up_at
is_in_isr
```

Followers leave ISR when they exceed `replica.lag.time.max.ms` or fail broker liveness checks. Followers rejoin ISR only after catching up to the leader’s current log end offset.

## High watermark behavior

Use **next-offset semantics**. In Tansu, `Watermark.high` already behaves like the next append offset: produce starts at `high.unwrap_or_default()` and then increments by record count. Keep that convention.

Leader high watermark:

```rust
leader_hw = min(local_leader_leo, leo_of_each_isr_replica)
```

High watermark must be monotonic within a leader epoch.

## Leader epoch behavior

Each replica persists:

```rust
Vec<LeaderEpochRecord> = [
    { epoch: 0, start_offset: 0 },
    { epoch: 1, start_offset: 123 },
    ...
]
```

When a broker becomes leader, it must append a new leader epoch record before accepting writes.

When a broker becomes follower, it must use `OffsetForLeaderEpoch` against the new leader to determine the truncation point before fetching.

## Truncation behavior

Follower truncation must remove:

* records at offsets `>= truncate_to`
* index entries at offsets `>= truncate_to`
* producer sequence state derived from truncated records
* leader epoch entries whose `start_offset >= truncate_to`
* watermark state above `truncate_to`

## Restart behavior

After restart:

* broker re-registers
* local replica logs are loaded
* leader epochs are loaded
* follower fetch loops resume
* leaders rebuild or reload follower progress
* high watermark remains safe and never jumps past replicated offsets

```rust
use std::collections::{BTreeMap, BTreeSet};

pub fn high_watermark_from_isr(
    leader_id: i32,
    leader_leo: i64,
    isr: &BTreeSet<i32>,
    follower_leos: &BTreeMap<i32, i64>,
) -> i64 {
    let mut hw = leader_leo;

    for broker_id in isr {
        if *broker_id == leader_id {
            continue;
        }

        match follower_leos.get(broker_id) {
            Some(leo) => hw = hw.min(*leo),
            None => hw = 0,
        }
    }

    hw
}

            high_watermark_from_isr(1, 12, &isr, &follower_leos)
```

Create `tansu-storage/src/replication/epoch.rs`.

```rust
use crate::replication::{LeaderEpoch, LeaderEpochRecord, Offset};

#[derive(Clone, Copy, Debug, Eq, PartialEq)]
pub enum EpochLookup {
    EndOffset(Offset),
    UnknownLeaderEpoch,
    FencedLeaderEpoch,
}

/// Return the end offset for `requested_epoch`.
///
/// Kafka semantics:
/// - If the requested epoch is older than the earliest known epoch, the caller is stale.
/// - If the epoch is known and there is a later epoch, return the next epoch's start offset.
/// - If the epoch is the latest known epoch, return current LEO.
pub fn end_offset_for_epoch(
    history: &[LeaderEpochRecord],
    current_leader_epoch: LeaderEpoch,
    requested_epoch: LeaderEpoch,
    log_end_offset: Offset,
) -> EpochLookup {
    if requested_epoch > current_leader_epoch {
        return EpochLookup::UnknownLeaderEpoch;
    }

    let Some(first) = history.first() else {
        return if requested_epoch == current_leader_epoch {
            EpochLookup::EndOffset(log_end_offset)
        } else {
            EpochLookup::UnknownLeaderEpoch
        };
    };

    if requested_epoch < first.epoch {
        return EpochLookup::FencedLeaderEpoch;
    }

    for window in history.windows(2) {
        let current = &window[0];
        let next = &window[1];

        if current.epoch == requested_epoch {
            return EpochLookup::EndOffset(next.start_offset);
        }
    }

    match history.last() {
        Some(last) if last.epoch == requested_epoch => EpochLookup::EndOffset(log_end_offset),
        _ => EpochLookup::UnknownLeaderEpoch,
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn next_epoch_start_is_end_offset() {
        let history = vec![
            LeaderEpochRecord { epoch: 0, start_offset: 0 },
            LeaderEpochRecord { epoch: 1, start_offset: 10 },
            LeaderEpochRecord { epoch: 2, start_offset: 20 },
        ];

        assert_eq!(
            EpochLookup::EndOffset(20),
            end_offset_for_epoch(&history, 2, 1, 25)
        );
    }

    #[test]
    fn latest_epoch_uses_leo() {
        let history = vec![
            LeaderEpochRecord { epoch: 0, start_offset: 0 },
            LeaderEpochRecord { epoch: 1, start_offset: 10 },
        ];

        assert_eq!(
            EpochLookup::EndOffset(15),
            end_offset_for_epoch(&history, 1, 1, 15)
        );
    }
}
```

# 6. Wire `OffsetForLeaderEpoch`

The protocol descriptors already exist, so the work is service wiring.

## Create `tansu-storage/src/service/offset_for_leader_epoch.rs`

The exact generated nested type names should be confirmed from `tansu-sans-io`, but the shape should be:

```rust
use rama::{Context, Service};
use tansu_sans_io::{
    ApiKey, ErrorCode, OffsetForLeaderEpochRequest, OffsetForLeaderEpochResponse,
    offset_for_leader_epoch_response::{
        OffsetForLeaderEpochResponsePartition,
        OffsetForLeaderEpochResponseTopic,
    },
};

use crate::{Error, Result, Storage, Topition};

#[derive(Clone, Copy, Debug, Default, Eq, Hash, Ord, PartialEq, PartialOrd)]
pub struct OffsetForLeaderEpochService;

impl ApiKey for OffsetForLeaderEpochService {
    const KEY: i16 = OffsetForLeaderEpochRequest::KEY;
}

impl<G> Service<G, OffsetForLeaderEpochRequest> for OffsetForLeaderEpochService
where
    G: Storage,
{
    type Response = OffsetForLeaderEpochResponse;
    type Error = Error;

    async fn serve(
        &self,
        ctx: Context<G>,
        req: OffsetForLeaderEpochRequest,
    ) -> Result<Self::Response, Self::Error> {
        let mut topics_out = vec![];

        for topic in req.topics.unwrap_or_default() {
            let mut partitions_out = vec![];

            for partition in topic.partitions.unwrap_or_default() {
                let tp = Topition::new(topic.topic.clone(), partition.partition);

                let outcome = ctx
                    .state()
                    .offset_for_leader_epoch(
                        &tp,
                        partition.current_leader_epoch,
                        partition.leader_epoch,
                    )
                    .await;

                let response_partition = match outcome {
                    Ok(outcome) => OffsetForLeaderEpochResponsePartition::default()
                        .partition(partition.partition)
                        .error_code(outcome.error_code.into())
                        .leader_epoch(outcome.leader_epoch)
                        .end_offset(outcome.end_offset),

                    Err(Error::Api(code)) => OffsetForLeaderEpochResponsePartition::default()
                        .partition(partition.partition)
                        .error_code(code.into())
                        .leader_epoch(-1)
                        .end_offset(-1),

                    Err(_) => OffsetForLeaderEpochResponsePartition::default()
                        .partition(partition.partition)
                        .error_code(ErrorCode::UnknownServerError.into())
                        .leader_epoch(-1)
                        .end_offset(-1),
                };

                partitions_out.push(response_partition);
            }

            topics_out.push(
                OffsetForLeaderEpochResponseTopic::default()
                    .topic(topic.topic)
                    .partitions(Some(partitions_out)),
            );
        }

        Ok(OffsetForLeaderEpochResponse::default()
            .throttle_time_ms(Some(0))
            .topics(Some(topics_out)))
    }
}
```

## `ReplicationManager`

Create `tansu-broker/src/replication/manager.rs`.

Responsibilities:

1. Load local broker ID.
2. Periodically list assigned replicas.
3. Start fetcher tasks for follower partitions.
4. Stop fetcher tasks when no longer follower.
5. On leader partitions, run ISR maintenance and high watermark advancement.
6. React to leadership changes.
7. Shut down cleanly on cancellation.

Skeleton:

```rust
use std::{
    collections::{BTreeMap, BTreeSet},
    time::{Duration, SystemTime},
};

use tansu_storage::{Storage, Topition};
use tokio::{task::JoinSet, time};
use tokio_util::sync::CancellationToken;
use tracing::{debug, error, instrument};

use super::fetcher::FollowerFetcher;

#[derive(Clone, Debug)]
pub struct ReplicationConfig {
    pub fetch_interval: Duration,
    pub replica_lag_time_max: Duration,
    pub fetch_min_bytes: u32,
    pub fetch_max_bytes: u32,
}

impl Default for ReplicationConfig {
    fn default() -> Self {
        Self {
            fetch_interval: Duration::from_millis(100),
            replica_lag_time_max: Duration::from_secs(30),
            fetch_min_bytes: 1,
            fetch_max_bytes: 1024 * 1024,

#[derive(Clone, Debug)]
pub struct ReplicationManager<S> {
    broker_id: i32,
    storage: S,
    config: ReplicationConfig,
    cancellation: CancellationToken,
}

impl<S> ReplicationManager<S>
where
    S: Storage + Clone + Send + Sync + 'static,
{
    pub fn new(
        broker_id: i32,
        storage: S,
        config: ReplicationConfig,
        cancellation: CancellationToken,
    ) -> Self {
        Self {
            broker_id,
            storage,
            config,
            cancellation,
        }
    }

    #[instrument(skip_all)]
    pub async fn run(self) {
        let mut set = JoinSet::new();
        let mut running_followers = BTreeSet::<Topition>::new();

        let mut tick = time::interval(Duration::from_millis(500));

        loop {
            tokio::select! {
                _ = self.cancellation.cancelled() => {
                    break;
                }

                _ = tick.tick() => {
                    match self.reconcile(&mut set, &mut running_followers).await {
                        Ok(()) => {}
                        Err(error) => error!(?error, "replication reconcile failed"),
                    }
                }

                Some(joined) = set.join_next(), if !set.is_empty() => {
                    debug!(?joined, "replication task exited");
                }
            }
        }

        set.abort_all();

        while set.join_next().await.is_some() {}
    }

    async fn reconcile(
        &self,
        set: &mut JoinSet<()>,
        running_followers: &mut BTreeSet<Topition>,
    ) -> tansu_storage::Result<()> {
        let assignments = self.local_replica_assignments().await?;

        for tp in assignments {
            let state = self.storage.partition_replica_state(&tp).await?;

            if state.local_role.is_follower() && !running_followers.contains(&tp) {
                let fetcher = FollowerFetcher::new(
                    self.broker_id,
                    tp.clone(),
                    self.storage.clone(),
                    self.config.clone(),
                    self.cancellation.clone(),
                );

                running_followers.insert(tp.clone());

                set.spawn(async move {
                    fetcher.run().await;
                });
            }

            if state.local_role.is_leader() {
                self.storage
                    .maybe_update_isr(&tp, SystemTime::now())
                    .await?;

                self.storage.advance_high_watermark(&tp).await?;
            }
        }

        Ok(())
    }

    async fn local_replica_assignments(&self) -> tansu_storage::Result<Vec<Topition>> {
        // Add this to Storage rather than deriving from Metadata response.
        // Metadata response is client-shaped and loses important local-role detail.
        todo!("Storage::local_replica_assignments")
    }
}
```

# 8. Follower fetcher

Create `tansu-broker/src/replication/fetcher.rs`.

The fetcher must:

1. Load current partition state.
2. Verify it is still follower.
3. Resolve leader endpoint.
4. Run epoch reconciliation before the first fetch and whenever fencing occurs.
5. Fetch from leader using replica fetch.
6. Append batches locally with original offsets.
7. Update local follower progress.
8. Update local high watermark from leader response.
9. Retry with backoff on transient network errors.

Skeleton:

```rust
use std::time::{Duration, SystemTime};

use tansu_storage::{
    replication::ReplicaRole,
    ReplicaAppendRequest, Storage, Topition,
};
use tokio::time;
use tokio_util::sync::CancellationToken;
use tracing::{debug, error, instrument};

use super::{client::ReplicationClient, manager::ReplicationConfig};

#[derive(Clone, Debug)]
pub struct FollowerFetcher<S> {
    broker_id: i32,
    topition: Topition,
    storage: S,
    config: ReplicationConfig,
    cancellation: CancellationToken,
}

impl<S> FollowerFetcher<S>
where
    S: Storage + Clone + Send + Sync + 'static,
{
    pub fn new(
        broker_id: i32,
        topition: Topition,
        storage: S,
        config: ReplicationConfig,
        cancellation: CancellationToken,
    ) -> Self {
        Self {
            broker_id,
            topition,
            storage,
            config,
            cancellation,
        }
    }

    #[instrument(skip_all, fields(topic = self.topition.topic(), partition = self.topition.partition()))]
    pub async fn run(self) {
        let mut tick = time::interval(self.config.fetch_interval);

        loop {
            tokio::select! {
                _ = self.cancellation.cancelled() => break,
                _ = tick.tick() => {
                    if let Err(error) = self.step().await {
                        error!(?error, "follower fetch step failed");
                    }
                }
            }
        }
    }

    async fn step(&self) -> tansu_storage::Result<()> {
        let state = self.storage.partition_replica_state(&self.topition).await?;

        if state.local_role != ReplicaRole::Follower {
            debug!(?state.local_role, "not follower anymore");
            return Ok(());
        }

        let local_stage = self.storage.offset_stage(&self.topition).await?;
        let local_leo = local_stage.high_watermark(); // later rename to log_end_offset

        let leader = self.resolve_leader(&state).await?;

        let client = ReplicationClient::connect(leader).await?;

        let response = client
            .fetch_from_leader(
                self.broker_id,
                &self.topition,
                local_leo,
                Some(state.leader_epoch),
                self.config.fetch_min_bytes,
                self.config.fetch_max_bytes,
            )
            .await?;

        for batch in response.batches {
            self.storage
                .append_as_follower(ReplicaAppendRequest {
                    source_broker_id: state.leader_id,
                    leader_epoch: state.leader_epoch,
                    topition: self.topition.clone(),
                    batch,
                })
                .await?;
        }

        self.storage
            .update_follower_progress(
                &self.topition,
                tansu_storage::replication::ReplicaProgress {
                    broker_id: self.broker_id,
                    log_end_offset: response.next_fetch_offset,
                    last_fetch_at: Some(SystemTime::now()),
                    last_caught_up_at: response
                        .caught_up
                        .then_some(SystemTime::now()),
                },
            )
            .await?;

        Ok(())
    }

    async fn resolve_leader(
        &self,
        state: &tansu_storage::replication::PartitionReplicaState,
    ) -> tansu_storage::Result<String> {
        // Use Storage::brokers or a new Storage::broker_endpoint.
        // Return "host:port" or Url.
        todo!("resolve leader broker endpoint")
    }
}
```

Important steering: do not use consumer fetch semantics here. Follower fetch must use `replica_id = broker_id`, and the leader must serve up to LEO, not only HW.

## Topic creation

When `replication_factor > 1`, assign replicas.

Rules:

1. If explicit `assignments` are provided, validate and persist them.
2. Else choose brokers round-robin from registered brokers.
3. If RF > registered broker count, return `InvalidReplicationFactor`.
4. For each partition:

   * `replicas = vec![...]`
   * `leader_id = replicas[0]`
   * `leader_epoch = 0`
   * `isr = replicas.clone() initially only if all replicas are caught up`
   * safer initial ISR: `{leader_id}` until followers fetch from offset 0
   * `leader_epochs = [{ epoch: 0, start_offset: 0 }]`

Initial ISR choice matters. For correctness, use `{leader_id}`. Followers join ISR after they have fetched through LEO.

# 14. ISR and high watermark algorithm

Create `tansu-storage/src/replication/isr.rs`.

```rust
use std::{
    collections::{BTreeMap, BTreeSet},
    time::{Duration, SystemTime},
};

use crate::replication::ReplicaProgress;

pub fn shrink_isr(
    leader_id: i32,
    current_isr: &BTreeSet<i32>,
    progress: &BTreeMap<i32, ReplicaProgress>,
    now: SystemTime,
    replica_lag_time_max: Duration,
) -> BTreeSet<i32> {
    current_isr
        .iter()
        .copied()
        .filter(|broker_id| {
            if *broker_id == leader_id {
                return true;
            }

            progress
                .get(broker_id)
                .and_then(|p| p.last_fetch_at)
                .and_then(|last_fetch| now.duration_since(last_fetch).ok())
                .is_some_and(|lag| lag <= replica_lag_time_max)
        })
        .collect()
}

pub fn expand_isr(
    leader_id: i32,
    replicas: &[i32],
    current_isr: &BTreeSet<i32>,
    progress: &BTreeMap<i32, ReplicaProgress>,
    leader_log_end_offset: i64,
) -> BTreeSet<i32> {
    let mut next = current_isr.clone();
    next.insert(leader_id);

    for broker_id in replicas {
        if *broker_id == leader_id {
            continue;
        }

        if progress
            .get(broker_id)
            .is_some_and(|p| p.log_end_offset >= leader_log_end_offset)
        {
            next.insert(*broker_id);
        }
    }

    next
}
```

High watermark advancement should be persisted with compare-and-swap or a SQL transaction. It must never decrease:

```rust
let computed = high_watermark_from_isr(...);
let next_hw = current_hw.max(computed);
```

# 15. Leader change and truncation

This is the most important safety area.

## Becoming leader

Storage method:

```rust
async fn become_leader(
    &self,
    topition: &Topition,
    new_epoch: i32,
) -> Result<PartitionReplicaState>
```

Implementation:

1. Lock partition metadata.
2. Verify local broker is in replicas.
3. Verify local broker is eligible to lead. For Phase 11, require local broker was in ISR to avoid unclean election.
4. Read local LEO.
5. Update partition state:

   * `leader_id = self.node`
   * `leader_epoch = new_epoch`
   * `isr = {self.node}`
6. Append leader epoch record:

   * `{ epoch: new_epoch, start_offset: local_leo }`
7. Persist metadata before accepting Produce.
8. Return new state.

## Becoming follower

Storage method:

```rust
async fn become_follower(
    &self,
    topition: &Topition,
    leader_id: i32,
    leader_epoch: i32,
) -> Result<PartitionReplicaState>
```

Implementation:

1. Stop local leader acceptance.
2. Load local latest epoch.
3. Ask leader with `OffsetForLeaderEpoch`.
4. Truncate to returned end offset.
5. Persist follower role.
6. Start fetcher.

Fetcher-side pseudo-code:

```rust
async fn reconcile_epochs(&self, client: &ReplicationClient) -> Result<()> {
    let local_epochs = self.storage.leader_epochs(&self.topition).await?;
    let Some(latest) = local_epochs.last() else {
        return Ok(());
    };

    let state = self.storage.partition_replica_state(&self.topition).await?;

    let truncate_to = client
        .offset_for_leader_epoch(
            self.broker_id,
            &self.topition,
            Some(state.leader_epoch),
            latest.epoch,
        )
        .await?;

    self.storage.truncate(&self.topition, truncate_to).await?;

    Ok(())
}
```

## Truncation details

For each storage engine:

```rust
async fn truncate(&self, topition: &Topition, offset: i64) -> Result<()> {
    // 1. Delete records whose base offset >= offset.
    // 2. If a batch starts before offset but extends past offset, either:
    //    a. split the batch, or
    //    b. truncate to the batch base offset.
    //
    // For Phase 11, choose b unless record-level slicing is already robust.
    //
    // 3. Update log_end_offset = offset.
    // 4. high_watermark = min(high_watermark, offset).
    // 5. last_stable_offset = min(last_stable_offset, offset).
    // 6. Remove leader epoch records with start_offset >= offset.
    // 7. Rebuild producer sequence state for affected partition.
    Ok(())
}
```

Steering: batch splitting is tricky because CRCs, record offsets, timestamps, and producer sequence state must remain valid. For Phase 11, truncating to whole-batch boundaries is safer, then tighten later if exact intra-batch truncation is required.

The most important architectural steering: **do not fake replication by sharing one global log across brokers**. For Phase 11 to be real, each broker needs its own replica log, its own leader epoch history, and its own truncation path. That one decision makes ISR, HW, follower fetch, and leader change tests meaningful rather than cosmetic.
