Absolutely — Phase 11 is the point where Tansu stops merely *presenting* Kafka metadata and starts behaving like a real replicated Kafka cluster. I would implement it as a focused replication slice with a clear boundary: **Phase 11 owns replica state, follower replication, ISR, high watermark, leader epochs, follower fetch, and truncation after an externally supplied leadership change. Phase 12 owns autonomous controller election and leadership decisions.**

Below is the plan I would use to finish Phase 11 end-to-end.

---

# 0. Current repo reality to anchor the plan

Tansu is currently described as a **stateless Kafka-compatible broker** with pluggable storage backends including PostgreSQL, libSQL, S3, and memory, so Phase 11 needs to preserve the shared-storage story while adding Kafka-compatible replicated-broker behavior. ([GitHub][1])

The current `Storage` abstraction is the main seam: it already owns broker registration, topic creation, produce, fetch, offset stage, metadata, maintenance, node ID, advertised listener, and similar broker-facing functions. That makes it the correct place to introduce partition leadership, ISR, high watermark, leader epochs, and follower append/truncate operations. ([GitHub][2])

The broker already calls `register_broker` at startup and periodically calls `storage.maintain(SystemTime::now())`, so Phase 11 can reuse those hooks for broker liveness, ISR aging, and background high-watermark maintenance. 

Two service files are especially important. `ProduceService` currently calls `ctx.state().produce(...)` directly and returns `current_leader(None)` even on normal responses, so it does not yet enforce “only the leader accepts produce.”  `FetchService` currently fetches from storage, then fills `high_watermark`, `last_stable_offset`, and `log_start_offset`, but normal responses set `diverging_epoch(None)` and `current_leader(None)`, so it does not yet expose Kafka follower-fetch or leader-epoch divergence semantics. 

The SQL schema currently has topic, topition, record, and watermark tables, but no durable broker table, replica assignment table, ISR table, leader epoch table, or follower progress table.    

The protocol pieces mostly exist in `tansu-sans-io`: `OffsetForLeaderEpochRequest` and response JSON definitions are present, and `FetchRequest` already contains replica ID, current leader epoch, last fetched epoch, follower log start offset, and replica state fields.    The broker storage routing table, however, currently registers fetch/metadata/produce/etc. but not an `OffsetForLeaderEpoch` service. 

That means Phase 11 should be implemented as: **metadata/schema first, storage trait extensions second, produce/fetch protocol semantics third, replication runtime fourth, and truncation/chaos tests last.**

# 1. Phase 11 definition of done

Phase 11 is complete only when all of the following are true:

| Area                 | Completion criterion                                                                                                                                               |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Partition metadata   | Every partition has durable `leader_id`, `leader_epoch`, `replicas`, `isr`, `offline_replicas`, `log_start_offset`, `leader log end offset`, and `high_watermark`. |
| Broker registry      | Each broker registers with broker ID, incarnation ID, listener, rack, heartbeat timestamp, and fenced/dead status.                                                 |
| Produce path         | Produce succeeds only on the partition leader. Produces to followers return `NotLeaderOrFollower` and include `current_leader`.                                    |
| `acks=all`           | Produce with `acks = -1` waits until the appended offset reaches high watermark or times out. It respects `min.insync.replicas`.                                   |
| Fetch path           | Client fetch reads only up to high watermark, except `read_committed` still obeys last stable offset. Follower fetch can read up to leader log end offset.         |
| Follower replication | A follower task fetches from the current leader, appends the exact batches locally, and reports replica progress.                                                  |
| ISR tracking         | Followers enter ISR only after catching up; they leave ISR when stale or lagging beyond thresholds. ISR state is durable.                                          |
| High watermark       | Leader advances HW monotonically to the minimum log end offset among ISR members.                                                                                  |
| Leader epochs        | Every leadership change creates a durable leader-epoch checkpoint with `epoch`, `leader_id`, and `start_offset`.                                                   |
| OffsetForLeaderEpoch | API key 23 is routed and returns correct epoch end offsets and leader-epoch fencing errors.                                                                        |
| Truncation           | A replica becoming follower discovers divergence and truncates to the leader’s epoch end offset before fetching.                                                   |
| Metadata             | `Metadata` and `DescribeTopicPartitions` return correct leader, leader epoch, replicas, ISR, and offline replicas.                                                 |
| Tests                | Unit, service, backend, and multi-broker integration tests cover normal replication, ISR shrink/expand, HW advancement, leader epoch lookup, and truncation.       |

# 2. Big architectural decision

I would add Phase 11 as two layers:

## Layer A: Replication metadata and primitive operations inside `tansu-storage`

This layer knows nothing about TCP loops. It provides durable facts and safe state transitions:

```rust
partition_state(topition)
register_broker(...)
broker_heartbeat(...)
assigned_topitions(broker_id)
append_as_leader(...)
append_as_follower(...)
truncate_to(...)
local_log_end_offset(...)
update_replica_progress(...)
maybe_expand_isr(...)
maybe_shrink_isr(...)
advance_high_watermark(...)
leader_epoch_history(...)
offset_for_leader_epoch(...)
become_leader(...)
become_follower(...)
```

## Layer B: Replication runtime inside a new `tansu-replication` crate

This layer runs on each broker. It watches assignments, starts follower tasks, performs `OffsetForLeaderEpoch`, sends follower `FetchRequest`s to leaders, appends fetched batches locally, updates progress, and reacts to leader changes.

I recommend a **new crate** instead of placing everything inside `tansu-broker`, because the runtime will have independent unit tests, metrics, Kafka client protocol calls, and retry/backoff logic.

```
tansu-replication/
  Cargo.toml
  src/lib.rs
  src/manager.rs
  src/follower.rs
  src/client.rs
  src/truncate.rs
  src/metrics.rs
  tests/cluster.rs
```

The broker should own lifecycle. Storage should own correctness. Replication runtime should own background work.

## Steering

Do **not** continue using “watermark high” as both append position and Kafka high watermark. In current storage, `Watermark.high` is effectively the next append offset/log end offset. In Kafka terms, that is **LEO**, not HW. Phase 11 should split these concepts:

```text
log_start_offset: earliest readable offset
leader_log_end_offset: next offset after leader’s last appended batch
replica_log_end_offset: next offset known for each follower
high_watermark: committed replicated offset visible to normal consumers
last_stable_offset: transaction-specific visibility boundary
```

If you keep `Watermark.high` as-is internally, rename it in code comments and APIs to `log_end_offset` and add a separate durable `high_watermark`.

# 7. Pure replication algorithms

## High watermark calculation

```rust
use std::collections::BTreeMap;

use crate::{BrokerId, Offset};

pub fn compute_high_watermark(
    leader_log_end_offset: Offset,
    isr: &[BrokerId],
    replica_log_end_offsets: &BTreeMap<BrokerId, Offset>,
) -> Offset {
    if isr.is_empty() {
        return 0;
    }

    let mut high_watermark = leader_log_end_offset;

    for broker_id in isr {
        let replica_leo = replica_log_end_offsets
            .get(broker_id)
            .copied()
            .unwrap_or(0);

        high_watermark = high_watermark.min(replica_leo);
    }

    high_watermark
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn high_watermark_is_minimum_log_end_offset_in_isr() {
        let progress = BTreeMap::from([
            (1, 10),
            (2, 7),
            (3, 9),
            (4, 1), // not ISR; ignored
        ]);

        assert_eq!(
            7,
            compute_high_watermark(10, &[1, 2, 3], &progress)
        );
    }

    #[test]
    fn single_leader_isr_high_watermark_is_leader_leo() {
        let progress = BTreeMap::from([(1, 42)]);

        assert_eq!(
            42,
            compute_high_watermark(42, &[1], &progress)
        );
    }
}
```

## OffsetForLeaderEpoch algorithm

Kafka’s `OffsetForLeaderEpoch` asks: “for epoch E, what is the end offset of that epoch?” Implement it over sorted checkpoints.

```rust
use crate::{EpochEndOffset, LeaderEpoch, LeaderEpochCheckpoint, Offset};
use tansu_sans_io::ErrorCode;

pub fn offset_for_leader_epoch_from_history(
    history: &[LeaderEpochCheckpoint],
    requested_epoch: LeaderEpoch,
    current_log_end_offset: Offset,
) -> EpochEndOffset {
    if history.is_empty() {
        return EpochEndOffset {
            error_code: ErrorCode::None,
            leader_epoch: -1,
            end_offset: current_log_end_offset,
        };
    }

    let Some(position) = history
        .iter()
        .position(|checkpoint| checkpoint.leader_epoch == requested_epoch)
    else {
        return EpochEndOffset {
            error_code: ErrorCode::UnknownLeaderEpoch,
            leader_epoch: requested_epoch,
            end_offset: -1,
        };
    };

    let end_offset = history
        .get(position + 1)
        .map(|next| next.start_offset)
        .unwrap_or(current_log_end_offset);

    EpochEndOffset {
        error_code: ErrorCode::None,
        leader_epoch: requested_epoch,
        end_offset,
    }
}
```

## Truncation decision

```rust
#[derive(Clone, Debug, Eq, PartialEq)]

pub fn decide_truncation(
```

### Follower append must be different from produce

Do **not** call `produce()` inside `append_as_follower`. Leader produce performs producer sequence validation, schema validation, transaction mutation, lake sink writes, and watermark allocation. Follower append must preserve the leader’s batch exactly and must not run side effects again.

```rust
async fn append_as_follower(
    &self,
    topition: &Topition,
    leader_epoch: LeaderEpoch,
    batches: Vec<deflated::Batch>,
) -> Result<Offset> {
    let state = self.partition_state(topition).await?;

    if state.leader_epoch != leader_epoch {
        return Err(Error::FencedLeaderEpoch {
            topition: topition.clone(),
            requested_epoch: leader_epoch,
            current_epoch: state.leader_epoch,
        });
    }

    let mut next_offset = self.local_log_end_offset(topition).await?;

    for batch in batches {
        if batch.base_offset != next_offset {
            return Err(Error::Message(format!(
                "follower append gap: expected offset {next_offset}, got {}",
                batch.base_offset
            )));
        }

        let location = replica_record_location(
            self.cluster.as_str(),
            topition,
            self.node,
            batch.base_offset,
        );

        let payload = self.encode(batch)?;
        self.object_store
            .put_opts(
                &location,
                payload,
                PutOptions {
                    mode: PutMode::Create,
                    attributes: Attributes::new(),
                    ..Default::default()
                },
            )
            .await?;

        next_offset += 1;
    }

    self.update_replica_progress(
        topition,
        self.node,
        state.log_start_offset,
        next_offset,
        SystemTime::now(),
    )
    .await?;

    Ok(next_offset)
}
```

Adjust offset advancement by `last_offset_delta + 1`, not hardcoded `+1`; this snippet is intentionally compact.

# 9. Metadata and topic creation changes

## Topic creation

When creating a topic with `replication_factor > 1`, assign replicas from registered brokers.

Algorithm:

```rust
fn assign_replicas(
    brokers: &[BrokerId],
    partition_count: i32,
    replication_factor: i16,
) -> Result<BTreeMap<i32, Vec<BrokerId>>> {
    let rf = usize::try_from(replication_factor)?;

    if brokers.len() < rf {
        return Err(Error::Api(ErrorCode::InvalidReplicationFactor));
    }

    let mut assignments = BTreeMap::new();

    for partition in 0..partition_count {
        let start = usize::try_from(partition)? % brokers.len();

        let replicas = (0..rf)
            .map(|i| brokers[(start + i) % brokers.len()])
            .collect::<Vec<_>>();

        assignments.insert(partition, replicas);
    }

    Ok(assignments)
}
```

Initial partition state:

```rust
PartitionState {
    leader_id: replicas[0],
    leader_epoch: 0,
    replicas,
    isr: vec![leader_id],
    offline_replicas: vec![],
    log_start_offset: 0,
    leader_log_end_offset: 0,
    high_watermark: 0,
}
```

## Steering

At creation time, do **not** put all replicas in ISR unless they actually have the leader’s log. For empty partitions, it is safe to put all live replicas in ISR if every replica starts at LEO 0. But if the topic is created while some brokers are unavailable, only the leader should start in ISR.

# 10. Produce path changes

## `acks=all` handling

Add helper:

```rust
async fn wait_for_high_watermark<G>(
    storage: &G,
    topition: &Topition,
    required_offset: i64,
    timeout: Duration,
) -> Result<()>
where
    G: Storage,
{
    let start = std::time::Instant::now();

    loop {
        let stage = storage.offset_stage(topition).await?;

        if stage.high_watermark() >= required_offset {
            return Ok(());
        }

        if start.elapsed() >= timeout {
            return Err(Error::NotEnoughReplicasAfterAppend {
                topition: topition.clone(),
                required_offset,
                high_watermark: stage.high_watermark(),
            });
        }

        tokio::time::sleep(Duration::from_millis(5)).await;
    }
}
```

## Steering

Kafka’s produce response `base_offset` is the base offset of the first batch appended. For `acks=all`, the append may have succeeded even if replication wait later times out. In that case Kafka uses `NOT_ENOUGH_REPLICAS_AFTER_APPEND`, not a silent rollback. Do not delete the batch on timeout.

Also, Phase 11 should finally make `acks` meaningful:

```text
acks = 0: append and do not wait; response may be suppressed at network layer if currently implemented.
acks = 1: return after leader append.
acks = -1/all: return after appended range reaches HW or timeout.
```

```rust
fn validate_fetch_epoch(
    topition: &Topition,
    supplied_epoch: Option<i32>,
    state: &PartitionState,

    match supplied_epoch {
        None | Some(-1) => Ok(()),

        Some(epoch) if epoch < state.leader_epoch => {

                topition: topition.clone(),

                topition: topition.clone(),
                requested_epoch: epoch,
                current_epoch: state.leader_epoch,
```

```rust
use rama::{Context, Service};
use tansu_sans_io::{
    ApiKey, ErrorCode, OffsetForLeaderEpochRequest, OffsetForLeaderEpochResponse,

use tracing::{error, instrument};

use crate::{Error, Result, Storage, Topition};

#[derive(Clone, Copy, Debug, Default, Eq, Hash, Ord, PartialEq, PartialOrd)]
pub struct OffsetForLeaderEpochService;

impl ApiKey for OffsetForLeaderEpochService {
    const KEY: i16 = OffsetForLeaderEpochRequest::KEY;
}

impl<G> Service<G, OffsetForLeaderEpochRequest> for OffsetForLeaderEpochService
where
    G: Storage,
{
    type Response = OffsetForLeaderEpochResponse;
    type Error = Error;

    #[instrument(skip(ctx, req))]
    async fn serve(
        &self,
        ctx: Context<G>,
        req: OffsetForLeaderEpochRequest,
    ) -> Result<Self::Response> {
        let mut topic_results = Vec::new();

        for topic in req.topics.unwrap_or_default() {
            let topic_name = topic.topic.clone();
            let mut partition_results = Vec::new();

            for partition in topic.partitions.unwrap_or_default() {
                let topition = Topition::new(topic_name.clone(), partition.partition);

                let result = ctx
                    .state()
                    .offset_for_leader_epoch(
                        &topition,
                        partition.current_leader_epoch,
                        partition.leader_epoch,
                    )
                    .await;

                let response_partition = match result {
                    Ok(epoch_end) => ResponseEpochEndOffset::default()
                        .partition(partition.partition)
                        .error_code(epoch_end.error_code.into())
                        .leader_epoch(Some(epoch_end.leader_epoch))
                        .end_offset(epoch_end.end_offset),

                    Err(err) => {
                        error!(?err, ?topition);

                        ResponseEpochEndOffset::default()
                            .partition(partition.partition)
                            .error_code(err.kafka_error_code().into())
                            .leader_epoch(Some(-1))
                            .end_offset(-1)
                    }
                };

                partition_results.push(response_partition);
            }

            topic_results.push(
                OffsetForLeaderTopicResult::default()
                    .topic(topic_name)
                    .partitions(Some(partition_results)),
            );
        }

        Ok(OffsetForLeaderEpochResponse::default()
            .throttle_time_ms(Some(0))
            .topics(Some(topic_results)))
    }
}
```

```rust
use tokio::{task::JoinSet, time};
use tokio_util::sync::CancellationToken;
use tracing::{debug, error, instrument};

use crate::follower::FollowerTask;

#[derive(Clone, Debug)]
pub struct ReplicationConfig {
    pub poll_interval: Duration,
    pub fetch_max_bytes: i32,
    pub fetch_min_bytes: i32,
    pub fetch_max_wait: Duration,
    pub replica_lag_timeout: Duration,
}

impl Default for ReplicationConfig {
    fn default() -> Self {
        Self {
            poll_interval: Duration::from_millis(500),
            fetch_max_bytes: 4 * 1024 * 1024,
            fetch_min_bytes: 1,
            fetch_max_wait: Duration::from_millis(500),
            replica_lag_timeout: Duration::from_secs(10),
        }
    }
}

#[derive(Clone, Debug)]
pub struct ReplicationManager<S> {
    broker_id: BrokerId,
    storage: S,
    config: ReplicationConfig,
    cancellation: CancellationToken,
}

impl<S> ReplicationManager<S>
where
    S: Storage + Clone + Send + Sync + 'static,
{
    pub fn new(
        broker_id: BrokerId,
        storage: S,
        config: ReplicationConfig,
        cancellation: CancellationToken,
    ) -> Self {
        Self {
            broker_id,
            storage,
            config,
            cancellation,
        }
    }

    #[instrument(skip_all)]
    pub async fn run(self) -> Result<()> {
        let mut tasks: BTreeMap<Topition, CancellationToken> = BTreeMap::new();
        let mut join_set = JoinSet::new();
        let mut interval = time::interval(self.config.poll_interval);

        loop {
            tokio::select! {
                _ = self.cancellation.cancelled() => {
                    for (_, token) in tasks {
                        token.cancel();
                    }
                    break;
                }

                _ = interval.tick() => {
                    self.storage
                        .broker_heartbeat(self.broker_id, std::time::SystemTime::now())
                        .await?;

                    let assignments = self.storage
                        .assigned_topitions(self.broker_id)
                        .await?;

                    self.reconcile(assignments, &mut tasks, &mut join_set).await?;
                }

                Some(joined) = join_set.join_next(), if !join_set.is_empty() => {
                    if let Err(err) = joined {
                        error!(?err, "replication task join error");
                    }
                }
            }
        }

        while join_set.join_next().await.is_some() {}

        Ok(())
    }

    async fn reconcile(
        &self,
        assignments: Vec<PartitionState>,
        tasks: &mut BTreeMap<Topition, CancellationToken>,
        join_set: &mut JoinSet<()>,
    ) -> Result<()> {
        for state in assignments {
            let should_follow =
                state.is_replica(self.broker_id) && !state.is_leader(self.broker_id);

            if should_follow && !tasks.contains_key(&state.topition) {
                let token = self.cancellation.child_token();

                tasks.insert(state.topition.clone(), token.clone());

                let task = FollowerTask::new(
                    self.broker_id,
                    state.topition.clone(),
                    self.storage.clone(),
                    self.config.clone(),
                    token,
                );

                join_set.spawn(async move {
                    if let Err(err) = task.run().await {
                        error!(?err, "follower replication task failed");
                    }
                });
            }

            if !should_follow {
                if let Some(token) = tasks.remove(&state.topition) {
                    token.cancel();
                }
            }
        }

        debug!(active_followers = tasks.len());
        Ok(())
    }
}
```

# 14. Follower task

## High-level loop

```text
for assigned follower partition:
  read partition state
  if local broker is no longer follower: exit
  find leader broker endpoint
  perform epoch reconciliation
  truncate if needed
  send FetchRequest with replica_id = local broker ID
  append returned batches as follower
  update replica progress
  sleep/backoff
```

## `src/follower.rs`

```rust
use std::time::{Duration, SystemTime};

use tansu_sans_io::{
    ErrorCode, FetchRequest, IsolationLevel,
    fetch_request::{FetchPartition, FetchTopic},
};
use tansu_storage::{BrokerId, Result, Storage, Topition};
use tokio::time;
use tokio_util::sync::CancellationToken;
use tracing::{debug, error, instrument};

use crate::{client::ReplicationClient, manager::ReplicationConfig, truncate};

#[derive(Clone, Debug)]
pub struct FollowerTask<S> {
    broker_id: BrokerId,
    topition: Topition,
    storage: S,
    config: ReplicationConfig,
    cancellation: CancellationToken,
}

impl<S> FollowerTask<S>
where
    S: Storage + Clone + Send + Sync + 'static,
{
    pub fn new(
        broker_id: BrokerId,
        topition: Topition,
        storage: S,
        config: ReplicationConfig,
        cancellation: CancellationToken,
    ) -> Self {
        Self {
            broker_id,
            topition,
            storage,
            config,
            cancellation,
        }
    }

    #[instrument(skip_all, fields(topic = %self.topition.topic(), partition = self.topition.partition()))]
    pub async fn run(self) -> Result<()> {
        loop {
            tokio::select! {
                _ = self.cancellation.cancelled() => break,
                result = self.replicate_once() => {
                    if let Err(err) = result {
                        error!(?err, "replication iteration failed");
                        time::sleep(Duration::from_millis(250)).await;
                    }
                }
            }
        }

        Ok(())
    }

    async fn replicate_once(&self) -> Result<()> {
        let state = self.storage.partition_state(&self.topition).await?;

        if state.is_leader(self.broker_id) || !state.is_replica(self.broker_id) {
            return Ok(());
        }

        truncate::reconcile_with_leader(
            &self.storage,
            &self.topition,
            state.leader_id,
            state.leader_epoch,
        )
        .await?;

        let local_leo = self.storage.local_log_end_offset(&self.topition).await?;
        let client = ReplicationClient::for_leader(&self.storage, state.leader_id).await?;

        let response = client
            .fetch(FetchRequest::default()
                .replica_id(Some(self.broker_id))
                .max_wait_ms(i32::try_from(self.config.fetch_max_wait.as_millis()).unwrap_or(500))
                .min_bytes(self.config.fetch_min_bytes)
                .max_bytes(Some(self.config.fetch_max_bytes))
                .isolation_level(Some(IsolationLevel::ReadUncommitted.into()))
                .topics(Some(vec![
                    FetchTopic::default()
                        .topic(Some(self.topition.topic().to_owned()))
                        .partitions(Some(vec![
                            FetchPartition::default()
                                .partition(self.topition.partition())
                                .current_leader_epoch(Some(state.leader_epoch))
                                .fetch_offset(local_leo)
                                .last_fetched_epoch(Some(state.leader_epoch))
                                .log_start_offset(Some(state.log_start_offset))
                                .partition_max_bytes(self.config.fetch_max_bytes),
                        ])),
                ])))
            .await?;

        let batches = response
            .responses
            .unwrap_or_default()
            .into_iter()
            .flat_map(|topic| topic.partitions.unwrap_or_default())
            .filter_map(|partition| {
                let error = ErrorCode::try_from(partition.error_code).ok()?;

                if error != ErrorCode::None {
                    debug!(?error, "follower fetch partition error");
                    return None;
                }

                partition.records.map(|frame| frame.batches)
            })
            .flatten()
            .collect::<Vec<_>>();

        if !batches.is_empty() {
            let new_leo = self
                .storage
                .append_as_follower(&self.topition, state.leader_epoch, batches)
                .await?;

            self.storage
                .update_replica_progress(
                    &self.topition,
                    self.broker_id,
                    state.log_start_offset,
                    new_leo,
                    SystemTime::now(),
                )
                .await?;
        }

        Ok(())
    }
}
```

## Steering

The follower task should **never** allocate offsets. It appends batches at the exact offsets assigned by the leader. If the next fetched batch’s `base_offset` does not match local LEO, the follower must run leader-epoch reconciliation and truncate or reset.

# 15. Truncation on leader change

Create `tansu-replication/src/truncate.rs`.

```rust
use tansu_storage::{BrokerId, LeaderEpoch, Result, Storage, Topition};
use tracing::{debug, instrument};

#[instrument(skip(storage))]
pub async fn reconcile_with_leader<S>(
    storage: &S,
    topition: &Topition,
    leader_id: BrokerId,
    leader_epoch: LeaderEpoch,
) -> Result<()>
where
    S: Storage,
{
    let _ = leader_id;

    let local_history = storage.leader_epoch_history(topition).await?;
    let local_leo = storage.local_log_end_offset(topition).await?;

    let Some(last_local_epoch) = local_history.last().map(|e| e.leader_epoch) else {
        return Ok(());
    };

    let leader_epoch_end = storage
        .offset_for_leader_epoch(topition, Some(leader_epoch), last_local_epoch)
        .await?;

    if leader_epoch_end.end_offset >= 0 && local_leo > leader_epoch_end.end_offset {
        debug!(
            local_leo,
            truncate_to = leader_epoch_end.end_offset,
            "truncating divergent follower suffix"
        );

        storage.truncate_to(topition, leader_epoch_end.end_offset).await?;
    }

    Ok(())
}
```

## More complete truncation flow

The above local-only snippet assumes storage can ask the leader. In the real follower loop, `OffsetForLeaderEpoch` must be sent to the leader broker over the network. So the final flow should be:

```text
1. Local follower reads its latest leader epoch E_local.
2. Follower sends OffsetForLeaderEpoch(E_local) to current leader.
3. Leader returns end offset O for E_local.
4. If local LEO > O, follower truncates to O.
5. Follower fetches from O.
```

## Steering

Do not truncate below HW. If the local divergent suffix includes committed offsets, that means leadership state is corrupt. Return an internal error and fail loudly.

```rust
let stage = storage.offset_stage(topition).await?;
if truncate_to < stage.high_watermark() {
    return Err(Error::Message(format!(
        "refusing to truncate below HW: truncate_to={truncate_to}, hw={}",
        stage.high_watermark()
    )));
}
```

# 16. ISR tracking

ISR changes belong on the leader.

## Shrink ISR

A follower should leave ISR when:

```text
now - last_fetch_at > replica.lag.time.max.ms
or leader_log_end_offset - follower_log_end_offset > replica.lag.max.messages equivalent
or broker is fenced/offline
```

Implement in `maintain(now)` for each leader partition:

```rust
pub fn shrink_isr(
    state: &PartitionState,
    progress: &BTreeMap<BrokerId, ReplicaProgress>,
    now: SystemTime,
    max_lag: Duration,
) -> Vec<BrokerId> {
    state
        .isr
        .iter()
        .copied()
        .filter(|broker_id| {
            if *broker_id == state.leader_id {
                return true;
            }

            let Some(replica) = progress.get(broker_id) else {
                return false;
            };

            let fresh = replica
                .last_fetch_at
                .elapsed()
                .map(|elapsed| elapsed <= max_lag)
                .unwrap_or(false);

            let caught_up = replica.log_end_offset >= state.high_watermark;

            fresh && caught_up
        })
        .collect()
}
```

## Expand ISR

A follower can rejoin ISR when:

```text
replica.log_end_offset >= leader_log_end_offset
and follower has fetched recently
and broker is not fenced/offline
```

```rust
pub fn maybe_expand_isr(
    state: &PartitionState,
    progress: &BTreeMap<BrokerId, ReplicaProgress>,
    now: SystemTime,
    max_lag: Duration,
) -> Vec<BrokerId> {
    let mut isr = state.isr.clone();

    for broker_id in &state.replicas {
        if isr.contains(broker_id) {
            continue;
        }

        let Some(replica) = progress.get(broker_id) else {
            continue;
        };

        let fresh = now
            .duration_since(replica.last_fetch_at)
            .map(|elapsed| elapsed <= max_lag)
            .unwrap_or(false);

        if fresh && replica.log_end_offset >= state.leader_log_end_offset {
            isr.push(*broker_id);
        }
    }

    isr.sort_unstable();
    isr.dedup();
    isr
}
```

## Steering

Changing ISR should be atomic with leader epoch/partition state reads. In SQL, do it in a transaction. In `DynoStore`, do it inside `meta.with_mut(...)`.

# 17. High watermark advancement

Leader should advance HW after:

1. leader append,
2. follower fetch progress update,
3. ISR shrink/expand,
4. leadership change.

Algorithm:

```rust
async fn recompute_and_store_high_watermark<S>(
    storage: &S,
    topition: &Topition,
) -> Result<i64>
where
    S: Storage,
{
    let state = storage.partition_state(topition).await?;
    let progress = storage.replica_progress(topition).await?;

    let mut leo_by_broker = progress
        .into_iter()
        .map(|p| (p.broker_id, p.log_end_offset))
        .collect::<BTreeMap<_, _>>();

    leo_by_broker.insert(state.leader_id, state.leader_log_end_offset);

    let proposed = compute_high_watermark(
        state.leader_log_end_offset,
        &state.isr,
        &leo_by_broker,
    );

    storage.advance_high_watermark(topition, proposed).await
}
```

## Steering

HW must be monotonic. Never decrease HW during ISR shrink. If a follower falls out of ISR, HW may advance faster, but not backward.

# 18. Leader epochs

## When to create a leader epoch

Create a checkpoint when a partition leader changes or when the same leader is explicitly bumped to a new epoch.

```rust
async fn become_leader(
    &self,
    topition: &Topition,
    leader_id: BrokerId,
    leader_epoch: LeaderEpoch,
) -> Result<PartitionState> {
    let leo = self.local_log_end_offset(topition).await?;

    self.insert_leader_epoch(LeaderEpochCheckpoint {
        topition: topition.clone(),
        leader_epoch,
        leader_id,
        start_offset: leo,
        started_at: SystemTime::now(),
    }).await?;

    self.set_partition_leader(topition, leader_id, leader_epoch).await
}
```

## OffsetForLeaderEpoch behavior

Given checkpoints:

```text
epoch 0 starts at offset 0
epoch 1 starts at offset 100
epoch 2 starts at offset 150
current LEO = 180
```

Then:

```text
OffsetForLeaderEpoch(0) => end_offset 100
OffsetForLeaderEpoch(1) => end_offset 150
OffsetForLeaderEpoch(2) => end_offset 180
```

This is the core truncation primitive.

# 19. Broker lifecycle integration

Modify `tansu-broker/src/broker.rs`.

After broker registration but before accepting traffic, start replication manager alongside listener.

Current `serve()` is:

```rust
pub async fn serve(&mut self, started: Instant) -> Result<()> {
    self.register().await?;
    self.listen(started).await
}
```

Change to:

```rust
pub async fn serve(&mut self, started: Instant) -> Result<()> {
    self.register().await?;

    let replication = tansu_replication::ReplicationManager::new(
        self.node_id,
        self.storage.clone(),
        tansu_replication::ReplicationConfig::default(),
        self.cancellation.child_token(),
    );

    let mut set = tokio::task::JoinSet::new();

    set.spawn(async move {
        replication.run().await
    });

    set.spawn({
        let mut broker = self.clone();
        async move {
            broker.listen(started).await
        }
    });

    tokio::select! {
        result = set.join_next() => {
            match result {
                Some(Ok(Ok(()))) => Ok(()),
                Some(Ok(Err(err))) => Err(err),
                Some(Err(join_err)) => Err(Error::Message(join_err.to_string())),
                None => Ok(()),
            }
        }

        _ = self.cancellation.cancelled() => {
            set.abort_all();
            Ok(())
        }
    }
}
```

You may need to restructure `listen(&self, ...)` to avoid cloning the whole broker if the type bounds make that awkward. The key is that replication and network service share the same cancellation lifecycle.

# 22. Produce/fetch error mapping

Use these mappings consistently:

| Condition                                           | ErrorCode                                                                                                                    |
| --------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| Produce to non-leader                               | `NotLeaderOrFollower`                                                                                                        |
| Fetch from non-leader for normal partition          | `NotLeaderOrFollower`                                                                                                        |
| Supplied leader epoch lower than current            | `FencedLeaderEpoch`                                                                                                          |
| Supplied leader epoch higher than current           | `UnknownLeaderEpoch`                                                                                                         |
| Offset below log start                              | `OffsetOutOfRange`                                                                                                           |
| `acks=all` before append and ISR below minimum      | `NotEnoughReplicas`                                                                                                          |
| `acks=all` append succeeded but HW did not catch up | `NotEnoughReplicasAfterAppend`                                                                                               |
| Follower unavailable                                | `ReplicaNotAvailable` if enum exists, otherwise `UnknownServerError` internally and avoid exposing it unless protocol has it |
| Unknown topic ID                                    | `UnknownTopicId`                                                                                                             |
| Unknown topic/partition                             | `UnknownTopicOrPartition`                                                                                                    |

# 28. Final completion checklist

Before calling Phase 11 done, I would require this exact checklist to pass:

```text
[ ] Broker registry stores 3 brokers with distinct IDs/listeners.
[ ] Topic RF=3 creates three distinct replicas per partition.
[ ] Metadata reports correct leader, leader epoch, replicas, ISR, offline replicas.
[ ] Produce to leader succeeds.
[ ] Produce to follower returns NotLeaderOrFollower with current_leader.
[ ] acks=1 returns after leader append.
[ ] acks=all returns only after HW reaches appended offset.
[ ] min.insync.replicas is enforced.
[ ] Consumer fetch does not read above HW.
[ ] Follower fetch can read above HW up to leader LEO.
[ ] Follower append preserves leader offsets and batches exactly.
[ ] Follower progress updates durable LEO.
[ ] ISR expands after follower catch-up.
[ ] ISR shrinks after stale follower.
[ ] HW is monotonic and equals min LEO of ISR.
[ ] Leader epoch checkpoint created at every leadership transition.
[ ] OffsetForLeaderEpoch returns correct end offsets.
[ ] Stale leader epoch returns FencedLeaderEpoch.
[ ] Future leader epoch returns UnknownLeaderEpoch.
[ ] Follower truncates divergent suffix before fetching new leader.
[ ] Truncation refuses to truncate below HW.
[ ] Broker shutdown cancels replication tasks cleanly.
[ ] Kafka CLI topic describe shows correct Replicas/Isr.
[ ] Multi-broker integration tests pass for memory/object-store and at least one SQL backend.
```

The most important steering point: **do not hide replication inside existing `produce()` and `fetch()` alone.** Phase 11 needs durable partition state plus explicit follower append/truncate APIs. Once those primitives are correct, the Kafka protocol services become straightforward, and Phase 12’s controller can later drive leadership changes without rewriting the replication layer.
