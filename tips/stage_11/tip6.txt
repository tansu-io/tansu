# Phase 11: finish the replication layer for Tansu

The big architectural point: **do not treat “shared durable storage” as Kafka replication**. Tansu’s current README describes it as a “stateless Kafka-compatible broker with pluggable storage,” including PostgreSQL, SQLite, S3, and memory, and the current quickstart shows `--replication-factor=1` with a single node acting as leader/ISR in the example output. That is a great base, but Phase 11 needs broker-local replica state, leader/follower roles, ISR movement, high-watermark advancement, leader epochs, and truncation after leadership changes. ([GitHub][1])

Kafka-compatible replication means clients must see topic partitions replicated across brokers, and clients must produce/fetch through the current partition leader discovered via metadata. The official Kafka protocol describes partitions as ordered commit logs replicated with a replication factor, and says clients send publish/fetch requests to the broker currently acting as leader. ([Apache Kafka][2])

Below is the plan I would use to finish **all of Phase 11** in Tansu.

---

# 0. Phase 11 completion bar

Phase 11 is complete when Tansu can run a **3-broker cluster** with a topic such as:

```bash
kafka-topics \
  --bootstrap-server localhost:9092 \
  --create \
  --topic replicated \
  --partitions 3 \
  --replication-factor 3 \
  --config min.insync.replicas=2
```

and all of the following are true:

1. `kafka-topics --describe` shows correct `Leader`, `Replicas`, and `Isr` for every partition.
2. Java Kafka clients and librdkafka clients can produce with `acks=all`.
3. A follower fetches from the leader using Kafka’s normal `Fetch` request with `replica_id >= 0`; consumers still use `replica_id = -1`. Kafka’s Fetch schema explicitly distinguishes follower fetches from consumer fetches via `replica_id`, and newer fetch versions include `current_leader_epoch`, `last_fetched_epoch`, and follower `log_start_offset`. ([Apache Kafka][2])
4. The leader tracks follower log-end offsets and ISR membership.
5. The leader’s high watermark is monotonic and is the minimum replicated offset across the ISR.
6. Consumers never read beyond the high watermark.
7. Followers replicate records that are not yet committed, because followers must catch up to the leader’s log-end offset, not only the high watermark.
8. `acks=all` waits until the produced offset is committed by ISR replication or times out.
9. If ISR count is below `min.insync.replicas`, Produce returns Kafka-compatible errors: `NOT_ENOUGH_REPLICAS` before append or `NOT_ENOUGH_REPLICAS_AFTER_APPEND` after append. Kafka defines these exact error codes for insufficient in-sync replicas. ([Apache Kafka][2])
10. If a request reaches a non-leader/non-replica, Tansu returns `NOT_LEADER_OR_FOLLOWER`, matching Kafka’s protocol error semantics. ([Apache Kafka][2])
11. On leader change, replicas use leader epochs to find the common log prefix and truncate divergent tails.
12. `OffsetForLeaderEpoch` works for follower recovery and returns the epoch end offset; Kafka’s protocol defines `OffsetForLeaderEpoch` with `current_leader_epoch`, `leader_epoch`, `replica_id`, and response `end_offset`. ([Apache Kafka][2])
13. Old leader epochs are fenced with `FENCED_LEADER_EPOCH`; future epochs return `UNKNOWN_LEADER_EPOCH`, matching Kafka’s epoch-fencing behavior. ([Apache Kafka][2])
14. Crash recovery reloads local replica logs, high-watermark checkpoints, leader-epoch checkpoints, and ISR metadata without offset corruption.
15. There are deterministic unit tests, multi-broker integration tests, protocol golden tests, and client-compatibility tests.

# 1. Important scope boundary: Phase 11 versus Phase 12

Your milestone 11 says:

> Leader/follower replication, ISR tracking, high watermark advancement, leader epochs, follower fetch, truncation on leader change.

Your milestone 12 says:

> Broker registration, partition leadership, metadata quorum, failure detection, leader election, partition reassignment.

So Phase 11 should **not** attempt to build the final cluster controller. Instead, Phase 11 should build the complete replication **data plane** plus controller-facing hooks.

That means:

```text
Phase 11 owns:
- local replica logs
- leader append path
- follower fetch path
- ISR state machine
- high watermark advancement
- leader epoch storage
- OffsetForLeaderEpoch
- truncation after leadership change
- replica progress tracking
- manual/test leadership transitions

Phase 12 owns:
- automatic failure detection
- durable metadata quorum
- automatic leader election
- reassignment orchestration
- rolling controller upgrades
```

But Phase 11 still needs an explicit interface that Phase 12 can call:

```rust
become_leader(topition, assignment, epoch)
become_follower(topition, leader, epoch)
remove_replica(topition)
update_assignment(topition, replicas, isr, epoch)
```

That way Phase 12 can replace the temporary/manual/test controller without rewriting replication.

# 2. Current Tansu seams to use

Tansu already has good seams for this work:

* The workspace contains separate crates for broker, storage, service framing, protocol sans-I/O code, client tooling, and CLI tooling. ([GitHub][3])
* `tansu-storage` currently exposes the central `Storage` abstraction, with storage backends including memory, S3/object storage, PostgreSQL, SQLite/libSQL, and Turso. ([GitHub][4])
* The existing storage trait already has `produce`, `fetch`, `offset_stage`, `metadata`, `register_broker`, `brokers`, and topic/admin operations. That is exactly where replication semantics should be enforced or wrapped. ([GitHub][4])
* `OffsetStage` already contains `last_stable`, `high_watermark`, and `log_start`, which is a useful compatibility anchor for Phase 11 because replication primarily changes how high watermark is computed and persisted. ([GitHub][4])
* `tansu-broker` registers the broker with storage, starts the TCP listener, builds services from storage/coordinator, and periodically calls `storage.maintain(SystemTime::now())`. That is the right place to start and supervise replication workers. ([GitHub][5])
* `tansu-sans-io/message` contains Kafka protocol JSON definitions, including `FetchRequest.json` and `FetchResponse.json`; that crate is the right place to verify or add generated request/response support for `OffsetForLeaderEpoch` and any inter-broker APIs used by the data plane. ([GitHub][6])
* The workspace currently uses Rust 2024 and forbids unsafe code, so the replication design should stay fully safe Rust. ([GitHub][3])

# 3. Architecture decision: add a replicated-log mode, do not mutate every backend first

I would implement Phase 11 as an opt-in **replicated local log mode** first:

```text
replicated-log://var/lib/tansu?metadata=postgres://...
replicated-log://var/lib/tansu?metadata=sqlite://...
replicated-log://var/lib/tansu?metadata=memory://...
```

The local broker-owned log is the source of truth for that broker’s replica. Metadata/configuration can still live in the existing pluggable storage layer.

Why this is the cleanest path:

1. Kafka replication is about **multiple broker-local logs** converging through leader/follower fetch, not about every broker reading the same shared object/database record.
2. Followers must maintain independent local log-end offset, high-watermark checkpoint, log-start offset, and leader-epoch checkpoint.
3. Divergent followers must truncate local data after leader change.
4. The Produce path must support `acks=1` and `acks=all`, and that requires per-follower progress.
5. Shared storage can remain valuable later for tiered storage, metadata, recovery, or lake integrations, but it should not hide Phase 11’s replica state.

This avoids ripping apart the existing Tansu storage backends too early. The current storage abstraction can remain usable for non-replicated mode, while replicated mode wraps or extends it.

# 5. Core data model

Create `tansu-storage/src/replication/types.rs`.

```rust
use std::{
    collections::{BTreeMap, BTreeSet},
    time::SystemTime,
};

use serde::{Deserialize, Serialize};

use crate::Topition;

pub type BrokerId = i32;
pub type LeaderEpoch = i32;
pub type Offset = i64;

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct ReplicaAssignment {
    pub topition: Topition,
    pub leader: BrokerId,
    pub leader_epoch: LeaderEpoch,
    pub replicas: Vec<BrokerId>,
    pub isr: BTreeSet<BrokerId>,
    pub adding_replicas: BTreeSet<BrokerId>,
    pub removing_replicas: BTreeSet<BrokerId>,
    pub min_insync_replicas: usize,
}

impl ReplicaAssignment {
    pub fn contains_replica(&self, broker_id: BrokerId) -> bool {
        self.replicas.contains(&broker_id)
    }

    pub fn is_leader(&self, broker_id: BrokerId) -> bool {
        self.leader == broker_id
    }

    pub fn is_in_sync(&self, broker_id: BrokerId) -> bool {
        self.isr.contains(&broker_id)
    }
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub enum ReplicaRole {
    Leader {
        epoch: LeaderEpoch,
    },
    Follower {
        leader: BrokerId,
        epoch: LeaderEpoch,
    },
    Offline,
    None,
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct ReplicaState {
    pub topition: Topition,
    pub broker_id: BrokerId,
    pub role: ReplicaRole,
    pub assignment: Option<ReplicaAssignment>,
    pub log_start_offset: Offset,
    pub local_log_end_offset: Offset,
    pub high_watermark: Offset,
    pub last_stable_offset: Offset,
    pub leader_epoch: LeaderEpoch,
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct ReplicaProgress {
    pub broker_id: BrokerId,
    pub log_start_offset: Offset,
    pub log_end_offset: Offset,
    pub last_fetch_at: Option<SystemTime>,
    pub last_caught_up_at: Option<SystemTime>,
    pub last_fetch_epoch: LeaderEpoch,
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct LeaderReplicaState {
    pub assignment: ReplicaAssignment,
    pub local: ReplicaState,
    pub followers: BTreeMap<BrokerId, ReplicaProgress>,
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct AppendResult {
    pub base_offset: Offset,
    pub last_offset: Offset,
    pub log_end_offset: Offset,
    pub high_watermark: Offset,
    pub leader_epoch: LeaderEpoch,
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct FetchReplicaRequest {
    pub replica_id: BrokerId,
    pub topition: Topition,
    pub fetch_offset: Offset,
    pub max_bytes: u32,
    pub current_leader_epoch: LeaderEpoch,
    pub last_fetched_epoch: LeaderEpoch,
    pub follower_log_start_offset: Offset,
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct FetchReplicaResponse {
    pub high_watermark: Offset,
    pub log_start_offset: Offset,
    pub leader_epoch: LeaderEpoch,
    pub records: Vec<tansu_sans_io::record::deflated::Batch>,
    pub diverging_epoch: Option<EpochEndOffset>,
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct EpochEntry {
    pub epoch: LeaderEpoch,
    pub start_offset: Offset,
}

#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct EpochEndOffset {
    pub leader_epoch: LeaderEpoch,
    pub end_offset: Offset,
}
```

The model deliberately separates:

```text
local_log_end_offset  = this broker’s local end offset
high_watermark        = committed offset visible to consumers
last_stable_offset    = transaction-aware visibility; Phase 14 completes it
leader_epoch          = epoch used to detect stale leaders and divergent logs
```

```rust
use crate::{Error, Topition};
use tansu_sans_io::ErrorCode;

#[derive(Clone, Debug, thiserror::Error)]
pub enum ReplicationError {
    #[error("broker {broker_id} is not the leader for {topition:?}")]
    NotLeader {
        broker_id: i32,
        topition: Topition,
    },

    #[error("broker {broker_id} is not a replica for {topition:?}")]
    NotReplica {
        broker_id: i32,
        topition: Topition,
    },

    #[error("not enough in-sync replicas: isr={isr}, min_isr={min_isr}")]
    NotEnoughReplicas {
        isr: usize,
        min_isr: usize,
    },

    #[error("leader epoch is fenced: request={request_epoch}, current={current_epoch}")]
    FencedLeaderEpoch {
        request_epoch: i32,
        current_epoch: i32,
    },

    #[error("unknown leader epoch: request={request_epoch}, current={current_epoch}")]
    UnknownLeaderEpoch {
        request_epoch: i32,
        current_epoch: i32,
    },

    #[error("offset out of range: requested={requested}, log_start={log_start}, log_end={log_end}")]
    OffsetOutOfRange {
        requested: i64,
        log_start: i64,
        log_end: i64,
    },
}

impl ReplicationError {
    pub fn error_code(&self) -> ErrorCode {
        match self {
            Self::NotLeader { .. } | Self::NotReplica { .. } => ErrorCode::NotLeaderOrFollower,
            Self::NotEnoughReplicas { .. } => ErrorCode::NotEnoughReplicas,
            Self::FencedLeaderEpoch { .. } => ErrorCode::FencedLeaderEpoch,
            Self::UnknownLeaderEpoch { .. } => ErrorCode::UnknownLeaderEpoch,
            Self::OffsetOutOfRange { .. } => ErrorCode::OffsetOutOfRange,
        }
    }
}

impl From<ReplicationError> for Error {
    fn from(value: ReplicationError) -> Self {
        Error::Api(value.error_code())
    }
}
```

Create a trait in `tansu-storage/src/replication/state.rs`:

```rust
use async_trait::async_trait;
use tansu_sans_io::{IsolationLevel, record::deflated};

use crate::{Result, Topition};

use super::{

    FetchReplicaRequest,

#[async_trait]
pub trait ReplicationStorage: Send + Sync + Clone + 'static {

    async fn list_local_replicas(&self) -> Result<Vec<ReplicaState>>;

    async fn become_leader(
        &self,
        topition: &Topition,

    async fn become_follower(

    async fn append_as_leader(
        &self,
        topition: &Topition,
        epoch: LeaderEpoch,
        batch: deflated::Batch,
    ) -> Result<AppendResult>;

    async fn append_as_follower(
        &self,
        topition: &Topition,
        leader: BrokerId,
        epoch: LeaderEpoch,
        batch: deflated::Batch,
    ) -> Result<AppendResult>;

    async fn fetch_for_consumer(
        &self,
        topition: &Topition,
        offset: Offset,
        min_bytes: u32,

    ) -> Result<Vec<deflated::Batch>>;

    async fn fetch_for_follower(
        &self,
        request: FetchReplicaRequest,
    ) -> Result<FetchReplicaResponse>;

    async fn update_follower_progress(
        &self,
        topition: &Topition,
        follower: ReplicaProgress,
    ) -> Result<Offset>;

    async fn offset_for_leader_epoch(
        &self,
        topition: &Topition,
        current_leader_epoch: LeaderEpoch,
        requested_epoch: LeaderEpoch,
    ) -> Result<EpochEndOffset>;

    async fn truncate_to(
        &self,
        topition: &Topition,
        offset: Offset,
    ) -> Result<()>;

    async fn set_high_watermark(
        &self,
        topition: &Topition,
        high_watermark: Offset,
    ) -> Result<()>;

    async fn wait_for_high_watermark(
        &self,
        topition: &Topition,
        offset_exclusive: Offset,
        timeout: std::time::Duration,
    ) -> Result<bool>;
}
```

This trait makes replication explicit and avoids smuggling leader/follower logic into generic `produce` and `fetch`.

# 7. Local replicated log

Create `tansu-storage/src/replication/local.rs`.

Suggested disk layout:

```text
/var/lib/tansu/
  cluster.meta
  brokers/
    1/
      replicas/
        <topic-id-or-topic-name>/
          00000000000000000000.log
          00000000000000000000.index
          00000000000000000000.timeindex
          leader-epoch-checkpoint
          high-watermark-checkpoint
          recovery-point-checkpoint
          replica-state.json
```

Use topic IDs if the current metadata layer has them available; otherwise start with sanitized topic names and add topic-ID migration later.

Minimum file invariants:

```text
.log:
- append-only record batches
- offset-ordered
- CRC-valid record batches
- never mutate except truncate tail

.index:
- maps relative offset -> physical position
- rebuilt on recovery if missing/corrupt

.timeindex:
- maps max timestamp -> offset
- rebuilt on recovery if missing/corrupt

leader-epoch-checkpoint:
- monotonically increasing epochs
- each epoch has start offset

- never greater than local log-end offset

- assignment
- local log start

- leader epoch
```

# 8. Leader epoch checkpoint

Create `tansu-storage/src/replication/leader_epoch.rs`.

This is one of the most important correctness pieces in Phase 11. It lets a follower ask the new leader, “what offset ended epoch X?” and truncate divergent local data.

```rust
use serde::{Deserialize, Serialize};

use super::{EpochEndOffset, EpochEntry, LeaderEpoch, Offset};

#[derive(Clone, Debug, Default, Eq, PartialEq, Serialize, Deserialize)]
pub struct LeaderEpochTable {
    entries: Vec<EpochEntry>,
}

impl LeaderEpochTable {
    pub fn entries(&self) -> &[EpochEntry] {
        &self.entries
    }

    pub fn current_epoch(&self) -> Option<LeaderEpoch> {
        self.entries.last().map(|entry| entry.epoch)
    }

    pub fn current_start_offset(&self) -> Option<Offset> {
        self.entries.last().map(|entry| entry.start_offset)
    }

    pub fn assign_epoch(&mut self, epoch: LeaderEpoch, start_offset: Offset) {
        if let Some(last) = self.entries.last() {
            assert!(
                epoch >= last.epoch,
                "leader epoch must not move backwards: epoch={epoch}, last={}",
                last.epoch,
            );

            if epoch == last.epoch {
                return;
            }

            assert!(
                start_offset >= last.start_offset,
                "epoch start offset must not move backwards: start={start_offset}, last={}",
                last.start_offset,
            );
        }

        self.entries.push(EpochEntry {
            epoch,
            start_offset,
        });
    }

    pub fn end_offset_for(
        &self,
        requested_epoch: LeaderEpoch,
        log_end_offset: Offset,
    ) -> Option<EpochEndOffset> {
        let index = self
            .entries
            .iter()
            .position(|entry| entry.epoch == requested_epoch)?;

        let end_offset = self
            .entries
            .get(index + 1)
            .map(|entry| entry.start_offset)
            .unwrap_or(log_end_offset);

        Some(EpochEndOffset {
            leader_epoch: requested_epoch,
            end_offset,
        })
    }

    pub fn truncate_from_offset(&mut self, offset: Offset) {
        while let Some(last) = self.entries.last() {
            if last.start_offset >= offset {
                self.entries.pop();
            } else {
                break;
            }
        }
    }
}
```

# 9. High watermark calculation

Create `tansu-storage/src/replication/high_watermark.rs`.

The leader’s high watermark should be:

```text
min(log_end_offset of every broker in ISR)
```

bounded by the leader’s own LEO, and **never decreasing**.

```rust
use std::collections::{BTreeMap, BTreeSet};

use super::{BrokerId, Offset, ReplicaProgress};

pub fn compute_high_watermark(
    leader_id: BrokerId,
    leader_log_end_offset: Offset,
    isr: &BTreeSet<BrokerId>,
    progress: &BTreeMap<BrokerId, ReplicaProgress>,
    min_insync_replicas: usize,
    current_high_watermark: Offset,
) -> Offset {
    if isr.len() < min_insync_replicas {
        return current_high_watermark;
    }

    let mut candidate = leader_log_end_offset;

    for broker_id in isr {
        let leo = if *broker_id == leader_id {
            leader_log_end_offset
        } else {
            progress
                .get(broker_id)
                .map(|p| p.log_end_offset)
                .unwrap_or(current_high_watermark)
        };

        candidate = candidate.min(leo);
    }

    current_high_watermark.max(candidate.min(leader_log_end_offset))
}
```

# 10. ISR state machine

Create `tansu-storage/src/replication/isr.rs`.

ISR rules for Phase 11:

```text
A replica is eligible to join ISR when:
- it is assigned as a replica
- it has fetched from the current leader epoch
- its LEO has caught up to leader LEO
- it is not explicitly fenced/offline

A replica leaves ISR when:
- it stops fetching longer than replica.lag.time.max.ms
- it reports a stale epoch
- it truncates/rebuilds
- Phase 12 reports the broker dead
```

Use a deterministic function for testability:

```rust
use std::{
    collections::{BTreeMap, BTreeSet},
    time::{Duration, SystemTime},
};

use super::{BrokerId, Offset, ReplicaAssignment, ReplicaProgress};

#[derive(Clone, Debug, Eq, PartialEq)]
pub struct IsrUpdate {
    pub next_isr: BTreeSet<BrokerId>,
    pub added: BTreeSet<BrokerId>,
    pub removed: BTreeSet<BrokerId>,
}

pub fn update_isr(
    now: SystemTime,
    leader_id: BrokerId,
    leader_log_end_offset: Offset,
    assignment: &ReplicaAssignment,
    progress: &BTreeMap<BrokerId, ReplicaProgress>,
    current_isr: &BTreeSet<BrokerId>,
    replica_lag_timeout: Duration,
) -> IsrUpdate {
    let mut next = BTreeSet::new();

    for broker_id in &assignment.replicas {
        if *broker_id == leader_id {
            next.insert(*broker_id);
            continue;
        }

        let Some(p) = progress.get(broker_id) else {
            continue;
        };

        let fetch_recent = p
            .last_fetch_at
            .and_then(|last| now.duration_since(last).ok())
            .is_some_and(|age| age <= replica_lag_timeout);

        let caught_up = p.log_end_offset >= leader_log_end_offset;

        if fetch_recent && caught_up {
            next.insert(*broker_id);
        } else if current_isr.contains(broker_id) && fetch_recent {
            // Keep recently fetching replicas in ISR if they were previously caught up.
            // The watchdog handles lag timeout; avoid flapping on transient append bursts.
            next.insert(*broker_id);
        }
    }

    let added = next.difference(current_isr).copied().collect();
    let removed = current_isr.difference(&next).copied().collect();

    IsrUpdate {
        next_isr: next,
        added,
        removed,
    }
}
```

For production, tune this with Kafka-like settings:

```text

replica.fetch.wait.max.ms = 500
replica.fetch.min.bytes = 1
replica.fetch.max.bytes = 1048576
replica.socket.timeout.ms = 30000
```

# 11. Produce path changes

Touch:

```text
tansu-storage/src/service/produce.rs
```

The Produce service must enforce:

```text
Client produce may only append on the leader.
acks=0: append and no meaningful response body
acks=1: respond after local leader append
acks=all/-1: respond after HW passes produced offset, or return timeout / not enough replicas
```

Kafka-compatible errors to return:

```text
NOT_LEADER_OR_FOLLOWER:
- this broker is neither current leader nor replica for the partition
- produce sent to follower

NOT_ENOUGH_REPLICAS:
- ISR count < min.insync.replicas before append

NOT_ENOUGH_REPLICAS_AFTER_APPEND:
- leader appended but ISR did not replicate before timeout
```

Sketch:

```rust
use std::time::Duration;

use tansu_sans_io::ErrorCode;

use crate::{
    Result,
    Topition,
    replication::{ReplicationStorage, ReplicaRole},
};

pub async fn produce_replicated<S>(
    storage: S,
    topition: Topition,
    required_acks: i16,
    timeout: Duration,
    batch: tansu_sans_io::record::deflated::Batch,
) -> Result<ProducePartitionOutcome>
where
    S: ReplicationStorage,
{
    let state = storage.load_replica_state(&topition).await?;

    let ReplicaRole::Leader { epoch } = state.role else {
        return Ok(ProducePartitionOutcome::error(
            ErrorCode::NotLeaderOrFollower,
        ));
    };

    let assignment = state
        .assignment
        .clone()
        .ok_or_else(|| crate::Error::Api(ErrorCode::UnknownTopicOrPartition))?;

    if required_acks == -1 && assignment.isr.len() < assignment.min_insync_replicas {
        return Ok(ProducePartitionOutcome::error(
            ErrorCode::NotEnoughReplicas,
        ));
    }

    let append = storage
        .append_as_leader(&topition, epoch, batch)
        .await?;

    match required_acks {
        0 | 1 => Ok(ProducePartitionOutcome::success(
            append.base_offset,
            ErrorCode::None,
        )),

        -1 => {
            let committed = storage
                .wait_for_high_watermark(
                    &topition,
                    append.last_offset + 1,
                    timeout,
                )
                .await?;

            if committed {
                Ok(ProducePartitionOutcome::success(
                    append.base_offset,
                    ErrorCode::None,
                ))
            } else {
                Ok(ProducePartitionOutcome::success(
                    append.base_offset,
                    ErrorCode::NotEnoughReplicasAfterAppend,
                ))
            }
        }

        _ => Ok(ProducePartitionOutcome::error(ErrorCode::InvalidRequiredAcks)),
    }
}

#[derive(Clone, Debug)]
pub struct ProducePartitionOutcome {
    pub base_offset: i64,
    pub error_code: ErrorCode,
}

impl ProducePartitionOutcome {
    pub fn success(base_offset: i64, error_code: ErrorCode) -> Self {
        Self {
            base_offset,
            error_code,
        }
    }

    pub fn error(error_code: ErrorCode) -> Self {
        Self {
            base_offset: -1,
            error_code,
        }
    }
}
```

Critical steering:

* Do not wait for all replicas, only the ISR.
* Do not move HW backwards if ISR shrinks.
* If `acks=all` times out after append, return `NOT_ENOUGH_REPLICAS_AFTER_APPEND`, not `NOT_ENOUGH_REPLICAS`.
* The base offset must be assigned before response and must not change after replication.
* The response error must be per-topic-partition, not only top-level.

# 12. Fetch path changes

Touch:

```text
tansu-storage/src/service/fetch.rs
```

You need two different Fetch modes.

## 12.1 Consumer fetch

```text
replica_id == -1
```

Consumer fetch behavior:

```text
- request can go only to leader for now
- return records from requested offset up to HW
- if isolation_level=read_committed, cap at LSO
- return high_watermark in response
- do not expose uncommitted tail
```

## 12.2 Follower fetch

```text
replica_id >= 0
```

Follower fetch behavior:

```text
- request can go to leader
- requester must be assigned replica
- validate current_leader_epoch
- return records from requested offset up to leader LEO, not only HW
- include high_watermark
- update follower progress using request fetch_offset and response result
- include diverging_epoch/current_leader when epoch mismatch is detected
```

Kafka’s Fetch request schema explicitly says `replica_id` is the broker ID of the follower or `-1` for a consumer. ([Apache Kafka][2])

Sketch:

```rust
pub enum FetchCaller {
    Consumer,
    Follower { broker_id: i32 },
}

impl FetchCaller {
    pub fn from_replica_id(replica_id: i32) -> Self {
        if replica_id < 0 {
            Self::Consumer
        } else {
            Self::Follower {
                broker_id: replica_id,
            }
        }
    }
}

pub async fn fetch_replicated<S>(
    storage: S,
    caller: FetchCaller,
    topition: Topition,
    fetch_offset: i64,
    min_bytes: u32,
    max_bytes: u32,
    isolation: IsolationLevel,
    current_leader_epoch: Option<i32>,
    last_fetched_epoch: Option<i32>,
    follower_log_start_offset: Option<i64>,
) -> Result<FetchPartitionOutcome>
where
    S: ReplicationStorage,
{
    match caller {
        FetchCaller::Consumer => {
            let batches = storage
                .fetch_for_consumer(
                    &topition,
                    fetch_offset,
                    min_bytes,
                    max_bytes,
                    isolation,
                )
                .await?;

            let state = storage.load_replica_state(&topition).await?;

            Ok(FetchPartitionOutcome {
                error_code: ErrorCode::None,
                high_watermark: state.high_watermark,
                log_start_offset: state.log_start_offset,
                records: batches,
                diverging_epoch: None,
            })
        }

        FetchCaller::Follower { broker_id } => {
            let request = FetchReplicaRequest {
                replica_id: broker_id,
                topition: topition.clone(),
                fetch_offset,
                max_bytes,
                current_leader_epoch: current_leader_epoch.unwrap_or(-1),
                last_fetched_epoch: last_fetched_epoch.unwrap_or(-1),
                follower_log_start_offset: follower_log_start_offset.unwrap_or(0),
            };

            let response = storage.fetch_for_follower(request).await?;

            Ok(FetchPartitionOutcome {
                error_code: ErrorCode::None,
                high_watermark: response.high_watermark,
                log_start_offset: response.log_start_offset,
                records: response.records,
                diverging_epoch: response.diverging_epoch,
            })
        }
    }
}

#[derive(Clone, Debug)]
pub struct FetchPartitionOutcome {
    pub error_code: ErrorCode,
    pub high_watermark: i64,
    pub log_start_offset: i64,
    pub records: Vec<tansu_sans_io::record::deflated::Batch>,
    pub diverging_epoch: Option<EpochEndOffset>,
}
```

Critical steering:

* Consumer fetch must never read beyond HW.
* Follower fetch must be allowed to read beyond HW up to LEO.
* Follower fetch must update leader-side follower progress after the leader knows what it actually sent.
* Follower fetch with stale `current_leader_epoch` must return `FENCED_LEADER_EPOCH`.
* Follower fetch with future `current_leader_epoch` must return `UNKNOWN_LEADER_EPOCH`.
* Fetch response should include the current leader fields for modern protocol versions when errors indicate stale routing.

# 14. Follower fetcher

Create:

```text
tansu-broker/src/replication/fetcher.rs
tansu-broker/src/replication/client.rs
```

The follower fetcher loop:

```text
1. Load local replica state.
2. If follower, connect to leader advertised listener.
3. Query OffsetForLeaderEpoch when needed.
4. Fetch from local LEO.
5. Append returned records as follower.
6. Persist leader high watermark locally.
7. Repeat.
8. On epoch divergence, truncate and retry.
9. On network error, backoff.
10. On role change/shutdown, exit cleanly.
```

Sketch:

```rust
use std::time::Duration;

use tokio_util::sync::CancellationToken;
use tracing::{debug, error, info, warn};

use tansu_storage::{
    Topition,
    replication::{
        FetchReplicaRequest,
        ReplicationStorage,
        ReplicaRole,
    },
};

use super::client::ReplicationClient;

#[derive(Clone, Debug)]
pub struct FollowerFetcher<S, C> {
    pub broker_id: i32,
    pub topition: Topition,
    pub storage: S,
    pub client: C,
    pub cancellation: CancellationToken,
    pub fetch_max_bytes: u32,
    pub retry_backoff: Duration,
}

impl<S, C> FollowerFetcher<S, C>
where
    S: ReplicationStorage,
    C: ReplicationClient + Clone + Send + Sync + 'static,
{
    pub async fn run(self) {
        loop {
            if self.cancellation.is_cancelled() {
                break;
            }

            match self.step().await {
                Ok(()) => {}
                Err(error) => {
                    warn!(?error, ?self.topition, "replica fetch failed");
                    tokio::select! {
                        _ = tokio::time::sleep(self.retry_backoff) => {}
                        _ = self.cancellation.cancelled() => break,
                    }
                }
            }
        }

        info!(?self.topition, "follower fetcher stopped");
    }

    async fn step(&self) -> tansu_storage::Result<()> {
        let state = self.storage.load_replica_state(&self.topition).await?;

        let ReplicaRole::Follower { leader, epoch } = state.role else {
            debug!(?self.topition, ?state.role, "not follower; fetcher exits");
            self.cancellation.cancel();
            return Ok(());
        };

        let fetch_offset = state.local_log_end_offset;
        let last_fetched_epoch = state.leader_epoch;

        let response = self
            .client
            .fetch_from_leader(FetchReplicaRequest {
                replica_id: self.broker_id,
                topition: self.topition.clone(),
                fetch_offset,
                max_bytes: self.fetch_max_bytes,
                current_leader_epoch: epoch,
                last_fetched_epoch,
                follower_log_start_offset: state.log_start_offset,
            })
            .await?;

        if let Some(diverging) = response.diverging_epoch {
            self.storage
                .truncate_to(&self.topition, diverging.end_offset)
                .await?;
            return Ok(());
        }

        for batch in response.records {
            self.storage
                .append_as_follower(
                    &self.topition,
                    leader,
                    response.leader_epoch,
                    batch,
                )
                .await?;
        }

        self.storage
            .set_high_watermark(&self.topition, response.high_watermark)
            .await?;

        Ok(())
    }
}
```

# 15. Leader-side follower progress

When a follower fetches from the leader, the leader should update:

```text
follower.log_start_offset
follower.log_end_offset
follower.last_fetch_at
follower.last_caught_up_at
follower.last_fetch_epoch
```

But be careful: the follower’s LEO after response is:

```text
response_end_offset = fetch_offset + number_of_records_returned
```

If the response contains no records, the follower’s known LEO is at least the request fetch offset, assuming the request was valid.

Leader-side pseudocode:

```rust
pub async fn fetch_for_follower(
    &self,
    request: FetchReplicaRequest,
) -> Result<FetchReplicaResponse> {
    let state = self.load_replica_state(&request.topition).await?;

    let ReplicaRole::Leader { epoch } = state.role else {
        return Err(ReplicationError::NotLeader {
            broker_id: self.broker_id(),
            topition: request.topition,
        }.into());
    };

    if request.current_leader_epoch < epoch {
        return Err(ReplicationError::FencedLeaderEpoch {
            request_epoch: request.current_leader_epoch,
            current_epoch: epoch,
        }.into());
    }

    if request.current_leader_epoch > epoch {
        return Err(ReplicationError::UnknownLeaderEpoch {
            request_epoch: request.current_leader_epoch,
            current_epoch: epoch,
        }.into());
    }

    let assignment = state.assignment.as_ref().expect("leader has assignment");

    if !assignment.contains_replica(request.replica_id) {
        return Err(ReplicationError::NotReplica {
            broker_id: request.replica_id,
            topition: request.topition,
        }.into());
    }

    let local_leo = state.local_log_end_offset;

    if request.fetch_offset < state.log_start_offset || request.fetch_offset > local_leo {
        let diverging = self
            .offset_for_leader_epoch(
                &request.topition,
                epoch,
                request.last_fetched_epoch,
            )
            .await
            .ok();

        return Ok(FetchReplicaResponse {
            high_watermark: state.high_watermark,
            log_start_offset: state.log_start_offset,
            leader_epoch: epoch,
            records: vec![],
            diverging_epoch: diverging,
        });
    }

    let records = self
        .local_log(&request.topition)
        .read_batches(
            request.fetch_offset,
            request.max_bytes,
            local_leo, // follower can read to LEO
        )
        .await?;

    let response_end_offset = records
        .last()
        .map(|batch| batch_last_offset_exclusive(batch))
        .unwrap_or(request.fetch_offset);

    self.update_follower_progress(
        &request.topition,
        ReplicaProgress {
            broker_id: request.replica_id,
            log_start_offset: request.follower_log_start_offset,
            log_end_offset: response_end_offset,
            last_fetch_at: Some(SystemTime::now()),
            last_caught_up_at: (response_end_offset >= local_leo)
                .then_some(SystemTime::now()),
            last_fetch_epoch: epoch,
        },
    )
    .await?;

    Ok(FetchReplicaResponse {
        high_watermark: state.high_watermark,
        log_start_offset: state.log_start_offset,
        leader_epoch: epoch,
        records,
        diverging_epoch: None,
    })
}
```

Critical steering:

* Never update follower progress to an offset greater than the leader’s LEO.
* Do not trust follower-reported LEO blindly.
* Derive progress from the leader’s own fetch response.
* Use follower-reported `log_start_offset` only for metadata/diagnostics and future retention safety.

# 16. Delayed produce acknowledgements


When `acks=all`, the Produce request must wait until:

```text
high_watermark >= last_offset + 1
```

Use `Notify` or watch channels per partition. A clean design:

```rust
use std::{
    collections::BTreeMap,
    sync::Arc,
    time::{Duration, Instant},
};

use tokio::sync::{Mutex, Notify};

use crate::Topition;

#[derive(Clone, Debug, Default)]
pub struct HighWatermarkWaiters {
    inner: Arc<Mutex<BTreeMap<Topition, Arc<Notify>>>>,
}

impl HighWatermarkWaiters {
    pub async fn notify(&self, topition: &Topition) {
        let notify = {
            let inner = self.inner.lock().await;
            inner.get(topition).cloned()
        };

        if let Some(notify) = notify {
            notify.notify_waiters();
        }
    }

    pub async fn wait_until<F>(
        &self,
        topition: &Topition,
        timeout: Duration,
        mut condition: F,
    ) -> bool
    where
        F: FnMut() -> bool,
    {
        let deadline = Instant::now() + timeout;

        loop {
            if condition() {
                return true;
            }

            let notify = {
                let mut inner = self.inner.lock().await;
                inner
                    .entry(topition.clone())
                    .or_insert_with(|| Arc::new(Notify::new()))
                    .clone()
            };

            let remaining = deadline.saturating_duration_since(Instant::now());

            if remaining.is_zero() {
                return condition();
            }

            let woke = tokio::time::timeout(remaining, notify.notified()).await;

            if woke.is_err() {
                return condition();
            }
        }
    }
}
```

# 17. Truncation on leader change

Create:

```text
tansu-storage/src/replication/truncate.rs
```

Follower recovery procedure after it learns a new leader/epoch:

```text
1. Load local leader epoch table.
2. Find last local epoch.
3. Send OffsetForLeaderEpoch to the new leader:
   - current_leader_epoch = new epoch
   - leader_epoch = last local epoch
4. Leader returns end offset for that epoch.
5. If local LEO > returned end offset, truncate to returned end offset.
6. If local LEO < returned end offset, fetch from local LEO.
7. If no common epoch exists, fall back to earliest safe truncation:
   - truncate to leader log start if local log is incompatible
   - or require snapshot/tiered catch-up later
8. Start follower fetch loop.
```

Sketch:

```rust
pub async fn reconcile_with_leader<S, C>(
    storage: S,
    client: C,
    topition: Topition,
    current_leader_epoch: i32,
) -> Result<()>
where
    S: ReplicationStorage,
    C: ReplicationClient,
{
    let state = storage.load_replica_state(&topition).await?;
    let last_local_epoch = state.leader_epoch;

    let epoch_end = client
        .offset_for_leader_epoch(
            &topition,
            current_leader_epoch,
            last_local_epoch,
        )
        .await?;

    if state.local_log_end_offset > epoch_end.end_offset {
        storage.truncate_to(&topition, epoch_end.end_offset).await?;
    }

    Ok(())
}
```

Integration test scenario:

```text

2. Partition leader = 1, epoch = 10.
3. Produce offsets 0..10; brokers 1 and 2 have them.
4. Broker 3 is partitioned.
5. Broker 1 receives extra offsets 10..15 but they are not committed.
6. Leadership moves to broker 2, epoch = 11.
7. Broker 1 becomes follower of broker 2.
8. Broker 1 asks OffsetForLeaderEpoch(epoch=10).
9. Broker 2 returns end_offset=10.
```

```rust
    partition_count: i32,
    replication_factor: usize,
```

# 19. Broker replication supervisor

Create `tansu-broker/src/replication/supervisor.rs`.

Responsibilities:

```text
- load local broker ID
- periodically read partition assignments from storage
- for each assigned partition:
  - if local broker is leader: ensure leader state/watchdog exists
  - if local broker is follower: ensure follower fetcher task exists
  - if no longer assigned: stop task
- handle cancellation
- expose test hooks for manual leadership changes
```

Sketch:

```rust
use std::collections::BTreeMap;

use tokio::task::JoinSet;
use tokio_util::sync::CancellationToken;
use tracing::{debug, info};

use tansu_storage::{
    Topition,
    replication::{ReplicationStorage, ReplicaRole},
};

use super::{client::KafkaReplicationClient, fetcher::FollowerFetcher};

#[derive(Debug)]
pub struct ReplicationSupervisor<S> {
    broker_id: i32,
    storage: S,
    cancellation: CancellationToken,
    tasks: BTreeMap<Topition, CancellationToken>,
}

impl<S> ReplicationSupervisor<S>
where
    S: ReplicationStorage,
{
    pub fn new(
        broker_id: i32,
        storage: S,
        cancellation: CancellationToken,
    ) -> Self {
        Self {
            broker_id,
            storage,
            cancellation,
            tasks: BTreeMap::new(),
        }
    }

    pub async fn run(mut self) -> tansu_storage::Result<()> {
        let mut interval = tokio::time::interval(std::time::Duration::from_secs(1));
        let mut join_set = JoinSet::new();

        loop {
            tokio::select! {
                _ = self.cancellation.cancelled() => break,

                _ = interval.tick() => {
                    self.reconcile(&mut join_set).await?;
                }

                Some(result) = join_set.join_next(), if !join_set.is_empty() => {
                    debug!(?result, "replication task exited");
                }
            }
        }

        for token in self.tasks.values() {
            token.cancel();
        }

        while join_set.join_next().await.is_some() {}

        info!("replication supervisor stopped");
        Ok(())
    }

    async fn reconcile(
        &mut self,
        join_set: &mut JoinSet<()>,
    ) -> tansu_storage::Result<()> {
        let replicas = self.storage.list_local_replicas().await?;

        for replica in replicas {
            match replica.role {
                ReplicaRole::Leader { .. } => {
                    // Ensure ISR watchdog exists.
                    self.ensure_leader_watchdog(replica.topition, join_set).await?;
                }

                ReplicaRole::Follower { leader, .. } => {
                    self.ensure_follower_fetcher(
                        replica.topition,
                        leader,
                        join_set,
                    ).await?;
                }

                ReplicaRole::Offline | ReplicaRole::None => {}
            }
        }

        Ok(())
    }

    async fn ensure_leader_watchdog(
        &mut self,
        topition: Topition,
        join_set: &mut JoinSet<()>,
    ) -> tansu_storage::Result<()> {
        // Add ISR watchdog in PR 5.
        Ok(())
    }

    async fn ensure_follower_fetcher(
        &mut self,
        topition: Topition,
        leader: i32,
        join_set: &mut JoinSet<()>,
    ) -> tansu_storage::Result<()> {
        if self.tasks.contains_key(&topition) {
            return Ok(());
        }

        let token = self.cancellation.child_token();
        self.tasks.insert(topition.clone(), token.clone());

        let client = KafkaReplicationClient::for_leader(leader).await?;
        let fetcher = FollowerFetcher {
            broker_id: self.broker_id,
            topition,
            storage: self.storage.clone(),
            client,
            cancellation: token,
            fetch_max_bytes: 1024 * 1024,
            retry_backoff: std::time::Duration::from_millis(250),
        };

        join_set.spawn(async move {
            fetcher.run().await;
        });

        Ok(())
    }
}
```

# 22. Recovery behavior

On broker restart:

```text

2. Load replica-state.json.
3. Recover each log:
   - scan segments

   - truncate partial tail
```

Checkpoint write should be atomic:

```rust
pub async fn atomic_write(path: &Path, bytes: &[u8]) -> std::io::Result<()> {
    let tmp = path.with_extension("tmp");

    {
        let mut file = tokio::fs::File::create(&tmp).await?;
        file.write_all(bytes).await?;
        file.sync_all().await?;
    }

    tokio::fs::rename(&tmp, path).await?;

    if let Some(parent) = path.parent() {
        let dir = tokio::fs::File::open(parent).await?;
        dir.sync_all().await?;
    }

    Ok(())
}
```

# 23. Compatibility errors and edge cases

Map these carefully:

| Condition                                                          | Kafka-compatible error                                                       |
| ------------------------------------------------------------------ | ---------------------------------------------------------------------------- |
| Produce sent to follower                                           | `NOT_LEADER_OR_FOLLOWER`                                                     |
| Fetch sent to broker that is not replica                           | `NOT_LEADER_OR_FOLLOWER`                                                     |
| Follower fetch has stale leader epoch                              | `FENCED_LEADER_EPOCH`                                                        |
| Follower fetch has future leader epoch                             | `UNKNOWN_LEADER_EPOCH`                                                       |
| `acks=all`, ISR below min ISR before append                        | `NOT_ENOUGH_REPLICAS`                                                        |
| `acks=all`, append happened but replication timeout/min ISR failed | `NOT_ENOUGH_REPLICAS_AFTER_APPEND`                                           |
| Fetch offset below log start or beyond LEO                         | `OFFSET_OUT_OF_RANGE`, plus modern diverging/snapshot fields where supported |
| Assigned replica missing local log                                 | `REPLICA_NOT_AVAILABLE` or `KAFKA_STORAGE_ERROR`, depending point of failure |
| Topic deleted while fetcher is running                             | `UNKNOWN_TOPIC_OR_PARTITION` or task cancellation                            |

Kafka’s protocol defines `NOT_LEADER_OR_FOLLOWER` for leader/follower-only requests and `REPLICA_NOT_AVAILABLE` for unavailable replicas. ([Apache Kafka][2])

```text
Offset invariants:
- log_start_offset <= high_watermark <= local_log_end_offset
- last_stable_offset <= high_watermark, until transactions are fully implemented

ISR invariants:

- HW is min LEO over ISR

Fetch invariants:
- consumer fetch end <= HW
- follower fetch end <= LEO
- stale epoch is fenced
- future epoch is unknown
- non-replica fetch fails

Metadata invariants:
- leader is one of replicas
- ISR is subset of replicas
- replicas list ordering is stable
- leader epoch in metadata matches storage epoch
```

```rust
pub fn validate_replica_state(state: &ReplicaState) {
```

## 29.1 Model test for replication safety

Create:

```text
tansu-storage/tests/replication_model.rs
```

Simplified model:

```rust
#[derive(Clone, Debug)]
struct ModelPartition {
    leader: i32,
    epoch: i32,
    logs: BTreeMap<i32, Vec<i64>>,
    isr: BTreeSet<i32>,
    hw: usize,
}

impl ModelPartition {
    fn append(&mut self, value: i64) {
        self.logs.get_mut(&self.leader).unwrap().push(value);
    }

    fn replicate_one(&mut self, follower: i32) {
        let leader_log = self.logs[&self.leader].clone();
        let follower_log = self.logs.get_mut(&follower).unwrap();

        if follower_log.len() < leader_log.len() {
            follower_log.push(leader_log[follower_log.len()]);
        }
    }

    fn recompute_hw(&mut self) {
        let min_len = self
            .isr
            .iter()
            .map(|broker| self.logs[broker].len())
            .min()
            .unwrap_or(self.hw);

        self.hw = self.hw.max(min_len);
    }

    fn committed_prefix(&self) -> &[i64] {
        &self.logs[&self.leader][..self.hw]
    }
}
```

Property:

```rust
proptest! {
    #[test]
    fn committed_prefix_is_present_on_all_isr_replicas(ops in operation_sequence()) {
        let mut model = ModelPartition::new();

        for op in ops {
            model.apply(op);
            model.recompute_hw();

            let committed = model.committed_prefix().to_vec();

            for broker in &model.isr {
                prop_assert!(
                    model.logs[broker].starts_with(&committed),
                    "broker {broker} does not contain committed prefix",
                );
            }
        }
    }
}
```

This catches mistakes where HW advances beyond a replica’s actual log.

```rust
    let storage = TestReplicationStorage::new_with_min_isr(2).await;
    let tp = Topition::new("topic", 0);

    storage.become_leader_for_test(&tp, 1).await;
    storage.set_isr(&tp, [1, 2]).await;

    let append = storage
        .append_as_leader(&tp, 1, test_batch("uncommitted"))
        .await
        .unwrap();

    assert_eq!(storage.high_watermark(&tp).await, 0);
    assert_eq!(append.log_end_offset, 1);

    let consumer = storage
        .fetch_for_consumer(
            &tp,
            0,
            1,
            1024 * 1024,
            IsolationLevel::ReadUncommitted,
        )
        .await
        .unwrap();

    assert!(consumer.is_empty());

    let follower = storage
        .fetch_for_follower(FetchReplicaRequest {
            replica_id: 2,
            topition: tp.clone(),
            fetch_offset: 0,
            max_bytes: 1024 * 1024,
            current_leader_epoch: 1,
            last_fetched_epoch: -1,
            follower_log_start_offset: 0,
        })
        .await
        .unwrap();

    assert_eq!(follower.records.len(), 1);
}
```

This test is extremely important. If it fails, Tansu will either expose uncommitted records to consumers or starve followers.

# 30. Practical steering and pitfalls

## 30.1 Do not compress/decompress during replication

Followers must append the same record batches the leader accepted. Re-encoding can break CRCs, producer IDs, sequence numbers, timestamps, and transaction markers.

## 30.2 Do not conflate high watermark and log-end offset

This is the most common replication bug.

```text
LEO = local physical end of log
HW  = highest offset known replicated to ISR
LSO = transaction-aware stable offset
```

Consumers read to HW/LSO. Followers fetch to LEO.

## 30.3 Do not make ISR purely offset-based

A follower that has the right offset but has not fetched in too long should be removed. Otherwise a dead follower can keep `acks=all` appearing healthy.

## 30.4 Do not let HW decrease

Even after ISR shrink, restart, or leader change, HW must be monotonic for a given leader epoch and must not expose committed-offset rollback to consumers.

## 30.5 Do not trust follower-reported progress

The leader should infer follower progress from the fetch offset and records it actually sends.

## 30.6 Do not finish Phase 11 with only private RPC replication

Use the Kafka Fetch path for follower replication. It exercises the same protocol surface real Kafka brokers use, and it prevents a fake “replication works internally but Kafka compatibility is broken” milestone.

## 30.7 Do not hide leader election inside Phase 11

Build `become_leader` and `become_follower` APIs. Use test/manual leadership transitions. Let Phase 12 own automatic election and metadata quorum.

## 30.8 Keep RF=1 behavior working

Every test added for replication should also ensure the current RF=1 mode keeps working. Tansu’s current examples use replication factor 1, so regression here would be painful. ([GitHub][1])

# 31. Final acceptance matrix

Phase 11 is finished only when this matrix is green:

| Area              | Acceptance test                                           |
| ----------------- | --------------------------------------------------------- |
| Local replica log | Append, read, recover, truncate, rebuild indexes          |
| Leader append     | Produce only accepted by leader                           |
| Follower fetch    | Follower uses Fetch with `replica_id >= 0`                |
| Consumer fetch    | Consumer uses `replica_id = -1` and reads only to HW      |
| ISR               | Followers join/leave ISR deterministically                |
| High watermark    | HW = min ISR LEO and never decreases                      |
| `acks=all`        | Waits for HW; returns correct insufficient-replica errors |
| Epochs            | Leader epoch checkpoint persisted and fenced              |
| Truncation        | Divergent follower/old leader tail is removed             |
| Metadata          | Leader/replicas/ISR/epoch visible to Kafka clients        |
| Recovery          | Restart resumes without offset corruption                 |
| Compatibility     | Kafka CLI, Java client, librdkafka produce/consume work   |
| Safety            | No unsafe Rust; workspace lint policy remains satisfied   |

The most important end-to-end test is:

```text
3 brokers
RF=3
min.insync.replicas=2
produce acks=all
kill or pause one follower
produce still succeeds
pause second follower
produce acks=all fails with NOT_ENOUGH_REPLICAS
resume followers
followers catch up
ISR returns to 3
manual leader change
old leader truncates divergent tail
consumers see only committed records
```

That test proves Phase 11 is not merely “copy records to followers,” but a real Kafka-compatible replication layer.
