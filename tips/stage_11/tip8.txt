## Phase 11 goal

Phase 11 should end with Tansu supporting **broker-owned partition replicas** with Kafka-compatible replication semantics: leaders accept writes, followers replicate with follower Fetch requests, ISR is maintained, high watermark advances safely, leader epochs prevent divergent logs, and followers truncate correctly after leadership changes.

The key architectural point: Tansu currently advertises itself as a **stateless Kafka-compatible broker with pluggable storage** such as PostgreSQL, SQLite, S3, and memory; the docs explicitly frame this as avoiding broker-replicated storage durability. The README examples also currently show node `111` as both leader and ISR for partitions, and `tansu-broker` still has a fixed `NODE_ID = 111`. Phase 11 therefore should not be bolted onto the existing “shared durable storage” mode as an afterthought. It should introduce an explicit **replicated local-log mode** while preserving the current storage engines as “shared storage” mode. ([GitHub][1]) 

A second key point: Tansu already has the protocol foundation. The broker routes Kafka API requests through `tansu-broker/src/service/storage.rs`, while `tansu-storage` exposes `FetchService`, `ProduceService`, `MetadataService`, and a `Storage` abstraction. That is great leverage, but the current Produce path drops replication-critical fields like `acks` and timeout before calling storage, and the Fetch path does not propagate follower identity or leader epoch into storage. Those two services must change.

# 1. Define Phase 11 completion semantics

Treat Phase 11 as complete only when these semantics are true.

## 1.1 Partition role semantics

For every `Topition`:

```text
role ∈ {Leader, Follower, Offline}
leader_epoch is monotonically increasing
partition_epoch is monotonically increasing on assignment/ISR/leader metadata changes
replicas is the assigned replica set
isr is the subset allowed to elect a clean leader
leader ∈ replicas or None
isr ⊆ replicas
```

Only the **current leader** accepts client Produce. Followers reject Produce with `NOT_LEADER_OR_FOLLOWER` and include the current leader when the protocol version supports it.

Consumer Fetch should be served only by the leader for Phase 11 unless you intentionally implement Kafka’s follower-fetch-by-rack behavior later. Follower Fetch, however, must be accepted by leaders when `replica_id >= 0`.

## 1.2 High watermark semantics

Use Kafka’s “next offset” convention:

```text
log_start_offset <= high_watermark <= log_end_offset
```

Consumers may only see records with offsets `< high_watermark`.

For a leader:

```text
leader_high_watermark =
    min(log_end_offset(replica) for every replica in ISR)
```

The leader’s own log end offset counts as its replica progress. For followers, the high watermark is learned from leader Fetch responses and must be persisted or checkpointed.

Important nuance: do **not** advance a follower’s replicated offset on the leader merely because the leader sent records to that follower. A Kafka-style leader should advance follower progress when the follower’s **next Fetch request** asks from the offset after records it has already appended. In other words, the follower’s requested `fetch_offset` is the acknowledgement. This avoids acknowledging data the follower has not actually persisted.

## 1.3 ISR semantics

A follower is in-sync when:

```text
same assigned partition
same or compatible leader epoch
has fetched from the current leader recently enough
has caught up to the leader log end offset or is within the allowed lag window
```

Kafka commonly removes followers from ISR when they have not fetched or have not caught up within `replica.lag.time.max.ms`; Red Hat’s Kafka docs describe the default as `30000ms` and also document high-watermark checkpointing defaults. Use those as initial compatibility defaults unless Tansu chooses a different documented default. ([Red Hat Documentation][2])

## 1.4 Leader epoch semantics

Every leadership change increments `leader_epoch`. Kafka’s leader-epoch mechanism exists to prevent data loss and log divergence during leader changes; KIP-101 defines leader epoch as a monotonically increasing 32-bit value and describes using an epoch-to-start-offset sequence so followers can ask the new leader where to truncate. ([Apache Wiki][3])

For Tansu Phase 11:

```text
on become leader:
    append leader_epoch -> current log_end_offset
    flush leader epoch checkpoint
    stamp all future batches with this leader_epoch

on become follower:
    stop accepting writes
    stop stale fetchers
    reconcile local log with leader using OffsetForLeaderEpoch
    truncate divergent suffix
    start follower fetch loop
```

## 1.5 Produce acknowledgement semantics

For Produce:

```text
acks = 0:
    do not wait for local replication response; Kafka normally has no ProduceResponse body.

acks = 1:
    respond after leader local append and durability policy is satisfied.

acks = -1 / all:
    respond only when the appended end offset <= high_watermark,
    subject to min.insync.replicas and request timeout.
```

For `acks=all`:

```text
before append:
    if ISR size < min.insync.replicas:
        return NOT_ENOUGH_REPLICAS

after append:
    wait until high_watermark >= appended_end_offset
    if timeout:
        return NOT_ENOUGH_REPLICAS_AFTER_APPEND
```

Kafka producer docs describe `acks=all` as waiting for the full ISR to acknowledge, giving the strongest guarantee while at least one ISR remains alive. ([Red Hat Documentation][2])

# 2. Recommended architecture: add a replicated local-log mode

Do **not** try to turn PostgreSQL/S3/memory shared-storage mode into Kafka replication. Keep those modes working as they do today.

Add:

```text
storage=replicated://...
replication.mode=broker
log.dir=/var/lib/tansu/kafka-log
metadata.store=...
```

The replicated mode should own local segment files per broker and use existing shared storage only for metadata if needed.

Suggested modes:

```rust
pub enum StorageMode {
    Shared,        // existing memory/postgres/sqlite/s3/turso/slatedb behavior
    ReplicatedLog, // new broker-owned partition replicas
}
```

In `Shared` mode, `replication_factor` may still be accepted for compatibility, but Tansu can continue exposing the current synthetic metadata behavior until replicated mode is enabled.

In `ReplicatedLog` mode, `replication_factor` means actual broker replicas.

### Create `tansu-broker/src/replication/leader.rs`

Leader-side ISR and high-watermark logic.

```rust
use std::{collections::{BTreeMap, BTreeSet}, time::Instant};
use tansu_storage::replica::BrokerId;

#[derive(Clone, Copy, Debug, Eq, PartialEq)]
pub struct HighWatermarkResult {
    pub old: i64,
    pub new: i64,
    pub advanced: bool,
}

pub fn compute_high_watermark(
    leader: BrokerId,
    leader_log_end_offset: i64,
    log_start_offset: i64,
    isr: &BTreeSet<BrokerId>,
    follower_end_offsets: &BTreeMap<BrokerId, i64>,
) -> i64 {
    let min_isr_end = isr
        .iter()
        .map(|broker| {
            if *broker == leader {
                leader_log_end_offset
            } else {
                follower_end_offsets.get(broker).copied().unwrap_or(log_start_offset)
            }
        })
        .min()
        .unwrap_or(log_start_offset);

    min_isr_end.clamp(log_start_offset, leader_log_end_offset)
}

pub fn maybe_advance_high_watermark(
    current_hw: i64,
    computed_hw: i64,
) -> HighWatermarkResult {
    let new = computed_hw.max(current_hw);
    HighWatermarkResult {
        old: current_hw,
        new,
        advanced: new > current_hw,
    }
}
```

ISR update:

```rust
use std::time::Duration;
use tansu_storage::replica::LeaderEpoch;

pub fn should_remove_from_isr(
    now: Instant,
    progress: &FollowerProgress,
    leader_epoch: LeaderEpoch,
    leader_log_end_offset: i64,
    max_lag: Duration,
) -> bool {
    progress.last_fetch_leader_epoch != leader_epoch
        || now.duration_since(progress.last_fetch_at) > max_lag
        || progress.log_end_offset < leader_log_end_offset
            && now.duration_since(progress.last_fetch_at) > max_lag
}

pub fn can_add_to_isr(
    progress: &FollowerProgress,
    leader_epoch: LeaderEpoch,
    leader_log_end_offset: i64,
) -> bool {
    progress.last_fetch_leader_epoch == leader_epoch
        && progress.log_end_offset >= leader_log_end_offset
}
```

### Create `tansu-broker/src/replication/fetcher.rs`

Follower-side fetch loop.

```rust
use std::time::Duration;
use tokio_util::sync::CancellationToken;
use tracing::{debug, error, instrument};

use tansu_storage::{
    Topition,
    replica::{BrokerId, LeaderEpoch, ReplicaLog},
};

use crate::{
    interbroker::InterBrokerClientPool,
    replication::config::ReplicationConfig,
};

#[derive(Clone, Debug)]
pub struct FollowerFetcher<L> {
    pub local_broker: BrokerId,
    pub leader: BrokerId,
    pub topition: Topition,
    pub leader_epoch: LeaderEpoch,
    pub log: L,
    pub clients: InterBrokerClientPool,
    pub config: ReplicationConfig,
    pub cancellation: CancellationToken,
}

impl<L> FollowerFetcher<L>
where
    L: ReplicaLog,
{
    #[instrument(skip_all, fields(topic = %self.topition.topic(), partition = self.topition.partition()))]
    pub async fn run(self) {
        let mut backoff = self.config.fetch_backoff_initial;

        loop {
            if self.cancellation.is_cancelled() {
                break;
            }

            match self.replicate_once().await {
                Ok(()) => {
                    backoff = self.config.fetch_backoff_initial;
                }
                Err(error) => {
                    error!(?error, ?backoff, "replica fetch failed");
                    tokio::time::sleep(backoff).await;
                    backoff = (backoff * 2).min(self.config.fetch_backoff_max);
                }
            }
        }
    }

    async fn replicate_once(&self) -> tansu_storage::Result<()> {
        self.reconcile_epoch().await?;

        let watermarks = self.log.local_watermarks(&self.topition).await?;
        let fetch_offset = watermarks.log_end_offset;

        let response = self.clients
            .for_broker(self.leader)
            .await?
            .fetch_as_replica(
                self.local_broker,
                self.topition.clone(),
                self.leader_epoch,
                fetch_offset,
                self.config.replica_fetch_min_bytes,
                self.config.replica_fetch_max_bytes,
                self.config.replica_fetch_wait_max,
            )
            .await?;

        self.log
            .append_as_follower(
                &self.topition,
                self.leader_epoch,
                response.batches,
                response.high_watermark,
            )
            .await?;

        Ok(())
    }

    async fn reconcile_epoch(&self) -> tansu_storage::Result<()> {
        let local = self.log.latest_epoch(&self.topition).await?;

        let leader_end = self.clients
            .for_broker(self.leader)
            .await?
            .offset_for_leader_epoch(
                self.topition.clone(),
                self.leader_epoch,
                local,
            )
            .await?;

        let local_leo = self.log.local_watermarks(&self.topition).await?.log_end_offset;

        if leader_end.end_offset >= 0 && leader_end.end_offset < local_leo {
            self.log.truncate_to(&self.topition, leader_end.end_offset).await?;
        }

        Ok(())
    }
}
```

```rust
    log_end_offset: i64,
}

impl<L> ReplicationManager<L>
where
    L: ReplicaLog,
{
    pub async fn start(&self) -> tansu_storage::Result<()> {
        // Start metadata watcher, ISR timer, high-watermark checkpoint timer.
        Ok(())
    }

    pub async fn shutdown(&self) -> tansu_storage::Result<()> {
        self.cancellation.cancel();
        Ok(())
    }

    pub async fn apply_event(&self, event: ReplicationEvent) -> tansu_storage::Result<()> {
        match event {
            ReplicationEvent::BecomeLeader { topition, leader_epoch } => {
                self.log.become_leader(&topition, leader_epoch).await?;
                // update state, stop follower fetch task, enable Produce
            }
            ReplicationEvent::BecomeFollower { topition, leader, leader_epoch } => {
                self.log.become_follower(&topition, leader_epoch).await?;
                // spawn follower fetch task
            }
            _ => {}
        }

        Ok(())
    }

    pub async fn wait_for_high_watermark(
        &self,
        topition: &Topition,
        target_next_offset: i64,
        timeout: Duration,
    ) -> tansu_storage::Result<()> {
        let deadline = tokio::time::Instant::now() + timeout;

        loop {
            {
                let inner = self.inner.lock().await;
                if let Some(partition) = inner.partitions.get(topition) {
                    if partition.high_watermark >= target_next_offset {
                        return Ok(());
                    }
                }
            }

            tokio::select! {
                _ = self.notify_hwm.notified() => {}
                _ = tokio::time::sleep_until(deadline) => {
                    return Err(tansu_storage::Error::Api(
                        tansu_sans_io::ErrorCode::NotEnoughReplicasAfterAppend
                    ));
                }
            }
        }
    }

    pub async fn notify_high_watermark_advanced(&self) {
        self.notify_hwm.notify_waiters();
    }
}
```

```rust
            leader_epoch: requested_epoch,
```

```rust
        OffsetForLeaderTopicResult,
        OffsetForLeaderPartitionResult,

};
use tracing::instrument;

use crate::{
    Error, Result, Storage, Topition,
    replica::LeaderEpoch,
};

#[derive(Clone, Copy, Debug, Default, Eq, Hash, Ord, PartialEq, PartialOrd)]
pub struct OffsetForLeaderEpochService;

impl ApiKey for OffsetForLeaderEpochService {
    const KEY: i16 = OffsetForLeaderEpochRequest::KEY;
}

impl<G> Service<G, OffsetForLeaderEpochRequest> for OffsetForLeaderEpochService
where
    G: Storage,
{
    type Response = OffsetForLeaderEpochResponse;
    type Error = Error;

    #[instrument(skip(ctx, req))]
    async fn serve(
        &self,
        ctx: Context<G>,
        req: OffsetForLeaderEpochRequest,
    ) -> Result<Self::Response> {
        let mut topics = vec![];

        for topic in req.topics.unwrap_or_default() {
            let topic_name = topic.topic.clone();
            let mut partitions = vec![];

            for partition in topic.partitions.unwrap_or_default() {
                let tp = Topition::new(&topic_name, partition.partition);

                let current = partition
                    .current_leader_epoch
                    .map(LeaderEpoch);

                let requested = LeaderEpoch(partition.leader_epoch);

                let result = ctx
                    .state()
                    .epoch_end_offset(&tp, requested, current)
                    .await;

                let response_partition = match result {
                    Ok(epoch_end) => OffsetForLeaderPartitionResult::default()
                        .partition(partition.partition)
                        .error_code(ErrorCode::None.into())
                        .leader_epoch(epoch_end.leader_epoch.0)
                        .end_offset(epoch_end.end_offset),

                    Err(Error::FencedLeaderEpoch { .. }) => OffsetForLeaderPartitionResult::default()
                        .partition(partition.partition)
                        .error_code(ErrorCode::FencedLeaderEpoch.into())
                        .leader_epoch(-1)
                        .end_offset(-1),

                    Err(Error::UnknownLeaderEpoch { .. }) => OffsetForLeaderPartitionResult::default()
                        .partition(partition.partition)
                        .error_code(ErrorCode::UnknownLeaderEpoch.into())
                        .leader_epoch(-1)
                        .end_offset(-1),

                    Err(Error::Api(code)) => OffsetForLeaderPartitionResult::default()
                        .partition(partition.partition)
                        .error_code(code.into())
                        .leader_epoch(-1)
                        .end_offset(-1),

                    Err(_) => OffsetForLeaderPartitionResult::default()
                        .partition(partition.partition)
                        .error_code(ErrorCode::UnknownServerError.into())
                        .leader_epoch(-1)
                        .end_offset(-1),
                };

                partitions.push(response_partition);
            }

            topics.push(
                OffsetForLeaderTopicResult::default()
                    .topic(topic_name)
                    .partitions(Some(partitions)),
            );
        }

        Ok(OffsetForLeaderEpochResponse::default()
            .throttle_time_ms(Some(0))
            .topics(Some(topics)))
    }
}
```

```rust
async fn fetch_partition<G>(
    &self,
    ctx: &Context<G>,
    max_wait_ms: Duration,
    min_bytes: u32,
    max_bytes: &mut u32,
    isolation: IsolationLevel,
    topic: &str,
    fetch_partition: &FetchPartition,
    visibility: FetchVisibility,
) -> Result<PartitionData>
where
    G: Storage,
{
    let partition_index = fetch_partition.partition;
    let tp = Topition::new(topic, partition_index);

    if let FetchVisibility::Follower { replica_id } = visibility {
        // This is the ACK signal: follower is asking from fetch_offset,
        // meaning it has appended through fetch_offset - 1.
        ctx.state()
            .update_follower_fetch_progress(
                &tp,
                replica_id,
                fetch_partition.fetch_offset,
                fetch_partition.current_leader_epoch.map(LeaderEpoch),
            )
            .await?;
    }

    let mut offset = fetch_partition.fetch_offset;
    let mut batches = Vec::new();

    loop {
        if *max_bytes == 0 {
            break;
        }

        let mut fetched = ctx
            .state()
            .fetch_with_visibility(
                &tp,
                offset,
                min_bytes,
                *max_bytes,
                isolation,
                visibility,
                fetch_partition.current_leader_epoch.map(LeaderEpoch),
            )
            .await?;

        // existing byte accounting...
    }

    let offset_stage = ctx.state().offset_stage(&tp).await?;

    Ok(PartitionData::default()
        .partition_index(partition_index)
        .error_code(ErrorCode::None.into())
        .high_watermark(offset_stage.high_watermark())
        .last_stable_offset(Some(offset_stage.last_stable()))
        .log_start_offset(Some(offset_stage.log_start()))
        .records(if batches.is_empty() { None } else { Some(Frame { batches }) }))
}
```

# 4. Leader/follower replication flow

## 4.1 Produce flow on leader

```text
client Produce
  -> ProduceService
  -> Storage::produce_with_options
  -> validate this broker is leader
  -> validate current_leader_epoch
  -> validate ISR >= min.insync.replicas for acks=all
  -> append local batch with leader_epoch
  -> update leader LEO
  -> acks=1: respond
  -> acks=all: wait for HWM >= appended_end_offset
```

Code sketch:

```rust
async fn produce_replicated(
    storage: &impl Storage,
    replication: &ReplicationManager<impl ReplicaLog>,
    transaction_id: Option<&str>,
    tp: &Topition,
    batch: deflated::Batch,
    options: ProduceOptions,
) -> Result<ProduceResult> {
    let result = storage
        .produce_with_options(transaction_id, tp, batch, options.clone())
        .await?;

    if result.error != ErrorCode::None {
        return Ok(result);
    }

    let appended_end_offset = result.last_offset + 1;

    match options.required_acks {
        0 | 1 => Ok(result),

        -1 => {
            replication
                .wait_for_high_watermark(tp, appended_end_offset, options.timeout)
                .await?;

            Ok(result)
        }

        other => Err(Error::Message(format!("unsupported required_acks={other}"))),
    }
}
```

Steering: the current `ProduceService` state is only `Storage`; to access `ReplicationManager`, either:

1. Fold waiting into `ReplicatedStorage::produce_with_options()`, or
2. Change broker service state from `S` to a new `BrokerServiceState<S>` that contains both storage and replication manager.

For Phase 11, option 1 is easier. For long-term design, option 2 is cleaner.

---

## 4.2 Follower Fetch flow

```text
follower starts task for each assigned follower partition
  -> reconcile leader epoch
  -> local LEO = log_end_offset
  -> send Fetch(replica_id=local_broker, fetch_offset=local LEO)
  -> leader updates follower progress to fetch_offset
  -> leader returns records from fetch_offset to leader LEO, plus leader HW
  -> follower appends records
  -> follower updates local HW from response
  -> loop
```

Do not forget the ACK nuance:

```text
Leader should update follower progress to the fetch_offset in the request,
not to the end offset in the response.
```

That way, a follower only acknowledges data after it has appended it and asks for the next offset.

## 4.3 High watermark advancement

Leader-side algorithm:

```rust
async fn on_follower_fetch(
    state: &mut PartitionReplicaState,
    follower: BrokerId,
    fetch_offset: i64,
    now: Instant,
) -> Option<i64> {
    state.follower_progress.insert(follower, FollowerProgress {
        broker_id: follower,
        log_end_offset: fetch_offset,
        last_fetch_at: now,
        last_fetch_leader_epoch: state.leader_epoch,
    });

    let follower_offsets = state.follower_progress
        .iter()
        .map(|(broker, progress)| (*broker, progress.log_end_offset))
        .collect();

    let computed = compute_high_watermark(
        state.leader.unwrap(),
        state.local_log_end_offset,
        state.local_log_start_offset,
        &state.isr,
        &follower_offsets,
    );

    if computed > state.high_watermark {
        state.high_watermark = computed;
        Some(computed)
    } else {
        None
    }
}
```

Persist HWM periodically and after meaningful advancement:


## 4.4 ISR maintenance

Run a periodic task per leader partition:

```rust
async fn maintain_isr_for_partition(
    state: &mut PartitionReplicaState,
    now: Instant,
    config: &ReplicationConfig,
) -> IsrChange {
    let mut removed = vec![];
    let mut added = vec![];

    for broker in state.replicas.iter().copied() {
        if broker == state.leader.unwrap() {
            continue;
        }

        let progress = state.follower_progress.get(&broker);

        if state.isr.contains(&broker) {
            let remove = progress
                .map(|p| should_remove_from_isr(
                    now,
                    p,
                    state.leader_epoch,
                    state.local_log_end_offset,
                    config.replica_lag_time_max,
                ))
                .unwrap_or(true);

            if remove {
                state.isr.remove(&broker);
                removed.push(broker);
            }
        } else if let Some(progress) = progress {
            if can_add_to_isr(progress, state.leader_epoch, state.local_log_end_offset) {
                state.isr.insert(broker);
                added.push(broker);
            }
        }
    }

    IsrChange { added, removed }
}
```

# 5. Leader epoch and truncation details

## 5.1 Epoch checkpoint format

Use Kafka-style simple text or a compact binary format. Text is easier to debug:

```text
version: 0
entries: N
epoch start_offset
epoch start_offset
...
```

Example:

```text
0
3
5 0
6 120
7 190
```

Meaning:

```text
epoch 5 starts at offset 0
epoch 6 starts at offset 120
epoch 7 starts at offset 190
```

End offset lookup:

```text
requested epoch 5 => 120
requested epoch 6 => 190
requested epoch 7 => current log_end_offset
unknown epoch => -1
```

## 5.2 Follower truncation algorithm

```rust
async fn truncate_to_leader(
    log: &impl ReplicaLog,
    client: &InterBrokerClient,
    topition: &Topition,
    current_leader_epoch: LeaderEpoch,
) -> Result<()> {
    let mut epoch = log.latest_epoch(topition).await?;

    loop {
        let leader_end = client
            .offset_for_leader_epoch(
                topition.clone(),
                current_leader_epoch,
                epoch,
            )
            .await?;

        if leader_end.end_offset >= 0 {
            let local_leo = log.local_watermarks(topition).await?.log_end_offset;

            if leader_end.end_offset < local_leo {
                log.truncate_to(topition, leader_end.end_offset).await?;
            }

            return Ok(());
        }

        if epoch.0 <= 0 {
            log.truncate_to(topition, 0).await?;
            return Ok(());
        }

        epoch = LeaderEpoch(epoch.0 - 1);
    }
}
```

## 5.3 Become-leader algorithm

```text
1. confirm this broker is in assigned replicas
2. confirm either:
   a. clean election: this broker was in ISR, or
   b. unclean election explicitly enabled
3. stop follower fetch task for partition
4. open local log
5. append leader_epoch -> log_end_offset checkpoint
6. flush epoch checkpoint
7. set role=Leader
8. initialize follower progress:
   leader => local LEO
   each follower => unknown/log_start
9. compute initial HWM:
   if old HW is persisted, start from it
   do not move HW backwards
10. accept Produce
```

## 5.4 Become-follower algorithm

```text
1. fence local Produce immediately
2. stop leader-side ISR timers for partition
3. start epoch reconciliation against new leader
4. truncate local divergent suffix
5. set role=Follower
6. start follower fetch task
7. expose only local high watermark to consumers if follower reads are later enabled
```

# 6. Metadata and protocol correctness

## 6.1 Metadata response fields

For every partition:

```text
leader_id = current leader or -1
leader_epoch = current leader epoch
replica_nodes = assigned replicas
isr_nodes = in-sync replicas
offline_replicas = assigned replicas not alive / unavailable
```

Kafka metadata examples expose partition fields such as `replicas`, `isr`, `leader`, `leaderEpoch`, and `partitionEpoch`; Tansu should preserve equivalent information internally even when older Metadata API versions only expose a subset. ([Apache Kafka][6])

## 6.2 Error mapping

Use these mappings consistently:

| Internal condition                                 | Kafka error                              |
| -------------------------------------------------- | ---------------------------------------- |
| Broker is not current leader or follower           | `NOT_LEADER_OR_FOLLOWER`                 |
| Request leader epoch < current epoch               | `FENCED_LEADER_EPOCH`                    |
| Request leader epoch > current epoch               | `UNKNOWN_LEADER_EPOCH`                   |
| Assigned replica missing                           | `REPLICA_NOT_AVAILABLE`                  |
| Topic/partition missing                            | `UNKNOWN_TOPIC_OR_PARTITION`             |
| ISR below minimum before append                    | `NOT_ENOUGH_REPLICAS`                    |
| ISR/HW did not reach append before timeout         | `NOT_ENOUGH_REPLICAS_AFTER_APPEND`       |
| Fetch offset below log start or beyond valid range | `OFFSET_OUT_OF_RANGE`                    |
| Truncation below HW attempted                      | internal fatal or `UNKNOWN_SERVER_ERROR` |

## 6.3 Fetch visibility

```rust
match visibility {
    FetchVisibility::Consumer => {
        max_visible = if isolation == ReadCommitted {
            last_stable_offset
        } else {
            high_watermark
        };
    }
    FetchVisibility::Follower { .. } => {
        max_visible = log_end_offset;
    }
}
```

This split is mandatory. If followers only fetch to HW, replicas can never catch up to uncommitted leader data, and `acks=all` will stall.

# 10. Phase 11 “done” checklist

Phase 11 is complete when all of this passes:

```text
[ ] Broker has configurable broker_id; fixed NODE_ID is only a default.
[ ] Replicated local-log storage mode exists and is opt-in.
[ ] Partition assignments contain leader, replicas, ISR, leader_epoch, partition_epoch.
[ ] Metadata and DescribeTopicPartitions return true leader/replica/ISR data.
[ ] Produce only succeeds on current leader.
[ ] Produce validates current leader epoch where supplied.
[ ] Produce acks=all waits for HWM and respects min.insync.replicas.
[ ] Fetch distinguishes consumer fetch from follower fetch.
[ ] Consumer Fetch is capped at high watermark / last stable offset.
[ ] Follower Fetch can read to leader log end offset.
[ ] Follower Fetch updates follower progress using requested fetch_offset.
[ ] Leader recomputes and persists high watermark.
[ ] ISR removes lagging followers and re-adds caught-up followers.
[ ] Leader epoch checkpoint is created, flushed, recovered, and truncated.
[ ] OffsetForLeaderEpoch API is routed and implemented.
[ ] Followers truncate divergent logs on leader change.
[ ] Clean leader failover preserves all acknowledged records.
[ ] Unclean leader election is disabled by default.
[ ] Multi-broker integration tests pass with real TCP.
[ ] Java Kafka client and librdkafka can produce/consume through failover.
[ ] Existing shared storage modes still pass their current tests.
```

The most important implementation steering is this: **do the protocol-visible changes first**—Produce options, Fetch visibility, OffsetForLeaderEpoch routing, true Metadata fields—then wire the replication manager and local log behind them. That keeps every sub-step testable with Kafka clients instead of building a large hidden subsystem that only fails at the end.
