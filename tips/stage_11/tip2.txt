## Phase 11 goal: real Kafka-style replication, not “shared-storage replication”

Tansu’s current design is intentionally different from Kafka: the docs say brokers are stateless, do not replicate storage, have no ISR concept, and every broker can act as leader for every topic partition. The README also documents node `111` as the leader/ISR and says all brokers are node `111`. ([Tansu][1]) The code lines up with that: the CLI broker builder currently calls `.node_id(NODE_ID)`, and `NODE_ID` is defined as `111`.  

For Phase 11, do **not** fake Kafka replication by having multiple brokers read/write the same object-store path. Kafka replication means each partition has one leader, zero or more followers, per-replica log end offsets, an ISR, a high watermark, leader epochs, and follower truncation after leader change. Kafka’s own replication model is primary-backup: one leader per partition, followers replicate from the leader, the ISR tracks fully caught-up replicas, LEO is the tail of each replica log, and HW is the committed offset visible to consumers. ([Apache Wiki][2]) Kafka’s `acks=all` and `min.insync.replicas` semantics also require enough in-sync replicas before acknowledged writes are considered safe. ([Apache Wiki][3]) Leader epochs are used to detect stale leaders and truncate divergent logs after leader changes. ([Apache Wiki][4])

So Phase 11 should introduce a new **replicated broker mode** while preserving today’s stateless/shared-storage mode as a compatibility mode.

---

# Definition of “Phase 11 complete”

Phase 11 is done when Tansu can run a real multi-broker cluster with these properties:

1. **Unique brokers**: brokers have distinct `node.id` values and distinct advertised listeners.
2. **Replica assignment**: topic partitions have replica sets like `[1, 2, 3]`, not `[111, 111, 111]`.
3. **Leader-only produce**: only the partition leader accepts Produce for a partition. Followers return `NOT_LEADER_OR_FOLLOWER` or the version-appropriate Kafka error, with current leader metadata where supported.
4. **Follower replication**: followers fetch from leaders, append exact record batches at exact leader offsets, and advance their local LEO.
5. **ISR tracking**: leaders track follower fetch progress and remove lagging/dead replicas from ISR.
6. **High watermark advancement**: HW advances monotonically to the minimum LEO of replicas in ISR, subject to `min.insync.replicas`.
7. **Produce acks**:

   * `acks=0`: no wait.
   * `acks=1`: leader local append only.
   * `acks=all/-1`: wait until the produced offset is replicated to the required ISR set, or return `NOT_ENOUGH_REPLICAS` / `NOT_ENOUGH_REPLICAS_AFTER_APPEND`.
8. **Consumer visibility**: Fetch for clients returns data only up to HW, not merely leader LEO.
9. **Follower fetch**: inter-broker/follower Fetch can read beyond HW up to leader LEO.
10. **Leader epochs**: each leader change increments and persists a partition leader epoch.
11. **Truncation on leader change**: a broker becoming follower reconciles using leader epoch checkpoints and truncates divergent uncommitted tail before appending from the new leader.
12. **Restart recovery**: brokers recover local replica logs, leader epoch checkpoints, HW checkpoints, and replica state without offset corruption.
13. **Metadata correctness**: Kafka CLI/AdminClient sees correct `Leader`, `Replicas`, `Isr`, `OfflineReplicas`, and `LeaderEpoch` where the protocol version exposes it.
14. **Single-node compatibility**: existing single-broker behavior and tests continue to pass with `replication.factor=1`.

# Critical design decision

## Add `replication.mode`

Tansu currently makes durability the storage engine’s job. Phase 11 should not silently break that model. Add an explicit broker config:

```text
--replication-mode shared-storage | kafka
```

Recommended defaults:

```text
shared-storage   # current Tansu behavior
kafka            # new Phase 11 mode
```

In `shared-storage` mode, keep today’s stateless behavior for users who want it.

In `kafka` mode:

* each broker stores its **own local replica log**;
* metadata/coordination can still use the existing storage engine until Phase 12 replaces it with a real controller/quorum;
* followers replicate over broker-to-broker network;
* metadata advertises real leaders, replicas, and ISR.

This split is important because `DynoStore` currently has one global `clusters/{cluster}/topics/{topic}/partitions/{partition}/records/{offset}.batch` layout and a single `Watermark` per partition, which is perfect for Tansu’s current shared-storage model but not sufficient for Kafka-style per-broker replica logs.

# Current repo areas Phase 11 must change

Tansu already has useful hooks:

* `Broker` has `node_id`, `cluster_id`, `incarnation_id`, listener URLs, storage, and maintenance loop. Broker startup calls `register()` and maintenance calls `storage.maintain(...)`, which is a good place to plug in broker heartbeats, leader health checks, and minimal leader election before Phase 12. 
* `Storage` already exposes `produce`, `fetch`, `offset_stage`, `metadata`, `brokers`, `node`, `advertised_listener`, and `maintain`, but not replica progress, ISR, leader epoch, follower append, truncation, or per-replica log state. 
* `RequestChannelService` mirrors `Storage` through `Request`/`Response` enums, so every new replication method added to `Storage` must also be added there. 
* `FetchService` currently calls `storage.fetch(...)` and then `offset_stage(...)`, returning HW/LSO/log-start, but it does not distinguish client fetch from follower fetch or enforce leader epochs. 
* `ProduceService` currently calls `storage.produce(...)` per batch and returns success/error, but it does not check partition leadership, ISR, min ISR, or wait for follower acknowledgements. 
* `MetadataService` delegates metadata construction to storage, so storage backends must emit real leader/replica/ISR state. 
* `null.rs` and existing metadata implementations currently emit one broker and duplicate `self.node` as every replica/ISR member, which must be fixed in replicated mode.

## Step 0: Introduce explicit replicated mode

Do this first so Phase 11 can be merged incrementally without breaking current Tansu users.

### CLI change

In `tansu-cli/src/cli/broker.rs`, add:

```rust
#[derive(Clone, Copy, Debug, Eq, PartialEq, clap::ValueEnum)]
pub enum ReplicationMode {
    /// Current Tansu behavior: brokers are stateless and use shared storage.
    SharedStorage,

    /// Kafka-like behavior: per-broker replica logs, leaders, followers, ISR.
    Kafka,
```

Then extend `Arg`:

Then replace:


with:

```rust

```

### Broker default ID change

In `tansu-broker/src/lib.rs`:

```rust
/// Compatibility default for current single-node/shared-storage Tansu mode.
/// In Kafka replication mode, users must supply unique node IDs.
pub const DEFAULT_NODE_ID: i32 = 111;
```

Keep `NODE_ID` temporarily as deprecated if many tests import it:

```rust

```

### Steering

Do **not** make `--node-id` optional in `replication-mode=kafka` if multiple brokers are configured. If two brokers start with the same node ID, fail startup loudly:

```rust
if replication_mode == ReplicationMode::Kafka && self.node_id == DEFAULT_NODE_ID {
    tracing::warn!(
        node_id = self.node_id,
        "using compatibility node id in kafka replication mode; this is only valid for a one-broker cluster"
    );
}
```

Later, broker registration will reject duplicate live broker IDs.

That broker-local prefix is the key difference from current shared-storage `records/{offset}.batch`.

## Step 2: Add a `ReplicaLog` abstraction

Create `tansu-storage/src/replication/log.rs`.

This avoids bolting all replication logic into `Storage::produce`/`Storage::fetch`.

```rust
use async_trait::async_trait;
use tansu_sans_io::{IsolationLevel, record::deflated};

use crate::{Result, Topition};
use super::types::{LeaderEpoch, Offset};

#[async_trait]
pub trait ReplicaLog: Clone + Send + Sync + 'static {
    async fn log_start_offset(&self, tp: &Topition) -> Result<Offset>;

    async fn log_end_offset(&self, tp: &Topition) -> Result<Offset>;

    async fn high_watermark(&self, tp: &Topition) -> Result<Offset>;

    async fn set_high_watermark(&self, tp: &Topition, hw: Offset) -> Result<()>;

    /// Leader append assigns offsets and stamps the current leader epoch.
    async fn append_as_leader(
        &self,
        tp: &Topition,
        leader_epoch: LeaderEpoch,
        batch: deflated::Batch,
    ) -> Result<Offset>;

    /// Follower append preserves the leader's base offset and leader epoch.
    async fn append_as_follower(
        &self,
        tp: &Topition,
        leader_epoch: LeaderEpoch,
        expected_base_offset: Offset,
        batch: deflated::Batch,
    ) -> Result<()>;

    /// Client reads must be capped by HW; follower reads can read to LEO.
    async fn read(
        &self,
        tp: &Topition,
        offset: Offset,
        max_bytes: u32,
        isolation: IsolationLevel,
        visibility: FetchVisibility,
    ) -> Result<Vec<deflated::Batch>>;

    async fn truncate_to(&self, tp: &Topition, offset: Offset) -> Result<()>;

    async fn leader_epoch_checkpoint(
        &self,
        tp: &Topition,
    ) -> Result<Vec<super::types::LeaderEpochCheckpoint>>;

    async fn append_leader_epoch_checkpoint(
        &self,
        tp: &Topition,
        leader_epoch: LeaderEpoch,
        start_offset: Offset,
    ) -> Result<()>;

    async fn end_offset_for_leader_epoch(
        &self,
        tp: &Topition,
        leader_epoch: LeaderEpoch,
    ) -> Result<Offset>;
}

#[derive(Clone, Copy, Debug, Eq, PartialEq)]
pub enum FetchVisibility {
    ClientCommitted,
    FollowerReplication,
}
```

### Steering

Follower append must be strict:

```rust
let local_leo = self.log_end_offset(tp).await?;
if local_leo != expected_base_offset {
    return Err(Error::Message(format!(
        "follower append offset mismatch: local_leo={local_leo}, expected={expected_base_offset}"
    )));
}
```

Never “renumber” follower batches. Kafka followers store the same offsets as the leader.

### Why this belongs in `Storage`

Tansu already routes API services through `Storage`. `RequestChannelService` wraps storage calls using a `Request`/`Response` enum. ## Step 6: Replica assignment for topic creation

In `tansu-storage/src/replication/assignment.rs`:

```rust
pub fn assign_replicas(
    brokers: &[BrokerEndpoint],
    partitions: i32,
    replication_factor: i16,
) -> Result<BTreeMap<i32, Vec<i32>>> {
    if replication_factor <= 0 {
        return Err(Error::Api(ErrorCode::InvalidReplicationFactor));
    }

    let rf = replication_factor as usize;

    if brokers.len() < rf {
        return Err(Error::Api(ErrorCode::InvalidReplicationFactor));
    }

    let broker_ids = brokers.iter().map(|b| b.broker_id).collect::<Vec<_>>();
    let mut assignments = BTreeMap::new();

    for partition in 0..partitions {
        let start = partition as usize % broker_ids.len();

        let replicas = (0..rf)
            .map(|i| broker_ids[(start + i) % broker_ids.len()])
            .collect::<Vec<_>>();

        assignments.insert(partition, replicas);
    }

    Ok(assignments)
}
```

In `CreateTopicsService` / backend `create_topic`:

* if request has explicit assignments, validate them;
* otherwise generate assignment from alive brokers;
* initialize partition state:

```rust
async fn wait_for_isr_ack<S>(
    storage: &S,
    tp: &Topition,
    required_offset_exclusive: i64,
    timeout_ms: i32,
) -> Result<()>
where
    S: Storage,
{
    let deadline = tokio::time::Instant::now()
        + Duration::from_millis(timeout_ms.max(0) as u64);

    loop {
        let state = storage.partition_replica_state(tp).await?;

        let min_isr = min_insync_replicas(storage, tp).await?;

        if state.isr.len() < min_isr {
            return Err(Error::Api(ErrorCode::NotEnoughReplicas));

            return Ok(());

        if tokio::time::Instant::now() >= deadline {
            return Err(Error::Api(ErrorCode::NotEnoughReplicasAfterAppend));
        }

        tokio::time::sleep(Duration::from_millis(10)).await;
    }
}
```

## Step 10: High watermark advancement

Create `tansu-storage/src/replication/hwm.rs`.

Kafka’s HW is effectively the highest offset replicated to all ISR members, so calculate it from ISR replica LEOs. Keep it monotonic.

```rust
use std::collections::{BTreeMap, BTreeSet};

pub fn recompute_high_watermark(
    old_hw: i64,
    isr: &BTreeSet<i32>,
    progress: &BTreeMap<i32, i64>,
    min_insync_replicas: usize,
) -> i64 {
    if isr.len() < min_insync_replicas {
        return old_hw;
    }

    let mut leos = isr
        .iter()
        .filter_map(|broker_id| progress.get(broker_id).copied())
        .collect::<Vec<_>>();

    if leos.len() != isr.len() {
        return old_hw;
    }

    leos.sort_unstable();

    let candidate = leos[0];

    candidate.max(old_hw)
}
```

Call this after:

* leader local append;
* follower fetch progress update;
* ISR expansion/shrink;
* broker heartbeat death detection.

Leader progress update should include leader’s own LEO:

```rust
state.progress.insert(
    leader_id,
    ReplicaProgress {
        broker_id: leader_id,
        log_end_offset: leader_leo,
        log_start_offset: leader_log_start,
        last_fetch_at: Some(now),
        last_caught_up_at: Some(now),
        leader_epoch: state.leader_epoch,
    },
);
```

### ISR shrink

A follower leaves ISR if:

```rust
now - follower.last_fetch_at > replica_lag_time_max_ms
```

or:

```rust
leader_leo - follower.log_end_offset > replica_lag_max_messages // optional
```

In Phase 11, start with time-based lag because it maps to Kafka’s `replica.lag.time.max.ms`.

### ISR expand

A follower can join ISR only when:

```rust
follower.log_end_offset >= leader_leo_at_check
&& follower.leader_epoch == current_leader_epoch
&& follower is alive
```

Safer policy:

```rust
follower.log_end_offset >= state.high_watermark
```

then require full catch-up to leader LEO before adding to ISR. For Phase 11 correctness, use full catch-up.

### Steering

Never advance HW based on non-ISR followers. Never add a follower to ISR before it has reconciled leader epochs and truncated divergent data.

## Step 11: Follower fetch loop

Create `tansu-broker/src/replication/fetcher.rs`.

```rust
pub struct ReplicaFetcher<S, C> {
    pub node_id: i32,
    pub storage: S,
    pub client: C,
    pub topition: Topition,
    pub config: ReplicationConfig,
}

impl<S, C> ReplicaFetcher<S, C>
where
    S: Storage + Clone,
    C: InterBrokerClient + Clone,
{
    pub async fn run(self, token: CancellationToken) -> Result<()> {
        let mut backoff = self.config.fetch_interval;

        loop {
            tokio::select! {
                _ = token.cancelled() => return Ok(()),
                _ = tokio::time::sleep(backoff) => {
                    match self.step().await {
                        Ok(()) => backoff = self.config.fetch_interval,
                        Err(err) => {
                            tracing::warn!(?err, tp = ?self.topition, "replica fetch step failed");
                            backoff = (backoff * 2).min(self.config.max_fetch_backoff);
                        }
                    }
                }
            }
        }
    }

    async fn step(&self) -> Result<()> {
        let state = self.storage.partition_replica_state(&self.topition).await?;

        if state.leader == Some(self.node_id) {
            return Ok(());
        }

        let Some(leader_id) = state.leader else {
            return Ok(());
        };

        if !state.replicas.contains(&self.node_id) {
            return Ok(());
        }

        self.ensure_epoch_reconciled(&state).await?;

        let fetch_offset = self.storage.local_log_end_offset(&self.topition).await?;

        let response = self.client
            .fetch_from_leader(
                leader_id,
                &self.topition,
                fetch_offset,
                state.leader_epoch,
                self.config.fetch_max_bytes,
            )
            .await?;

        for batch in response.batches {
            let expected = batch.base_offset;

            self.storage
                .append_replica_batch(
                    &self.topition,
                    state.leader_epoch,
                    expected,
                    batch,
                )
                .await?;
        }

        let new_leo = self.storage.local_log_end_offset(&self.topition).await?;

        self.storage
            .update_replica_progress(
                &self.topition,
                self.node_id,
                new_leo,
                response.log_start_offset,
                state.leader_epoch,
                SystemTime::now(),
            )
            .await?;

        Ok(())
    }
}
```

### Truncation before fetch

```rust
async fn ensure_epoch_reconciled(
    &self,
    state: &PartitionReplicaState,
) -> Result<()> {
    let local_epochs = self
        .storage
        .leader_epoch_checkpoints(&self.topition)
        .await?;

    let Some(last_local_epoch) = local_epochs.last().map(|e| e.leader_epoch) else {
        self.storage
            .record_leader_epoch(&self.topition, state.leader_epoch, 0)
            .await?;
        return Ok(());
    };

    if last_local_epoch == state.leader_epoch {
        return Ok(());
    }

    let leader_id = state.leader.ok_or(Error::Api(ErrorCode::LeaderNotAvailable))?;

    let truncation_offset = self.client
        .offset_for_leader_epoch(
            leader_id,
            &self.topition,
            state.leader_epoch,
            last_local_epoch,
        )
        .await?;

    self.storage
        .truncate_replica_log(&self.topition, truncation_offset)
        .await?;

    self.storage
        .record_leader_epoch(&self.topition, state.leader_epoch, truncation_offset)
        .await?;

    Ok(())
}
```

### Steering

The order matters:

1. observe new leader epoch;
2. call `OffsetForLeaderEpoch`;
3. truncate local log;
4. write new local epoch checkpoint;
5. only then fetch and append.

Do not append first and “fix later.”

## Step 12: Leader epoch checkpoints

Create `tansu-storage/src/replication/epoch.rs`.

Leader epoch checkpoint rules:

* When a broker becomes leader, append `(new_epoch, local_log_end_offset)`.
* When a follower truncates and starts following a new leader, append `(new_epoch, truncation_offset)`.
* When fetching, include current leader epoch.
* When serving `OffsetForLeaderEpoch`, return:

  * the start offset of the first epoch greater than requested epoch, or
  * current LEO if requested epoch is the current leader epoch.

```rust
pub fn end_offset_for_epoch(
    checkpoints: &[LeaderEpochCheckpoint],
    requested_epoch: i32,
    log_end_offset: i64,
) -> Option<i64> {
    for checkpoint in checkpoints {
        if checkpoint.leader_epoch > requested_epoch {
            return Some(checkpoint.start_offset);
        }
    }

    checkpoints
        .last()
        .filter(|last| requested_epoch <= last.leader_epoch)
        .map(|_| log_end_offset)
}
```

## Step 13: Minimal leader change handling before Phase 12

Milestone 12 is the full cluster controller, but Phase 11 cannot test truncation on leader change without some leader movement mechanism. Implement a **minimal storage-backed election** in Phase 11 and later replace/refine it in Phase 12.

Add `tansu-broker/src/replication/election.rs`.

Rules:

1. If current leader is alive, do nothing.
2. If current leader is dead:

   * prefer an alive broker in ISR;
   * if ISR is empty and `unclean.leader.election.enable=false`, leave leader unavailable;
   * if unclean election is enabled, choose the alive assigned replica with greatest known LEO.
3. Increment `leader_epoch`.
4. New leader writes epoch checkpoint at local LEO.
5. State update uses optimistic version/CAS to avoid double election.
6. Followers observe epoch change and truncate/fetch.

Pseudo-code:

```rust
pub async fn maybe_elect_leader<S>(
    storage: &S,
    tp: &Topition,
    now: SystemTime,
    cfg: &ReplicationConfig,
) -> Result<()>
where
    S: Storage,
{
    let state = storage.partition_replica_state(tp).await?;

    if let Some(leader) = state.leader {
        if broker_is_alive(storage, leader, now, cfg.broker_session_timeout).await? {
            return Ok(());
        }
    }

    let alive = storage
        .alive_brokers(now, cfg.broker_session_timeout)
        .await?
        .into_iter()
        .map(|b| b.broker_id)
        .collect::<BTreeSet<_>>();

    let candidate = state
        .isr
        .iter()
        .copied()
        .find(|id| alive.contains(id))
        .or_else(|| {
            if cfg.unclean_leader_election {
                state.replicas.iter().copied().find(|id| alive.contains(id))
            } else {
                None
            }
        });

    let Some(new_leader) = candidate else {
        tracing::warn!(?tp, "no eligible leader");
        return Ok(());
    };

    let new_epoch = state.leader_epoch + 1;
    let leo = storage.local_log_end_offset(tp).await.unwrap_or(state.high_watermark);

    storage
        .become_leader(tp, new_leader, new_epoch, leo)
        .await?;

    Ok(())
}
```

### Steering

Phase 11 minimal election is not the final controller. But it must still be safe enough to test epoch changes, stale leader fencing, HW behavior, and truncation.

## Step 15: Inter-broker client strategy

You have two implementation choices.

## Option A: Use Kafka Fetch / OffsetForLeaderEpoch internally

This is more compatible with Kafka’s model and helps test protocol correctness.

Pros:

* exercises real Kafka protocol codec;
* closer to Kafka;
* no custom RPC to maintain.

Cons:

* requires all relevant protocol structs/services in `tansu-sans-io`;
* needs connection pooling/auth/TLS handling.

## Option B: Add Tansu internal replication RPC

# Produce semantics in detail

## Before append

For each partition:

1. Load `PartitionReplicaState`.
2. Validate this broker is leader.
3. Validate `leader_epoch`.
4. Validate ISR size against `min.insync.replicas` for `acks=all`.
5. Append locally as leader.
6. Update leader progress.
7. Recompute HW.
8. For `acks=1`, respond.
9. For `acks=all`, wait for HW to reach `last_offset + 1`.

## Error mapping

| Condition                                               | Kafka error                        |
| ------------------------------------------------------- | ---------------------------------- |
| broker is not current leader                            | `NOT_LEADER_OR_FOLLOWER`           |
| no leader                                               | `LEADER_NOT_AVAILABLE`             |
| ISR size below min ISR before append                    | `NOT_ENOUGH_REPLICAS`              |
| append happened but ISR did not catch up before timeout | `NOT_ENOUGH_REPLICAS_AFTER_APPEND` |
| stale request leader epoch                              | `FENCED_LEADER_EPOCH`              |
| future request leader epoch                             | `UNKNOWN_LEADER_EPOCH`             |
| partition absent                                        | `UNKNOWN_TOPIC_OR_PARTITION`       |

# Fetch semantics in detail

## Client fetch

* `replica_id < 0`
* validate leader if request includes current leader epoch;
* fetch only `[fetch_offset, high_watermark)`;
* return HW, LSO, log-start;
* return `OFFSET_OUT_OF_RANGE` when needed;
* return `NOT_LEADER_OR_FOLLOWER` if request reached a non-leader.

## Follower fetch

* `replica_id >= 0`
* validate the requester is in assigned replicas;
* validate leader epoch;
* fetch `[fetch_offset, leader_log_end_offset)`, not capped by HW;
* update follower progress only after follower reports/appends. Do not trust request offset alone as durable progress unless protocol semantics guarantee it.

# ISR state machine

Create `tansu-broker/src/replication/leader.rs`.

```rust
pub async fn leader_maintenance<S>(
    storage: &S,
    tp: &Topition,
    cfg: &ReplicationConfig,
    now: SystemTime,
) -> Result<()>
where
    S: Storage,
{
    let mut state = storage.partition_replica_state(tp).await?;

    let Some(leader_id) = state.leader else {
        return Ok(());
    };

    let leader_leo = state
        .progress
        .get(&leader_id)
        .map(|p| p.log_end_offset)
        .unwrap_or(state.high_watermark);

    let mut changed = false;

    for broker_id in state.isr.clone() {
        if broker_id == leader_id {
            continue;
        }

        let lagged = state
            .progress
            .get(&broker_id)
            .and_then(|p| p.last_fetch_at)
            .map(|last| now.duration_since(last).unwrap_or_default() > cfg.replica_lag_time_max)
            .unwrap_or(true);

        if lagged {
            state.isr.remove(&broker_id);
            changed = true;
        }
    }

    for replica in state.replicas.clone() {
        if state.isr.contains(&replica) {
            continue;
        }

        let caught_up = state
            .progress
            .get(&replica)
            .map(|p| {
                p.leader_epoch == state.leader_epoch
                    && p.log_end_offset >= leader_leo
            })
            .unwrap_or(false);

        if caught_up {
            state.isr.insert(replica);
            changed = true;
        }
    }

    let progress = state
        .progress
        .iter()
        .map(|(broker, progress)| (*broker, progress.log_end_offset))
        .collect();

    let new_hw = recompute_high_watermark(
        state.high_watermark,
        &state.isr,
        &progress,
        cfg.min_insync_replicas,
    );

    if new_hw != state.high_watermark {
        state.high_watermark = new_hw;
        state.last_stable_offset = state.last_stable_offset.max(new_hw);
        changed = true;
    }

    if changed {
        storage
            .update_partition_replica_state(state.clone(), state.version.clone())
            .await
            .map_err(|err| match err {
                UpdateError::Outdated { .. } => Error::Api(ErrorCode::UnknownServerError),
                UpdateError::Error(err) => err,
            })?;
    }

    Ok(())
}
```

### Steering

`last_stable_offset` can equal HW until the transaction milestone is complete. Once Phase 14 is done, LSO must account for open transactions.

# Truncation scenarios to explicitly test

## Scenario A: follower behind, no divergence

1. Leader 1 has offsets `0..100`.
2. Follower 2 has offsets `0..60`.
3. Follower fetches from offset 60.
4. Follower appends `60..100`.
5. ISR expands to include 2.
6. HW advances.

Expected: no truncation.

## Scenario B: old leader has uncommitted tail

1. RF=3, ISR `{1,2}`.
2. Leader 1 appends offsets `10..15`, not replicated to 2.
3. Leader 1 dies.
4. Broker 2 becomes leader at epoch 2 with LEO 10.
5. Broker 1 restarts as follower.
6. Broker 1 calls OffsetForLeaderEpoch.
7. Broker 1 truncates to 10.
8. Broker 1 fetches epoch 2 data.

Expected: offsets `10..15` from old epoch are gone on broker 1.

## Scenario C: stale fetcher

1. Broker 2 follows leader 1 at epoch 1.
2. Leader changes to broker 3 at epoch 2.
3. Broker 2 sends fetch to broker 3 with epoch 1.

Expected: `FENCED_LEADER_EPOCH`.

## Scenario D: future epoch request

1. Broker 2 sends fetch with epoch 999.

Expected: `UNKNOWN_LEADER_EPOCH`.

# Compatibility details to be careful with

## `acks=all` waits for ISR, not all assigned replicas

If RF=3 and ISR is `{1,2}`, `acks=all` waits for brokers 1 and 2, not broker 3. But if `min.insync.replicas=3`, ISR `{1,2}` is insufficient and Produce must fail.

## HW must be monotonic

Even after ISR shrink, never reduce HW.

## Followers may fetch beyond HW

Consumers cannot. Followers must, or they will never replicate the leader’s uncommitted tail.

## Do not count dead brokers as ISR

A broker with a recent LEO but stale heartbeat should be removed from ISR.

## Do not re-add a follower to ISR after stale epoch

Follower must first truncate and catch up in the current leader epoch.

## Metadata update order matters

On leader change:

1. write new leader epoch;
2. update partition leader;
3. new leader writes epoch checkpoint;
4. followers truncate/fetch;
5. ISR grows after catch-up.

Do not advertise a stale ISR.
