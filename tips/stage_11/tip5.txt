Absolutely — Phase 11 is where Tansu crosses the big architectural line from “Kafka-compatible broker backed by durable storage” into “Kafka-compatible replicated log.” That is an exciting, high-leverage phase because it changes the meaning of metadata, produce acks, fetch visibility, broker identity, and recovery.

The current repo is especially important here: Tansu’s README describes it as a **stateless Kafka-compatible broker with pluggable storage**, and the storage README currently notes that example topic descriptions show **node 111 as leader and ISR for every partition**, with “all brokers” effectively represented as node 111. That is the exact behavior Phase 11 must replace with real per-partition leader, replica, ISR, epoch, and high-watermark state. ([GitHub][1])

Below is a very detailed implementation plan to finish **all of Phase 11**.

---

# Phase 11 target

Phase 11 should finish with this being true:

A Tansu cluster with 3 brokers can create a topic with `replication.factor=3`, elect or assign one leader per partition, replicate records from leader to followers, maintain ISR, advance high watermark only when ISR replicas have replicated, expose correct Kafka metadata, recover after broker restart, handle leader epoch changes, and truncate divergent follower logs after leader change.

Kafka compatibility expectations:

```bash
kafka-topics \
  --bootstrap-server localhost:9092 \
  --create \
  --topic phase11 \
  --partitions 3 \
  --replication-factor 3

kafka-topics \
  --bootstrap-server localhost:9092 \
  --describe \
  --topic phase11
```

Should eventually show something like:

```text
Topic: phase11 Partition: 0 Leader: 1 Replicas: 1,2,3 Isr: 1,2,3
Topic: phase11 Partition: 1 Leader: 2 Replicas: 2,3,1 Isr: 2,3,1
Topic: phase11 Partition: 2 Leader: 3 Replicas: 3,1,2 Isr: 3,1,2
```

Not the current “everything is node 111” behavior.

Kafka’s Fetch protocol already distinguishes follower fetches from normal consumer fetches via `replica_id`; normal consumers use `-1`, followers use their broker id. Kafka also uses `current_leader_epoch` to fence stale clients and replicas. Those details matter a lot for Phase 11 because the same Fetch API must serve both consumers and follower replicas. ([Apache Kafka][2])

# Core architectural decision

## Do not implement “replication” as cosmetic metadata

Tansu currently has strong storage durability options: PostgreSQL, SQLite/libSQL, S3, memory, and lake/schema integrations. But Kafka replication is not the same as “the storage backend is durable.” Kafka replication means:

```text
broker 1 has a local replica log
broker 2 has a local replica log
broker 3 has a local replica log

one broker is leader
other brokers fetch from the leader
ISR reflects which broker logs are caught up
HW reflects what is replicated enough to expose as committed
leader epoch records leadership history
followers truncate divergent tails after leader changes
```

So the big steering point is:

**Phase 11 must introduce per-broker replica logs.**

For object storage backends, that can still be implemented with shared object storage during development, but paths must be per broker. For example, the current `DynoStore` writes batches under:

```text
clusters/{cluster}/topics/{topic}/partitions/{partition}/records/{offset}.batch
```

Phase 11 should move replicated logs to a per-broker namespace:

```text
clusters/{cluster}/brokers/{broker_id}/topics/{topic}/partitions/{partition}/records/{offset}.batch
```

Cluster metadata remains shared, but log replicas are broker-specific.

That makes replication real: broker 2 does not “already have” broker 1’s records just because both use S3/PostgreSQL. Broker 2 must fetch, append, update its LEO, and rejoin ISR.

---

# Current repo map relevant to Phase 11

The repo has the right layering for this. Storage services and the protocol-facing service router are already separated. The `tansu-storage/src` tree contains service modules and storage backend implementations, including `dynostore.rs`, `pg.rs`, `lite.rs`, `limbo.rs`, `sql.rs`, and the `service` directory. ([GitHub][3])

The broker crate has `broker.rs`, `service.rs`, and subdirectories for broker/coordinator/service logic. ([GitHub][4])

Current broker startup registers the broker through `storage.register_broker(...)`, then listens for connections. The `BrokerRegistrationRequest` currently carries broker id, cluster id, incarnation id, and rack, but not enough endpoint/liveness state for a replicated cluster. 

Current protocol routing is clean: `tansu-broker/src/service/storage.rs` wires Kafka API keys like Produce, Fetch, Metadata, CreateTopics, and DescribeTopicPartitions to storage services. 

Current Produce service calls `ctx.state().produce(...)` directly and returns `current_leader(None)`, which means leader enforcement and leader hints are not yet implemented. 

Current Fetch service calls `ctx.state().fetch(...)`, then `offset_stage(...)`, and always returns `diverging_epoch(None)` and `current_leader(None)` for normal fetch responses. Phase 11 must change this to support follower fetches, epoch fencing, high-watermark visibility, and divergence detection. 

Current `DynoStore` metadata has producers, topics, and transactions, but no broker registry, partition leader, ISR, follower progress, or leader epoch metadata. Its `register_broker` currently returns `Ok(())`, `brokers()` returns only the local broker, and `create_topic` initializes topic/watermark state without true replica assignment.

```rust
    pub replicas: Vec<i32>,

    pub leader: i32,

    /// Monotonically increasing epoch for this partition leadership.
    pub leader_epoch: i32,

    /// Offset where the current leader epoch began.
    pub leader_epoch_start_offset: i64,

    /// In-sync replica broker ids.
    pub isr: BTreeSet<i32>,

    /// Replicas assigned but not currently in ISR.
    pub offline_replicas: BTreeSet<i32>,

    pub high_watermark: i64,

    /// Earliest retained offset.
    pub log_start_offset: i64,

    pub last_stable_offset: i64,

    pub log_end_offset: i64,

    /// Last offset known to be flushed/durable locally.
    pub durable_log_end_offset: i64,

    pub last_fetch_at_ms: i64,

    /// Last wall-clock millis at which this replica was fully caught up.
    pub last_caught_up_at_ms: i64,

    pub leader_epoch: i32,

pub struct LeaderEpochCheckpoint {

    pub created_at_ms: i64,
```

## High-watermark calculation

Create `tansu-replication/src/high_watermark.rs`:

```rust
use crate::model::{HighWatermarkUpdate, PartitionReplicationState};

pub fn compute_high_watermark(
    state: &PartitionReplicationState,
    leader_log_end_offset: i64,
) -> HighWatermarkUpdate {
    let before = state.high_watermark;

    if state.isr.is_empty() {
        return HighWatermarkUpdate {
            before,
            after: before,
            advanced: false,
        };
    }

    let min_isr_leo = state
        .isr
        .iter()
        .map(|broker_id| {
            if *broker_id == state.leader {
                leader_log_end_offset
            } else {
                state
                    .replica_progress
                    .get(broker_id)
                    .map(|p| p.log_end_offset)
                    .unwrap_or(state.log_start_offset)
            }
        })
        .min()
        .unwrap_or(before);

    // HW must never go backwards.
    let after = before.max(min_isr_leo);

    HighWatermarkUpdate {
        before,
        after,
        advanced: after > before,
    }
}
```

## ISR update logic

Create `tansu-replication/src/isr.rs`:

```rust
use crate::model::PartitionReplicationState;
use std::collections::BTreeSet;
use std::time::Duration;

#[derive(Clone, Debug, Eq, PartialEq)]
pub struct IsrConfig {
    pub replica_lag_time_max_ms: i64,
    pub catch_up_delta: i64,
}

impl Default for IsrConfig {
    fn default() -> Self {
        Self {
            replica_lag_time_max_ms: Duration::from_secs(30).as_millis() as i64,
            catch_up_delta: 0,
        }
    }
}

#[derive(Clone, Debug, Eq, PartialEq)]
pub struct IsrChange {
    pub before: BTreeSet<i32>,
    pub after: BTreeSet<i32>,
    pub added: BTreeSet<i32>,
    pub removed: BTreeSet<i32>,
}

pub fn refresh_isr(
    state: &PartitionReplicationState,
    leader_log_end_offset: i64,
    now_ms: i64,
    config: &IsrConfig,
) -> IsrChange {
    let before = state.isr.clone();
    let mut after = before.clone();

    for replica_id in &state.replicas {
        if *replica_id == state.leader {
            after.insert(*replica_id);
            continue;
        }

        let Some(progress) = state.replica_progress.get(replica_id) else {
            after.remove(replica_id);
            continue;
        };

        let is_timed_out =
            now_ms.saturating_sub(progress.last_fetch_at_ms) > config.replica_lag_time_max_ms;

        let is_caught_up =
            leader_log_end_offset.saturating_sub(progress.log_end_offset) <= config.catch_up_delta;

        if is_timed_out {
            after.remove(replica_id);
        } else if is_caught_up {
            after.insert(*replica_id);
        }
    }

    let added = after.difference(&before).copied().collect();
    let removed = before.difference(&after).copied().collect();

    IsrChange {
        before,
        after,
        added,
        removed,
    }
}
```

# Workstream 3: topic creation and replica assignment

## Goal

When creating topics, assign replicas to brokers and initialize leader/ISR/epoch state.

Current `CreateTopicsService` defaults `replication_factor=-1` to `1`, then delegates to `storage.create_topic(...)`. 

Phase 11 should change the storage implementation behind `create_topic` so that:

```text
replication_factor=1 -> one replica, leader is that broker, ISR contains leader
replication_factor=N -> N replicas assigned across live brokers
manual assignments -> honored exactly
not enough brokers -> InvalidReplicationFactor
```

## Assignment logic

In `tansu-replication/src/assignment.rs`:

```rust
use crate::error::{Error, Result};

#[derive(Clone, Debug, Eq, PartialEq)]
pub struct PartitionAssignment {
    pub partition: i32,
    pub replicas: Vec<i32>,
    pub leader: i32,
}

pub fn assign_replicas_round_robin(
    broker_ids: &[i32],
    partition_count: i32,
    replication_factor: i16,
) -> Result<Vec<PartitionAssignment>> {
    if replication_factor <= 0 {
        return Err(Error::InvalidReplicationFactor(replication_factor));
    }

    let rf = replication_factor as usize;

    if broker_ids.len() < rf {
        return Err(Error::NotEnoughBrokers {
            available: broker_ids.len(),
            required: rf,
        });
    }

    let mut assignments = Vec::with_capacity(partition_count as usize);

    for partition in 0..partition_count {
        let start = partition as usize % broker_ids.len();

        let replicas = (0..rf)
            .map(|i| broker_ids[(start + i) % broker_ids.len()])
            .collect::<Vec<_>>();

        let leader = replicas[0];

        assignments.push(PartitionAssignment {
            partition,
            replicas,
            leader,
        });
    }

    Ok(assignments)
}
```

## Storage create topic flow

Inside each backend’s `create_topic`:

```text
1. Validate topic does not already exist.
2. Resolve live brokers.
3. Resolve explicit assignment or round-robin assignment.
4. Create TopicMetadata.
5. For each partition:
   - create PartitionReplicationState
   - leader = first replica
   - leader_epoch = 0
   - leader_epoch_start_offset = 0
   - isr = {leader} initially, or all replicas if they are created empty and local LEO=0
   - high_watermark = 0
   - log_start_offset = 0
   - last_stable_offset = 0
   - progress for each replica LEO=0
6. Initialize empty local replica log namespace for each assigned replica.
```

Steering: For empty partitions, it is acceptable to put all assigned replicas in ISR immediately because all replicas are equally caught up at offset 0. If you want stricter semantics, initialize only leader in ISR and let followers enter ISR after their first successful fetch. The latter gives better test coverage of ISR expansion.

## Important offset rule

Leader append allocates offsets.

Follower append **must not** allocate offsets. It must preserve the leader’s `base_offset` and `last_offset_delta`.

If a follower receives a batch with `base_offset != local_log_end_offset`, it must reconcile:

```text
batch.base_offset < local LEO -> duplicate or divergence
batch.base_offset == local LEO -> append
batch.base_offset > local LEO -> gap; fetch missing data first
```

```rust
async fn wait_for_high_watermark<G: Storage>(
```

# Workstream 6: Fetch path split into client fetch and follower fetch

## Goal

The same Kafka Fetch API must serve:

```text
consumer fetch: replica_id = -1
follower fetch: replica_id = broker id
```

Kafka documents `replica_id` as the broker id of the follower, with normal consumers using `-1`. ([Apache Kafka][2])

Current `FetchService` does not distinguish these paths. 

## Add fetch kind detection

In `tansu-storage/src/service/fetch.rs`:

```rust
#[derive(Clone, Copy, Debug, Eq, PartialEq)]
enum FetchKind {
    Client,
    Follower { replica_id: i32 },
}

impl FetchKind {
    fn from_request(req: &FetchRequest) -> Self {
        // Field name depends on generated struct.
        // Older versions have req.replica_id.
        // Newer flexible versions may carry replica_state.
        match req.replica_id {
            id if id >= 0 => Self::Follower { replica_id: id },
            _ => Self::Client,
        }
    }
}
```

```rust
        replica_state.replica_id
    } else {
        req.replica_id
    }
}
```

## Client fetch rules

For normal consumers:

```text
If local broker is leader:
  serve from local leader log up to HW or LSO
If local broker is follower:
  return NotLeaderOrFollower unless preferred replica reads are deliberately enabled
```

In Phase 11, keep it simple: **no follower reads for consumers yet**.

## Follower fetch rules

For follower replicas:

```text

Leader validates replica is assigned.
Leader validates current_leader_epoch.
Leader returns records up to leader LEO, not just HW.
```


## Epoch fencing

When a fetch partition has `current_leader_epoch`, enforce it.

```rust
fn validate_leader_epoch(
    current: i32,
    provided: Option<i32>,
) -> Result<(), ErrorCode> {
    match provided {
        None | Some(-1) => Ok(()),
        Some(epoch) if epoch == current => Ok(()),
        Some(epoch) if epoch < current => Err(ErrorCode::FencedLeaderEpoch),
        Some(_) => Err(ErrorCode::UnknownLeaderEpoch),
    }
}
```

Kafka’s protocol docs state that `current_leader_epoch` fences old metadata: if the provided epoch is larger than the broker’s current epoch, return `UNKNOWN_LEADER_EPOCH`; if smaller, return `FENCED_LEADER_EPOCH`. ([Apache Kafka][2])

```rust
use std::{collections::BTreeSet, time::Duration};
use tokio::{task::JoinSet, time};
use tokio_util::sync::CancellationToken;
use tracing::{debug, error, instrument};

use tansu_storage::{Result, Storage, Topition};

#[derive(Clone, Debug)]
pub struct ReplicaManager<S> {
    broker_id: i32,
    cluster_id: String,
    storage: S,
    cancellation: CancellationToken,
    scan_interval: Duration,
}

impl<S> ReplicaManager<S>
where
    S: Storage + Clone,
{
    pub fn new(
        broker_id: i32,
        cluster_id: String,
        storage: S,
        cancellation: CancellationToken,
    ) -> Self {
        Self {
            broker_id,
            cluster_id,
            storage,
            cancellation,
            scan_interval: Duration::from_millis(500),
        }
    }

    #[instrument(skip_all)]
    pub async fn run(self) -> Result<()> {
        let mut tasks = JoinSet::new();
        let mut assigned = BTreeSet::<Topition>::new();

        loop {
            tokio::select! {
                _ = self.cancellation.cancelled() => {
                    tasks.abort_all();
                    break;
                }

                _ = time::sleep(self.scan_interval) => {
                    let wanted = self.storage
                        .follower_assignments(self.broker_id)
                        .await?;

                    for tp in wanted {
                        if assigned.insert(tp.clone()) {
                            let task = PartitionReplicaTask::new(
                                self.broker_id,
                                tp,
                                self.storage.clone(),
                                self.cancellation.clone(),
                            );

                            tasks.spawn(async move {
                                task.run().await
                            });
                        }
                    }
                }

                maybe = tasks.join_next(), if !tasks.is_empty() => {
                    if let Some(result) = maybe {
                        match result {
                            Ok(Ok(())) => {}
                            Ok(Err(error)) => error!(?error),
                            Err(error) => error!(?error),
                        }
                    }
                }
            }
        }

        Ok(())
    }
}
```

## Partition replica task

In `tansu-broker/src/replication/task.rs`:

```rust
use std::time::Duration;
use tokio::time;
use tokio_util::sync::CancellationToken;
use tracing::{debug, error, instrument};

use tansu_storage::{
    FollowerAppendRequest, ReplicaFetchRequest, Result, Storage, Topition,
};

pub struct PartitionReplicaTask<S> {
    broker_id: i32,
    topition: Topition,
    storage: S,
    cancellation: CancellationToken,
}

impl<S> PartitionReplicaTask<S>
where
    S: Storage + Clone,
{
    pub fn new(
        broker_id: i32,
        topition: Topition,
        storage: S,
        cancellation: CancellationToken,
    ) -> Self {
        Self {
            broker_id,
            topition,
            storage,
            cancellation,
        }
    }

    #[instrument(skip_all, fields(topic = %self.topition.topic(), partition = self.topition.partition()))]
    pub async fn run(self) -> Result<()> {
        let mut backoff = Duration::from_millis(50);

        loop {
            tokio::select! {
                _ = self.cancellation.cancelled() => break,
                result = self.replicate_once() => {
                    match result {
                        Ok(()) => {
                            backoff = Duration::from_millis(50);
                        }
                        Err(error) => {
                            error!(?error);
                            time::sleep(backoff).await;
                            backoff = (backoff * 2).min(Duration::from_secs(5));
                        }
                    }
                }
            }
        }

        Ok(())
    }

    async fn replicate_once(&self) -> Result<()> {
        let state = self.storage.replication_state(&self.topition).await?;

        if state.leader == self.broker_id {
            time::sleep(Duration::from_millis(250)).await;
            return Ok(());
        }

        if !state.replicas.contains(&self.broker_id) {
            time::sleep(Duration::from_secs(1)).await;
            return Ok(());
        }

        let local_leo = self.storage.local_log_end_offset(&self.topition).await?;

        let response = self.storage
            .fetch_from_leader(ReplicaFetchRequest {
                replica_id: self.broker_id,
                topition: self.topition.clone(),
                fetch_offset: local_leo,
                current_leader_epoch: Some(state.leader_epoch),
                log_start_offset: Some(state.log_start_offset),
                max_bytes: 1024 * 1024,
            })
            .await?;

        if let Some(diverging) = response.diverging_epoch {
            self.storage
                .truncate_to(&self.topition, diverging.end_offset, diverging.epoch)
                .await?;
            return Ok(());
        }

        for batch in response.batches {
            self.storage
                .append_as_follower(FollowerAppendRequest {
                    topition: self.topition.clone(),
                    leader_id: response.leader_id,
                    leader_epoch: response.leader_epoch,
                    batch,
                })
                .await?;
        }

        let leo = self.storage.local_log_end_offset(&self.topition).await?;

        self.storage
            .report_follower_position(
                &self.topition,
                self.broker_id,
                leo,
                response.leader_epoch,
            )
            .await?;

        Ok(())
    }
}
```

## Transport choice

There are two possible designs:

### Option A: internal storage-mediated fetch

`fetch_from_leader(...)` looks up the leader’s log directly via shared metadata/storage.

This is easier for Phase 11 tests, especially with object storage.

### Option B: real Kafka protocol fetch over TCP

The follower opens a connection to the leader’s advertised listener and sends Kafka Fetch requests.

This is better for Kafka compatibility because it exercises the same protocol path external clients use.

Recommended plan:

```text
PR 1: implement Option A to validate state machine and storage behavior.
PR 2: replace transport with Option B or make both available behind config.
```

Do not wait until Phase 12 for this. A follower fetch path that never goes through the Kafka Fetch service will leave protocol bugs hidden.

# Workstream 8: ISR tracking and high-watermark advancement

## Goal

Leader tracks follower progress and advances high watermark.

High watermark must be the minimum LEO of replicas in ISR, never decreasing.

## New storage method

In `tansu-storage/src/lib.rs`:

```rust
async fn report_follower_position(
    &self,
    topition: &Topition,
    replica_id: i32,
    log_end_offset: i64,
    leader_epoch: i32,
) -> Result<HighWatermarkUpdate>;
```

## Algorithm

When follower fetches from leader:

```text
1. Validate follower is assigned replica.
2. Update replica_progress[follower].last_fetch_at.
3. If follower reports LEO, store it.
4. If follower is caught up, add to ISR.
5. Remove timed-out ISR members during maintenance.
6. Recompute HW = min(LEO of ISR members).
7. Persist state atomically.
```

## DynoStore implementation sketch

Inside `DynoStore`:

```rust
async fn report_follower_position(
    &self,
    topition: &Topition,
    replica_id: i32,
    log_end_offset: i64,
    leader_epoch: i32,
) -> Result<HighWatermarkUpdate> {
    let now = SystemTime::now();

    self.meta
        .with_mut(&self.object_store, |meta| {
            let partition = meta
                .partitions
                .get_mut(topition.topic())
                .and_then(|p| p.get_mut(&topition.partition()))
                .ok_or(Error::Api(ErrorCode::UnknownTopicOrPartition))?;

            if partition.leader_epoch != leader_epoch {
                return Err(Error::Api(ErrorCode::FencedLeaderEpoch));
            }

            if !partition.replicas.contains(&replica_id) {
                return Err(Error::Api(ErrorCode::ReplicaNotAvailable));
            }

            let progress = partition
                .replica_progress
                .entry(replica_id)
                .or_insert_with(|| ReplicaProgressMetadata {
                    log_end_offset: 0,
                    durable_log_end_offset: 0,
                    last_fetch_at: now,
                    last_caught_up_at: now,
                    leader_epoch,
                });

            progress.log_end_offset = log_end_offset;
            progress.durable_log_end_offset = log_end_offset;
            progress.last_fetch_at = now;
            progress.leader_epoch = leader_epoch;

            let leader_leo = partition
                .replica_progress
                .get(&partition.leader)
                .map(|p| p.log_end_offset)
                .unwrap_or(partition.high_watermark);

            if log_end_offset >= leader_leo {
                progress.last_caught_up_at = now;
                partition.isr.insert(replica_id);
            }

            let update = compute_hw_from_partition(partition, leader_leo);

            partition.high_watermark = update.after;
            partition.last_stable_offset = partition.last_stable_offset.max(update.after);

            Ok(update)
        })
        .await
}
```

## Maintenance ISR shrink

The broker already periodically calls `storage.maintain(SystemTime::now())`. 

Use that hook for ISR shrink:

```rust
async fn maintain(&self, now: SystemTime) -> Result<()> {
    self.shrink_stale_isr(now).await?;
    self.expire_old_sessions(now).await?;
    self.cleanup_retention(now).await?;
    Ok(())
}
```

ISR shrink logic:

```text
For each partition where self.node is leader:
  for each ISR follower:
    if now - last_fetch_at > replica.lag.time.max.ms:
      remove from ISR
  recompute HW
  persist
```

## Produce `acks=all` and min ISR

Add topic config handling:

```text
min.insync.replicas
replica.lag.time.max.ms
```

Effective rule:

```text
acks=all produce is allowed only if ISR size >= min.insync.replicas.
If ISR shrinks below min ISR before append: NotEnoughReplicas.
If append succeeds locally but HW does not reach required offset by timeout: NotEnoughReplicasAfterAppend or RequestTimedOut.
```

# Workstream 9: leader epochs

## Goal

Persist every leader change:

```text
partition p0:
  epoch 0 leader 1 starts at offset 0
  epoch 1 leader 2 starts at offset 1034
  epoch 2 leader 1 starts at offset 2098
```

This enables:

```text
stale metadata fencing
follower divergence detection
truncation after leader change
OffsetForLeaderEpoch support
```

Kafka has an `OffsetForLeaderEpoch` API for looking up offsets by epoch. ([Apache Kafka][2])

## Add leader transition primitive

Although full controller belongs to Phase 12, Phase 11 needs a testable primitive:

```rust
async fn become_leader(
    &self,
    topition: &Topition,
    new_leader: i32,
    expected_current_leader_epoch: i32,
) -> Result<PartitionReplicationState>;
```

Rules:

```text
1. Only assigned replica can become leader.
2. New epoch = old epoch + 1.
3. New epoch start offset = new leader local LEO.
4. ISR must contain new leader.
5. High watermark must not decrease.
6. Persist leader epoch checkpoint.
7. Fence old leader.
```

## Code sketch

```rust
async fn become_leader(
    &self,
    topition: &Topition,
    new_leader: i32,
    expected_epoch: i32,
) -> Result<PartitionReplicationState> {
    let new_leader_leo = self
        .local_log_end_offset_for_broker(new_leader, topition)
        .await?;

    self.meta
        .with_mut(&self.object_store, |meta| {
            let partition = meta.partition_mut(topition)?;

            if partition.leader_epoch != expected_epoch {
                return Err(Error::Api(ErrorCode::FencedLeaderEpoch));
            }

            if !partition.replicas.contains(&new_leader) {
                return Err(Error::Api(ErrorCode::ReplicaNotAvailable));
            }

            if !partition.isr.contains(&new_leader) {
                return Err(Error::Api(ErrorCode::NotEnoughReplicas));
            }

            partition.leader = new_leader;
            partition.leader_epoch += 1;
            partition.leader_epoch_start_offset = new_leader_leo;

            partition.leader_epochs.push(LeaderEpochMetadata {
                leader_epoch: partition.leader_epoch,
                leader_id: new_leader,
                start_offset: new_leader_leo,
                created_at: SystemTime::now(),
            });

            Ok(partition.clone().into())
        })
        .await
}
```

# Workstream 10: truncation on leader change

## Goal

When a follower has records that are not in the new leader’s log, it must truncate them before continuing replication.

## Required cases

```text
Case A: follower is behind
  follower LEO < leader LEO
  action: fetch and append

Case B: follower is equal
  follower LEO == leader LEO
  action: no-op

Case C: follower has divergent tail
  follower LEO > leader epoch end offset
  action: truncate to leader-provided offset, then fetch
```

## Add truncation API

```rust
async fn truncate_to(
    &self,
    topition: &Topition,
    offset: i64,
    leader_epoch: i32,
) -> Result<()>;
```

## DynoStore object deletion

For per-broker object-store paths:

```text
clusters/{cluster}/brokers/{broker_id}/topics/{topic}/partitions/{partition}/records/{offset}.batch
```

Truncation can list `records/` and delete all objects whose base offset is `>= offset`.

Partial-batch truncation is harder. Kafka offsets can cut inside a batch. You need to choose one of two strategies.

### Strategy 1: batch-boundary truncation first

For Phase 11 initial implementation:

```text
Only truncate at batch base offsets.
If requested truncation cuts inside a batch, truncate to that batch’s base offset.
```

This is simpler but less fully Kafka-compatible.

### Strategy 2: exact partial-batch truncation

For feature completeness, implement exact trimming:

```text
1. Find batch with base_offset <= truncation_offset <= last_offset.
2. Decode batch.
3. Keep records with absolute_offset < truncation_offset.
4. Rebuild batch:
   - same producer id/epoch
   - adjusted last_offset_delta
   - adjusted record_count
   - recompute CRC
5. Replace object atomically.
6. Delete later batches.
7. Set local LEO = truncation_offset.
```

Recommended path:

```text
Implement Strategy 1 in first PR.
Add Strategy 2 before Phase 11 is marked complete.
```

# Workstream 12: backend implementation details

## DynoStore

This is the most important backend for Phase 11 because it already has rich object-store behavior.

### Paths

Use global cluster metadata:

```text
clusters/{cluster}/meta.json
clusters/{cluster}/brokers/{broker_id}.json
clusters/{cluster}/topics/{topic}/partitions/{partition}/replication.json
clusters/{cluster}/topics/{topic}/partitions/{partition}/leader-epochs.json
```

Use per-broker logs:

```text
clusters/{cluster}/brokers/{broker_id}/topics/{topic}/partitions/{partition}/watermark.json
clusters/{cluster}/brokers/{broker_id}/topics/{topic}/partitions/{partition}/records/{offset}.batch
```

### Why separate global and local watermarks?

You need both:

```text
global high watermark: committed offset for partition
local broker LEO: next offset in this broker’s replica log
```

Current `Watermark.high` is used like a local log-end allocator. In Phase 11 that cannot also be the replicated high watermark. Rename or split:

```rust
struct LocalReplicaWatermark {
    log_start_offset: i64,
    log_end_offset: i64,
    durable_log_end_offset: i64,
    timestamps: Option<BTreeMap<i64, i64>>,
}

struct PartitionMetadata {
    high_watermark: i64,
    last_stable_offset: i64,
    // ...
}
```

### Leader append path

```text
1. Load partition metadata.
2. Verify self.node == leader.
3. Load local leader watermark.
4. Assign base offset = local LEO.
5. Rewrite/validate batch base offset if needed.
6. Write batch object in broker namespace.
7. Advance local LEO.
8. Update leader replica progress.
9. Recompute HW.
10. Return LeaderAppendResult.
```

### Follower append path

```text
1. Load partition metadata.
2. Verify self.node is assigned follower.
3. Verify leader_epoch matches.
4. Load local follower watermark.
5. If batch.base_offset < local LEO: skip duplicate or validate byte equality.
6. If batch.base_offset > local LEO: return gap error.
7. Write batch object in follower broker namespace.
8. Advance local LEO.
```

## SQL backends

For SQL, the simplest safe implementation is one row per batch per broker replica:

```sql
CREATE TABLE IF NOT EXISTS replica_log_batch (
    cluster_id TEXT NOT NULL,
    broker_id INTEGER NOT NULL,
    topic_name TEXT NOT NULL,
    partition_id INTEGER NOT NULL,
    base_offset BIGINT NOT NULL,
    last_offset BIGINT NOT NULL,
    leader_epoch INTEGER NOT NULL,
    batch_bytes BLOB NOT NULL,
    created_at_ms BIGINT NOT NULL,
    PRIMARY KEY (cluster_id, broker_id, topic_name, partition_id, base_offset)
);
```

Truncate:

```sql
DELETE FROM replica_log_batch
WHERE cluster_id = ?
  AND broker_id = ?
  AND topic_name = ?
  AND partition_id = ?
  AND base_offset >= ?;
```

For partial-batch truncation, fetch and rewrite the affected row.

## Null/proxy backends

`null.rs` can return unsupported or single-replica behavior.

`proxy.rs` can delegate replication APIs if the target supports them.

# Workstream 15: OffsetForLeaderEpoch API

## Goal

Support follower reconciliation and Kafka client compatibility.

If `OffsetForLeaderEpoch` is already generated, add route and service:

```text

```

Add module:

```rust

```

Route it:

```rust
use tansu_sans_io::OffsetForLeaderEpochRequest;
use tansu_storage::OffsetForLeaderEpochService;

pub fn offset_for_leader_epoch<S>(
    builder: FrameRouteBuilder<(), Error>,
    storage: S,
) -> Result<FrameRouteBuilder<(), Error>, Error>
where
    S: Storage + Clone,
{
    builder
        .with_route(
            OffsetForLeaderEpochRequest::KEY,
            (
                MapErrLayer::new(Error::from),
                MapStateLayer::new(|_| storage),
                FrameRequestLayer::<OffsetForLeaderEpochRequest>::new(),
            )
                .into_layer(OffsetForLeaderEpochService)
                .boxed(),
        )
        .map_err(Into::into)
}
```

Response logic:

```text
For each requested topic partition:
  validate current leader epoch
  find requested leader_epoch in checkpoint list
  return end offset for that epoch
```

If requested epoch is the current epoch:

```text
end offset = current leader LEO
```

If requested epoch is older:

```text
end offset = start_offset of next epoch
```

If epoch unknown:

```text
return UNDEFINED_EPOCH_OFFSET or appropriate error
```

# Main risks and how to avoid them

## Risk 1: shared storage hides replication bugs

If every broker reads the same batch path, followers are fake. Use per-broker log namespaces.

## Risk 2: HW confused with LEO

Current watermark high behaves more like local log end offset. Phase 11 must split:

```text
LEO = local next offset on each broker
HW = committed cluster offset for partition
LSO = transactional visibility offset
```

## Risk 3: follower fetch accidentally limited by HW

Follower fetch must read leader data up to LEO, not HW. Consumer fetch is HW-limited.

## Risk 4: leader changes without epochs

A leader switch without persisted leader epoch history will eventually corrupt logs. Implement epochs before calling Phase 11 complete.

## Risk 5: ISR updates not atomic

ISR and HW updates must be persisted together under compare-and-swap/versioned metadata. DynoStore already has optimistic update machinery around metadata; use that pattern for partition replication state. 

## Risk 6: Phase 12 scope creep

Do not build the full controller here. Implement primitives:

```text
register broker
assign replicas
manual/test leader change
leader epoch checkpoint
follower reconciliation
```

Then Phase 12 can own automatic broker failure detection and elections.
