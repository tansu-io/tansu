## Phase 11 goal: a Kafka-compatible replication layer

For Tansu specifically, Phase 11 is a big architectural shift. The repo currently describes Tansu as a “stateless Kafka-compatible broker with pluggable storage,” and its README explicitly notes that in the current example “node 111 is the leader and ISR for each topic partition” and “all brokers are node 111.” ([GitHub][1]) ([GitHub][1])

Kafka’s model is different: the unit of replication is the topic partition; each partition has one leader and zero or more followers; followers pull from the leader; and ISR membership is based on broker liveness plus follower catch-up progress. ([Apache Kafka][2]) A write is not considered committed until every in-sync replica has received it, and the ISR is persisted in cluster metadata. ([Apache Kafka][2])

So Phase 11 should produce a **classic Kafka replication mode** while preserving Tansu’s current shared-storage/stateless mode behind a config switch. Do not rip out the existing design. Add:

```text
replication.mode = shared-storage | classic
```

For Phase 11 completion, `classic` mode must support:

1. Leader/follower partition roles.
2. Real replica assignments in metadata.
3. Followers fetching from leaders.
4. Exact-offset follower appends.
5. ISR shrink/expand.
6. High watermark advancement from ISR replica progress.
7. Persistent leader epochs.
8. OffsetForLeaderEpoch-based truncation on leader change.
9. Produce/fetch error semantics for wrong leader.
10. End-to-end tests with three brokers, leader failover simulation, and divergent-log truncation.

Phase 12 can replace the static/controller shim with a real cluster controller. Phase 11 still needs enough controller-facing hooks to receive/apply leadership changes.

## Current repo baseline

The workspace already has the right split: `tansu-broker`, `tansu-client`, `tansu-sans-io`, `tansu-service`, `tansu-storage`, and related crates are workspace members. The workspace also forbids unsafe code, which is excellent for a Rust replication implementation. 

The current `Storage` trait supports broker registration, topic creation, produce, fetch, offset stage, list offsets, offset commits, metadata, groups, configs, transactions, maintenance, and ping, but it does **not** expose replica state, follower progress, leader epochs, exact-offset follower append, or truncation primitives. ([GitHub][3])

The broker already registers itself at startup, then listens for client connections, and it has a periodic maintenance tick that calls `storage.maintain(SystemTime::now())`. That is a perfect place to attach broker heartbeat, ISR checks, and replication manager maintenance. 

The current broker route table wires many Kafka APIs, including Produce, Fetch, Metadata, CreateTopics, DeleteTopics, ListOffsets, and transactions, but it does not wire `LeaderAndIsr`, `AlterPartition`, or `OffsetForLeaderEpoch` services yet.  The protocol JSON files already exist for `LeaderAndIsrRequest`, `AlterPartitionRequest`, and `OffsetForLeaderEpochRequest`, so this phase should add service handlers rather than inventing a private protocol first.   

The current `ProduceService` calls `Storage::produce` directly and returns `current_leader(None)`, so Phase 11 must add leader validation and return Kafka leader errors when the request lands on a follower.  The current `FetchService` calls `Storage::fetch`, then `Storage::offset_stage`, and also returns `current_leader(None)`; Phase 11 must split **client fetch** from **replica fetch**, because followers must be allowed to fetch beyond the committed high watermark while clients must not.

# The architecture to build

## The core rule

Add a `ReplicaManager` at the broker layer and replication primitives at the storage layer.

```text
client Produce/Fetch
        |
        v
tansu-broker request router
        |
        v
tansu-storage services
        |
        v
Storage trait
        |
        +--> local partition log / durable storage
        +--> partition replica state
        +--> leader epoch cache
        +--> follower progress table
        +--> ISR/high watermark state
```

The broker owns network-facing replication tasks:

```text
Leader broker:
  - accepts Produce
  - serves follower Fetch
  - records follower progress
  - advances HW
  - shrinks/expands ISR
  - proposes ISR updates through AlterPartition/controller path

Follower broker:
  - discovers leader for assigned partitions
  - calls OffsetForLeaderEpoch
  - truncates divergent tail
  - fetches from leader
  - appends exact offsets locally
  - repeats forever until role changes
```

Storage owns durable truth:

```text
PartitionReplicaState:
  leader
  leader_epoch
  partition_epoch
  replicas
  isr
  adding_replicas
  removing_replicas
  recovery_state

ReplicaProgress:
  broker_id
  log_end_offset
  high_watermark
  last_fetch_at
  last_caught_up_at

LeaderEpochCheckpoint:
  leader_epoch
  start_offset
  end_offset / next epoch start
```

# Completion criteria for Phase 11

Phase 11 is not complete until all of these are true:

| Area            | Must be true                                                                                                                                                                                                                                            |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Metadata        | `MetadataResponse` and `DescribeTopicPartitionsResponse` show real brokers, leader, replicas, ISR, offline replicas, and leader epoch.                                                                                                                  |
| Produce         | Produce to leader succeeds; produce to follower returns `NotLeaderOrFollower` with `current_leader`; `acks=all` respects `min.insync.replicas`.                                                                                                         |
| Client fetch    | Client fetch returns records only up to high watermark.                                                                                                                                                                                                 |
| Replica fetch   | Follower fetch returns records up to leader LEO, not merely HW.                                                                                                                                                                                         |
| Follower append | Follower appends batches at the leader’s exact offsets; offsets and batch order match the leader.                                                                                                                                                       |
| ISR shrink      | A follower that stops fetching or falls behind is removed from ISR after `replica.lag.time.max.ms`. Kafka documents this config as the point after which a follower that has not fetched or caught up is removed from ISR. ([Apache Kafka][4])          |
| ISR expand      | A follower re-enters ISR only after catching up to the leader’s committed/log end position according to the configured policy.                                                                                                                          |
| HW advancement  | HW advances to the minimum LEO of ISR replicas, capped by leader LEO.                                                                                                                                                                                   |
| Leader epochs   | Each leadership change appends a persistent epoch checkpoint.                                                                                                                                                                                           |
| Truncation      | On leader change, followers use OffsetForLeaderEpoch and truncate divergent tails before fetching. Kafka’s OffsetForLeaderEpoch protocol includes `current_leader_epoch`, `leader_epoch`, and fencing semantics for old/new epochs. ([Apache Kafka][5]) |
| Crash/restart   | Broker restarts recover role, LEO, HW, ISR state, and leader epoch cache.                                                                                                                                                                               |
| Compatibility   | Kafka Java client and librdkafka can produce/fetch through leader changes without offset corruption.                                                                                                                                                    |

## 11.2 Add high watermark calculation

Create `tansu-storage/src/replication/high_watermark.rs`.

```rust
use std::collections::BTreeMap;

use super::{PartitionReplicaState, ReplicaProgress};

pub fn compute_high_watermark(
    state: &PartitionReplicaState,
    leader_log_end_offset: i64,
    progress: &BTreeMap<i32, ReplicaProgress>,
) -> i64 {
    let mut hw = leader_log_end_offset;

    for broker_id in &state.isr {
        let leo = if *broker_id == state.leader {
            leader_log_end_offset
        } else {
            progress
                .get(broker_id)
                .map(|p| p.log_end_offset)
                .unwrap_or(0)
        };

        hw = hw.min(leo);
    }

    hw.max(0)
}
```

* Store HW as the **next readable committed offset** consistently. Existing code appears to use offsets as positions, so keep this convention: if records 0..9 are committed, HW is 10.
* Client fetch can return batches whose last offset is `< HW`.
* Follower fetch can return batches beyond HW up to leader LEO.

## 11.3 Add ISR shrink/expand logic

Create `tansu-storage/src/replication/isr.rs`.

```rust
use std::collections::BTreeMap;
use std::time::{Duration, SystemTime};

use super::{PartitionReplicaState, ReplicaProgress};

#[derive(Clone, Copy, Debug, Eq, PartialEq)]
pub enum IsrDecision {
    Keep,
    Shrink,
    Expand,
}

pub fn follower_should_leave_isr(
    now: SystemTime,
    replica: &ReplicaProgress,
    leader_log_end_offset: i64,
    replica_lag_time_max: Duration,
) -> bool {
    if replica.log_end_offset >= leader_log_end_offset {
        return false;
    }

    now.duration_since(replica.last_caught_up_at.unwrap_or(replica.last_fetch_at))
        .map(|lag| lag >= replica_lag_time_max)
        .unwrap_or(false)
}

pub fn follower_can_join_isr(
    replica: &ReplicaProgress,
    leader_log_end_offset: i64,
) -> bool {
    replica.log_end_offset >= leader_log_end_offset
}

pub fn compute_isr(
    now: SystemTime,
    state: &PartitionReplicaState,
    leader_log_end_offset: i64,
    progress: &BTreeMap<i32, ReplicaProgress>,
    replica_lag_time_max: Duration,
) -> Vec<i32> {
    let mut next = Vec::with_capacity(state.replicas.len());

    for broker_id in &state.replicas {
        if *broker_id == state.leader {
            next.push(*broker_id);
            continue;
        }

        let Some(replica) = progress.get(broker_id) else {
            continue;
        };

        let in_current_isr = state.isr.contains(broker_id);

        if in_current_isr {
            if !follower_should_leave_isr(
                now,
                replica,
                leader_log_end_offset,
                replica_lag_time_max,
            ) {
                next.push(*broker_id);
            }
        } else if follower_can_join_isr(replica, leader_log_end_offset) {
            next.push(*broker_id);
        }
    }

    next.sort_unstable();
    next.dedup();
    next
}
```

Steering:

* Do **not** expand ISR just because a follower is alive. Expand only after it catches up.
* Do **not** shrink ISR on a single failed fetch; use the lag time threshold.
* In Kafka-like behavior, `replica.fetch.wait.max.ms` should be lower than `replica.lag.time.max.ms` to avoid low-throughput topics constantly shrinking ISR. Kafka’s broker config docs call out this relationship. ([Apache Kafka][4])

## 11.4 Add leader epoch cache

Create `tansu-storage/src/replication/epoch.rs`.

```rust
use super::{LeaderEpochCheckpoint, LeaderEpochEndOffset};

pub fn end_offset_for_epoch(
    checkpoints: &[LeaderEpochCheckpoint],
    requested_epoch: i32,
    log_end_offset: i64,
) -> LeaderEpochEndOffset {
    let mut sorted = checkpoints.to_vec();
    sorted.sort_by_key(|c| c.leader_epoch);

    for window in sorted.windows(2) {
        let current = &window[0];
        let next = &window[1];

        if current.leader_epoch == requested_epoch {
            return LeaderEpochEndOffset {
                leader_epoch: current.leader_epoch,
                end_offset: next.start_offset,
            };
        }
    }

    if let Some(last) = sorted.last() {
        if requested_epoch >= last.leader_epoch {
            return LeaderEpochEndOffset {
                leader_epoch: last.leader_epoch,
                end_offset: log_end_offset,
            };
        }
    }

    LeaderEpochEndOffset {
        leader_epoch: requested_epoch,
        end_offset: 0,
    }
}
```

Storage must persist checkpoints like:

```text
epoch 0 starts at offset 0
epoch 1 starts at offset 125
epoch 2 starts at offset 198
```

Then:

```text
OffsetForLeaderEpoch(epoch 0) => end_offset 125
OffsetForLeaderEpoch(epoch 1) => end_offset 198
OffsetForLeaderEpoch(epoch 2) => end_offset current LEO
```

Steering:

* Append a checkpoint every time a broker becomes leader for a partition.
* The checkpoint start offset is the local LEO at leadership start.
* Never overwrite existing epoch checkpoints except during truncation of divergent future epochs.
* Persist epoch checkpoints separately from segment files or record rows.

## 11.5 Add truncation decision logic

Create `tansu-storage/src/replication/truncate.rs`.

```rust
use super::LeaderEpochEndOffset;

#[derive(Clone, Copy, Debug, Eq, PartialEq)]
pub enum Truncation {
    None,
    ToOffset(i64),
}

pub fn decide_truncation(
    local_log_end_offset: i64,
    leader_epoch_end_offset: LeaderEpochEndOffset,
) -> Truncation {
    let target = leader_epoch_end_offset.end_offset;

    if target < local_log_end_offset {
        Truncation::ToOffset(target)
    } else {
        Truncation::None
    }
}
```

Steering:

* This is intentionally simple. The hard work is choosing the right epoch to query.
* A follower should query the leader for the highest local leader epoch it knows.
* If the leader returns a lower end offset than local LEO, truncate.
* After truncation, remove leader epoch checkpoints and producer-state snapshots after the target offset.
* Never truncate committed offsets below local HW unless this is explicitly an unclean election mode.

```rust
pub fn assign_replicas_round_robin(
    broker_ids: &[i32],
    partitions: i32,
    replication_factor: i16,
) -> Vec<Vec<i32>> {

        assignments.push(replicas);
    }

    assignments
}
```

* Initial ISR should include all assigned replicas only if all assigned brokers are registered/live. Otherwise, initial ISR should include the leader and live caught-up replicas.
* Initial leader can be `replicas[0]` until Phase 12 does balanced election.

```rust
async fn wait_for_commit<G>(
    ctx: &Context<G>,
    tp: &Topition,
    target_end_offset: i64,
```

# Fetch path changes

Touch `tansu-storage/src/service/fetch.rs`.

Kafka FetchRequest includes follower/replica fields in protocol versions; the official protocol describes follower-specific fields such as `replica_id`, `last_fetched_epoch`, and follower `log_start_offset`. ([Apache Kafka][5])

## Split fetch into two modes

```rust
    Client,
    Replica { replica_id: i32 },

```

## Client fetch

Client fetch should:

* Verify this broker is current leader or allowed follower read replica.
* Cap records at HW.
* Return `NotLeaderOrFollower` with current leader if this node is not allowed to serve the partition.
* Preserve existing long-polling behavior.

## Replica fetch

Replica fetch should:

* Only be served by the current leader.
* Verify requesting `replica_id` is in `state.replicas`.
* Use leader LEO, not HW, as read bound.
* Update follower progress based on requested fetch offset and returned data.
* Return leader epoch and HW in partition response.
* Set `current_leader` when epoch fencing happens.

Pseudo-code:

```rust
async fn fetch_partition_for_replica<G>(
    &self,
    ctx: &Context<G>,
    replica_id: i32,
    topic: &str,
    fetch_partition: &FetchPartition,
    min_bytes: u32,
    max_bytes: &mut u32,
) -> Result<PartitionData>
where
    G: Storage,
{
    let tp = Topition::new(topic, fetch_partition.partition);

    let Some(state) = ctx.state().partition_replica_state(&tp).await? else {
        return Ok(self.unknown_partition(fetch_partition.partition));
    };

    if state.leader != ctx.state().node() {
        return Ok(self.not_leader_partition(fetch_partition.partition, &state));
    }

    if !state.contains_replica(replica_id) {
        return Ok(self.error_partition(
            fetch_partition.partition,
            ErrorCode::UnknownServerError,
        ));
    }

    if let Some(epoch) = fetch_partition.current_leader_epoch {
        if epoch < state.leader_epoch {
            return Ok(self.fenced_epoch_partition(fetch_partition.partition, &state));
        }

        if epoch > state.leader_epoch {
            return Ok(self.unknown_epoch_partition(fetch_partition.partition, &state));
        }
    }

    let batches = ctx
        .state()
        .fetch_for_replica(
            &tp,
            replica_id,
            fetch_partition.fetch_offset,
            min_bytes,
            *max_bytes,
        )
        .await?;

    let fetched_bytes = batches.iter().map(|b| b.record_data.len() as u32).sum::<u32>();
    *max_bytes = max_bytes.saturating_sub(fetched_bytes);

    let follower_leo = batches
        .last()
        .map(|batch| batch.base_offset + batch.record_count as i64)
        .unwrap_or(fetch_partition.fetch_offset);

    let progress = ReplicaProgress {
        broker_id: replica_id,
        log_end_offset: follower_leo,
        high_watermark: ctx.state().offset_stage(&tp).await?.high_watermark(),
        last_fetch_at: SystemTime::now(),
        last_caught_up_at: None,
    };

    let progress = ctx.state().update_replica_progress(&tp, progress).await?;

    let leader_leo = ctx.state().local_log_end_offset(&tp).await?;
    let next_hw = compute_high_watermark(&state, leader_leo, &progress.replicas);

    ctx.state().update_high_watermark(&tp, next_hw).await?;

    Ok(PartitionData::default()
        .partition_index(fetch_partition.partition)
        .error_code(ErrorCode::None.into())
        .high_watermark(next_hw)
        .last_stable_offset(Some(next_hw))
        .log_start_offset(Some(progress.log_start_offset))
        .current_leader(Some(
            LeaderIdAndEpoch::default()
                .leader_id(state.leader)
                .leader_epoch(state.leader_epoch),
        ))
        .records(if batches.is_empty() {
            None
        } else {
            Some(Frame { batches })
        }))
}
```

Steering:

* The follower’s fetch offset is the follower’s local LEO.
* For replica fetch, do not hide uncommitted tail records.
* For client fetch, continue hiding records above HW.
* Use `current_leader_epoch` fencing in both client and replica fetch paths.
* Existing fetch code loops topic/partitions and long-polls; preserve that structure and branch at partition level.

# Follower append semantics

Add `Storage::append_follower_batch`.

This is not `produce()`. It must append a batch at the leader’s exact offset.

```rust
async fn append_follower_batch(
    &self,

    leader_epoch: i32,
    expected_base_offset: i64,
    batch: deflated::Batch,

    // Persist batch exactly at expected_base_offset.
    // Update local LEO to expected_base_offset + batch.record_count.
    // Do not advance HW here unless the leader sent a higher HW.

}
```

* Preserve the original batch bytes exactly.
* Do not rewrite base offsets.
* Do not reassign timestamps.

# LeaderAndIsr service

Create `tansu-storage/src/service/leader_and_isr.rs`.

This API is the bridge from controller/static shim to brokers. Phase 12 will produce these commands from a real controller; Phase 11 should handle them correctly.

Responsibilities:

1. Validate controller epoch/broker epoch when present.
2. Convert request partition states to `PartitionReplicaState`.
3. For each partition:

   * If local broker is new leader, call `become_leader`.
   * If local broker is follower, call `become_follower`.
   * If local broker is no longer replica, stop local follower task and optionally delete/quiesce replica.
4. Persist state.
5. Append leader epoch checkpoint when becoming leader.
6. Return partition-level errors.

Skeleton:

```rust
pub async fn apply_leader_and_isr_partition<G>(
    storage: &G,
    local_broker_id: i32,
    topic: &str,
    partition: &LeaderAndIsrPartitionState,
) -> Result<()>
where
    G: Storage,
{
    let tp = Topition::new(topic, partition.partition_index);

    let previous = storage.partition_replica_state(&tp).await?;

    let next = PartitionReplicaState {
        topition: tp.clone(),
        leader: partition.leader,
        leader_epoch: partition.leader_epoch,
        partition_epoch: partition.partition_epoch,
        replicas: partition.replicas.clone().unwrap_or_default(),
        isr: partition.isr.clone().unwrap_or_default(),
        adding_replicas: partition.adding_replicas.clone().unwrap_or_default(),
        removing_replicas: partition.removing_replicas.clone().unwrap_or_default(),
        recovery_state: partition
            .leader_recovery_state
            .map(LeaderRecoveryState::try_from)
            .transpose()?
            .unwrap_or_default(),
    };

    let became_leader = previous
        .as_ref()
        .map(|old| old.leader != local_broker_id && next.leader == local_broker_id)
        .unwrap_or(next.leader == local_broker_id);

    storage.update_partition_replica_state(next.clone()).await?;

    if became_leader {
        let leo = storage.local_log_end_offset(&tp).await?;
        storage
            .append_leader_epoch_checkpoint(LeaderEpochCheckpoint {
                topition: tp,
                leader_epoch: next.leader_epoch,
                start_offset: leo,
            })
            .await?;
    }

    Ok(())
}
```

Steering:

* Do not start network follower loops from storage service directly. Persist state and let broker `ReplicationManager` watch state.
* `LeaderAndIsr` should be idempotent. Reapplying the same state must be harmless.
* If leader epoch regresses, reject or ignore with a stale-controller/stale-epoch error.

# AlterPartition service

Create `tansu-storage/src/service/alter_partition.rs`.

This is how a leader proposes ISR changes. The repo already has `AlterPartitionRequest.json` with `BrokerId`, `BrokerEpoch`, topic data, partitions, `LeaderEpoch`, and ISR fields. 

Responsibilities:

1. Validate requester is current leader.
2. Validate request leader epoch equals stored leader epoch.
3. Validate new ISR is subset of replicas and includes leader.
4. Atomically update ISR and increment partition epoch.
5. Recompute HW after ISR change.
6. Return updated partition state.

Pseudo-code:

```rust
async fn alter_partition_isr<G>(
    storage: &G,
    broker_id: i32,
    tp: Topition,
    leader_epoch: i32,
    requested_isr: Vec<i32>,
) -> Result<PartitionReplicaState>
where
    G: Storage,
{
    let Some(mut state) = storage.partition_replica_state(&tp).await? else {
        return Err(Error::Api(ErrorCode::UnknownTopicOrPartition));
    };

    if state.leader != broker_id {
        return Err(Error::Api(ErrorCode::NotLeaderOrFollower));
    }

    if state.leader_epoch != leader_epoch {
        return Err(Error::Api(ErrorCode::FencedLeaderEpoch));
    }

    if !requested_isr.contains(&state.leader) {
        return Err(Error::Api(ErrorCode::InvalidRequest));
    }

    for broker in &requested_isr {
        if !state.replicas.contains(broker) {
            return Err(Error::Api(ErrorCode::InvalidRequest));
        }
    }

    state.isr = requested_isr;
    state.isr.sort_unstable();
    state.isr.dedup();
    state.partition_epoch += 1;

    storage.update_partition_replica_state(state.clone()).await?;

    let progress = storage.replica_progress(&tp).await?;
    let leader_leo = storage.local_log_end_offset(&tp).await?;
    let hw = crate::replication::high_watermark::compute_high_watermark(
        &state,
        leader_leo,
        &progress.replicas,
    );

    storage.update_high_watermark(&tp, hw).await?;

    Ok(state)
}
```

Steering:

* This method will become controller-mediated in Phase 12.
* For Phase 11, the leader can call local storage directly in tests or through this API.
* Keep `partition_epoch` monotonic and persist it.

## Follower task reconciliation

Create `tansu-broker/src/replication/watch.rs`.

```rust
#[derive(Default)]
struct FollowerTaskSet {
    handles: BTreeMap<Topition, tokio::task::JoinHandle<()>>,
}

impl FollowerTaskSet {
    async fn reconcile<S>(
        &mut self,
        storage: S,
        config: ReplicationConfig,
        states: Vec<PartitionReplicaState>,
    ) -> Result<(), Error>
    where
        S: Storage + Clone + Send + Sync + 'static,
    {
        let wanted: BTreeSet<Topition> = states
            .iter()
            .filter(|state| {
                state.leader != config.broker_id
                    && state.replicas.contains(&config.broker_id)
            })
            .map(|state| state.topition.clone())
            .collect();

        for state in states {
            if state.leader == config.broker_id {
                continue;
            }

            if !state.replicas.contains(&config.broker_id) {
                continue;
            }

            if self.handles.contains_key(&state.topition) {
                continue;
            }

            let task = FollowerReplicationTask::new(
                storage.clone(),
                config.clone(),
                state,
            );

            let topition = task.topition().clone();

            let handle = tokio::spawn(async move {
                if let Err(error) = task.run().await {
                    tracing::warn!(?error, "follower replication task exited");
                }
            });

            self.handles.insert(topition, handle);
        }

        let existing = self.handles.keys().cloned().collect::<Vec<_>>();

        for topition in existing {
            if !wanted.contains(&topition) {
                if let Some(handle) = self.handles.remove(&topition) {
                    handle.abort();
                }
            }
        }

        Ok(())
    }

    async fn stop_all(&mut self) {
        for (_, handle) in std::mem::take(&mut self.handles) {
            handle.abort();
        }
    }
}
```

```rust
impl<S> FollowerReplicationTask<S>
where
    S: Storage + Clone + Send + Sync + 'static,
{
    pub fn topition(&self) -> &Topition {
        &self.state.topition
    }

    pub async fn run(mut self) -> Result<(), Error> {
        loop {
            self.refresh_state().await?;

            if self.state.leader == self.config.broker_id {
                return Ok(());
            }

            self.truncate_to_leader_if_needed().await?;
            self.fetch_once().await?;

            tokio::time::sleep(Duration::from_millis(10)).await;
        }
    }

    async fn truncate_to_leader_if_needed(&self) -> Result<()> {
        let checkpoints = self
            .storage
            .leader_epoch_checkpoints(&self.state.topition)
            .await?;

        let Some(last_local_epoch) = checkpoints.iter().map(|c| c.leader_epoch).max() else {
            return Ok(());
        };

        let leader = self.connect_to_leader().await?;

        let leader_end = leader
            .offset_for_leader_epoch(
                &self.state.topition,
                self.state.leader_epoch,
                last_local_epoch,
            )
            .await?;

        let local_leo = self
            .storage
            .local_log_end_offset(&self.state.topition)
            .await?;

        match tansu_storage::replication::truncate::decide_truncation(local_leo, leader_end) {
            Truncation::None => Ok(()),
            Truncation::ToOffset(offset) => {
                self.storage.truncate_to_offset(&self.state.topition, offset).await
            }
        }
    }

    async fn fetch_once(&self) -> Result<()> {
        let leader = self.connect_to_leader().await?;

        let offset = self
            .storage
            .local_log_end_offset(&self.state.topition)
            .await?;

        let response = leader
            .fetch_replica(
                self.config.broker_id,
                &self.state.topition,
                self.state.leader_epoch,
                offset,
                self.config.replica_fetch_min_bytes,
                self.config.replica_fetch_max_bytes,
                self.config.replica_fetch_wait_max,
            )
            .await?;

        for batch in response.batches {
            self.storage
                .append_follower_batch(
                    &self.state.topition,
                    self.state.leader_epoch,
                    batch.base_offset,
                    batch,
                )
                .await?;
        }

        if let Some(hw) = response.high_watermark {
            self.storage
                .update_high_watermark(&self.state.topition, hw)
                .await?;
        }

        Ok(())
    }
}
```

Steering:

* Use exponential backoff on network errors, but do not sleep so long that ISR shrinks too aggressively.
* Re-read partition state before each fetch loop iteration.
* Stop if local broker is no longer a follower for the partition.
* Stop if leader changes; restart truncation handshake against the new leader.
* Use `OffsetForLeaderEpoch` before fetch after every leader epoch change.
* The follower should include its local log start offset in FetchRequest when the generated type supports it.

# Leader maintenance

Create `tansu-broker/src/replication/leader.rs`.

Leader maintenance should run periodically for partitions where local broker is leader.

```rust
impl<S> ReplicationManager<S>
where
    S: Storage + Clone + Send + Sync + 'static,
{
    async fn maintain_leader_partitions(&self) -> Result<(), Error> {
        let states = self
            .storage
            .partition_replica_states_for_broker(self.config.broker_id)
            .await?;

        let now = SystemTime::now();

        for state in states {
            if state.leader != self.config.broker_id {
                continue;
            }

            let progress = self.storage.replica_progress(&state.topition).await?;
            let leader_leo = self.storage.local_log_end_offset(&state.topition).await?;

            let next_isr = tansu_storage::replication::isr::compute_isr(
                now,
                &state,
                leader_leo,
                &progress.replicas,
                self.config.replica_lag_time_max,
            );

            if next_isr != state.isr {
                self.update_isr(state.clone(), next_isr).await?;
            }

            let latest = self
                .storage
                .partition_replica_state(&state.topition)
                .await?
                .unwrap_or(state);

            let progress = self.storage.replica_progress(&latest.topition).await?;
            let hw = tansu_storage::replication::high_watermark::compute_high_watermark(
                &latest,
                leader_leo,
                &progress.replicas,
            );

            self.storage.update_high_watermark(&latest.topition, hw).await?;
        }

        Ok(())
    }

    async fn update_isr(
        &self,
        mut state: PartitionReplicaState,
        next_isr: Vec<i32>,
    ) -> Result<()> {
        state.isr = next_isr;
        state.partition_epoch += 1;

        self.storage
            .update_partition_replica_state(state)
            .await?;

        Ok(())
    }
}
```

Steering:

* Initially call storage directly; later Phase 12 can enforce controller-mediated ISR changes.
* Emit metrics for `isr_shrink_total`, `isr_expand_total`, `partition_hw`, `partition_leo`, and `replica_lag`.
* High watermark checkpointing should persist periodically, not merely in memory.

```rust
pub async fn force_leader<S>(
    storage: &S,
    topition: &Topition,
    new_leader: i32,
) -> Result<PartitionReplicaState>
where
    S: Storage,
{
    let Some(mut state) = storage.partition_replica_state(topition).await? else {
        return Err(Error::Api(ErrorCode::UnknownTopicOrPartition));
    };

    if !state.isr.contains(&new_leader) {
        return Err(Error::Api(ErrorCode::PreferredLeaderNotAvailable));

            topition: topition.clone(),
            leader_epoch: state.leader_epoch,
            start_offset: leo,
        })
        .await?;

    Ok(state)
}
```

# Important edge cases

## Produce during leader change

A produce request with stale metadata must fail, not append.

Rules:

```text
if request leader epoch < current epoch => FencedLeaderEpoch
if this broker is not leader => NotLeaderOrFollower
if this broker was leader but has been demoted => NotLeaderOrFollower
```

## Fetch during leader change

Client fetch:

```text
old leader => NotLeaderOrFollower
new leader with HW not caught up => OffsetNotAvailable or empty response until HW is safe
```

Replica fetch:

```text
old epoch => FencedLeaderEpoch
future epoch => UnknownLeaderEpoch
wrong leader => NotLeaderOrFollower
```

## ISR empty or leader not in ISR

Do not allow persisted ISR without the leader unless implementing newer ELR semantics. For Phase 11, enforce:

```rust
assert!(state.isr.contains(&state.leader));
```

## Min ISR

For `acks=all`:

```text
ISR size < min.insync.replicas before append => NotEnoughReplicas
append succeeds but cannot commit before timeout => RequestTimedOut or NotEnoughReplicasAfterAppend
```

## Transactions

Phase 14 owns full transactions, but Phase 11 must not break scaffolding:

* HW cap applies to `read_uncommitted`.
* `last_stable_offset` must be `min(transaction_lso, high_watermark)` for `read_committed`.
* Transaction markers must replicate like normal batches.
* Follower promotion must recover producer/transaction state from replicated log.

## Compaction / retention

Phase 13 owns full retention/compaction, but Phase 11 must protect replication:

* Do not delete records below follower fetch offset if follower is in ISR.
* If follower falls behind retention, remove from ISR and require full replica rebuild.
* Return offset-out-of-range to follower when fetch offset < leader log start; follower must truncate/rebuild.

# Final Phase 11 “done” checklist

Use this as the merge gate:

```text
[ ] Storage has durable PartitionReplicaState.
[ ] Metadata returns real brokers, leaders, replicas, ISR, leader epochs.
[ ] CreateTopics assigns replicas correctly for explicit and implicit assignments.
[ ] Produce validates current leader.
[ ] Produce acks=all waits for HW.
[ ] Produce respects min.insync.replicas.
[ ] Client fetch is HW-capped.
[ ] Replica fetch is LEO-capped.
[ ] Replica fetch updates follower progress.
[ ] HW = min LEO of ISR replicas.
[ ] ISR shrinks after lag timeout.
[ ] ISR expands only after full catch-up.
[ ] Leader epoch checkpoint is appended on every leadership change.
[ ] OffsetForLeaderEpoch fences stale/future epochs.
[ ] Followers truncate divergent tails before fetching from new leader.
[ ] Follower append preserves exact leader offsets and batch bytes.
[ ] Broker replication manager starts/stops with broker lifecycle.
[ ] Broker heartbeat is persisted.
[ ] Restart recovers LEO, HW, ISR, leader epochs, and replica role.
[ ] Multi-broker integration test passes.
[ ] Kafka CLI topic describe shows correct leader/replica/ISR data.
[ ] Existing produce/fetch/idempotent tests still pass.
[ ] Shared-storage mode still works as before.
```

The most important implementation principle: **never let client-visible data move past the high watermark, and never let a follower append at an offset chosen locally.** If those two rules hold, leader changes and truncation become tractable instead of terrifying.
