Absolutely — Phase 11 is the point where this project becomes a *real clustered Kafka-compatible broker* rather than a stateless Kafka API façade. The most important design decision is that Phase 11 should be implemented as a **data-plane replication layer with explicit leader/follower roles**, while leaving Phase 12 to decide *which broker should be leader* through a real controller/quorum.

Right now, Tansu’s public docs describe the current architecture as stateless brokers where all brokers are equal, brokers do not replicate storage to each other, there is no ISR concept, and every broker can act as leader for every partition. That is excellent for the current shared-storage model, but Phase 11 must deliberately introduce a new replicated mode rather than quietly changing the existing behavior. ([Tansu][1])

Below is a full Phase 11 completion plan.

---

# Phase 11 objective

Finish:

> Leader/follower replication, ISR tracking, high watermark advancement, leader epochs, follower fetch, truncation on leader change.

For Tansu, Phase 11 is complete when:

1. A partition has one leader and zero or more followers.
2. Producers can only append through the leader.
3. Followers continuously replicate from the leader using Kafka-compatible follower fetch semantics.
4. The leader tracks follower progress and maintains ISR.
5. The leader advances high watermark only when records are replicated to the current ISR.
6. Consumers only see committed data up to high watermark.
7. Leader epochs are persisted and exposed through `OffsetForLeaderEpoch`.
8. A broker that becomes follower after a leader change truncates any divergent tail before resuming replication.
9. Metadata, produce responses, fetch responses, and partition descriptions expose leader/replica/ISR information correctly.
10. The whole thing survives restart, follower lag, follower rejoin, leader step-down, and clean failover.

Phase 11 should **not** require a full autonomous controller yet. Phase 12 owns broker failure detection, leader election, metadata quorum, and partition reassignment. Phase 11 should expose the hooks that Phase 12 will call: “apply this partition leadership/ISR/epoch state,” “become leader,” “become follower,” “truncate to leader,” and “update ISR.”

# Key design rule

Do **not** bolt replication directly onto the existing `Storage::produce` / `Storage::fetch` pair only.

Those methods are too high-level. Phase 11 needs lower-level primitives:

```text
append as leader
append as follower
fetch local log range
truncate local log
persist leader epoch checkpoint
read epoch end offset
update follower progress
compute high watermark
persist high watermark
apply partition leadership state
```

The existing `ProduceService` currently loops over produce batches and calls `ctx.state().produce(transaction_id, &tp, batch)`. It also currently sets `current_leader(None)` in produce responses.  That path must become leader-aware and `acks`-aware.

The existing `FetchService` already obtains `OffsetStage` from storage and places `high_watermark`, `last_stable_offset`, and `log_start_offset` into `FetchResponse.PartitionData`.  Phase 11 should make those offsets real replicated offsets rather than local/shared-storage approximations.

# Kafka compatibility targets for Phase 11

Phase 11 should support these Kafka protocol behaviors:

| Behavior           | Kafka-facing rule                                                                            |
| ------------------ | -------------------------------------------------------------------------------------------- |
| Client fetch       | Normal consumers use `FetchRequest.replica_id = -1`                                          |
| Follower fetch     | Replica fetchers use broker ID as `replica_id`                                               |
| Follower log start | `log_start_offset` in fetch is meaningful for follower requests                              |
| Epoch fencing      | `current_leader_epoch` fences stale replicas/clients                                         |
| ISR update         | ISR is updated through internal state and eventually `AlterPartition`/controller integration |
| Leader state       | Metadata describes leader, replicas, and ISR                                                 |
| Epoch lookup       | `OffsetForLeaderEpoch` returns the leader’s epoch end offsets                                |

Apache Kafka’s protocol documentation describes `FetchRequest.replica_id` as the broker ID of the follower, with normal consumers using `-1`; it also describes `current_leader_epoch` as the field used to fence requests with stale metadata. ([Apache Kafka][2]) Kafka’s `LeaderAndIsr` schema includes leader, leader epoch, ISR, partition epoch, and replicas, while `AlterPartition` is used to submit ISR changes with leader epoch and partition epoch. ([Apache Kafka][3])

# Recommended Phase 11 architecture

Create a new crate:

```text
tansu-replication/
```

Why a new crate? Because replication is not just “storage” and not just “broker service.” It is a state machine and actor system that coordinates both. Keeping it separate gives you clean testing and prevents `tansu-storage` from becoming a giant control-plane/data-plane tangle.

The shape should be:

```text
Kafka client
   |
   v
tansu-broker
   |
   +-- tansu-storage service adapters
   |
   +-- tansu-replication manager
          |
          +-- leader partition actors
          +-- follower fetch actors
          +-- ISR/high-watermark tracker
          +-- leader epoch reconciler
          +-- internal broker client
   |
   v
tansu-storage replication primitives
   |
   +-- local partition log
   +-- epoch checkpoint store
   +-- replica state metadata
```

Phase 11 should add a replicated cluster mode:

```text
--cluster-mode=stateless     # current behavior
--cluster-mode=replicated    # Phase 11 behavior
```

or:

```text
--replication-mode=none
--replication-mode=broker
```

Keep the current stateless/shared-storage behavior intact. In replicated mode, every broker owns a **local replica log** for assigned partitions.

### `tansu-replication/src/high_watermark.rs`

The high watermark is the largest offset that is safely replicated to all in-sync replicas.

```rust
use crate::types::{BrokerId, Offset, ReplicaProgress};
use std::collections::{BTreeMap, BTreeSet};

pub fn compute_high_watermark(
    old_high_watermark: Offset,
    leader_id: BrokerId,
    leader_log_end_offset: Offset,
    isr: &BTreeSet<BrokerId>,
    progress: &BTreeMap<BrokerId, ReplicaProgress>,
) -> Offset {
    if isr.is_empty() {
        return old_high_watermark;
    }

    let mut candidate = leader_log_end_offset;

    for broker_id in isr {
        if *broker_id == leader_id {
            candidate = candidate.min(leader_log_end_offset);
        } else if let Some(replica) = progress.get(broker_id) {
            candidate = candidate.min(replica.log_end_offset);
        } else {
            // Missing ISR progress means do not advance.
            return old_high_watermark;
        }
    }

    candidate.max(old_high_watermark)
}
```

Important invariant:

```text
log_start_offset <= high_watermark <= leader_log_end_offset
```

High watermark must be monotonic during a stable leadership epoch. It may only be lowered as part of explicit recovery/truncation, and that recovery path must be carefully fenced by leader epoch.

### `tansu-replication/src/isr.rs`

```rust
use crate::config::ReplicationConfig;
use crate::types::{BrokerId, Offset, PartitionReplicaState};
use std::collections::BTreeSet;
use std::time::SystemTime;

pub fn recompute_isr(
    now: SystemTime,
    cfg: &ReplicationConfig,
    state: &PartitionReplicaState,
) -> BTreeSet<BrokerId> {
    let mut next = BTreeSet::new();

    // Leader is always in ISR while it is valid leader.
    next.insert(state.leader_id);

    for broker_id in &state.replicas {
        if *broker_id == state.leader_id {
            continue;
        }

        let Some(progress) = state.progress.get(broker_id) else {
            continue;
        };

        let caught_up_to_hw = progress.log_end_offset >= state.high_watermark;
        let lag_time_ok = progress
            .last_fetch_at
            .and_then(|last| now.duration_since(last).ok())
            .is_some_and(|lag| lag <= cfg.replica_lag_time_max);

        let lag_bytes_ok = cfg
            .replica_lag_bytes_max
            .map(|max_lag| {
                state
                    .leader_log_end_offset
                    .saturating_sub(progress.log_end_offset)
                    <= max_lag
            })
            .unwrap_or(true);

        if caught_up_to_hw && lag_time_ok && lag_bytes_ok {
            next.insert(*broker_id);
        }
    }

    next
}
```

Steering: do not immediately remove a follower from ISR just because a single fetch timed out. Use `replica_lag_time_max` and last successful fetch/progress timestamps. This prevents ISR flapping.

### `tansu-replication/src/follower.rs`

Follower actor responsibilities:

1. Watch partition assignments.
2. Detect when local broker is follower.
3. Connect to current leader.
4. Reconcile leader epoch.
5. Truncate divergent local log.
6. Fetch records from leader.
7. Append records locally preserving offsets.
8. Persist leader high watermark locally.
9. Repeat until cancelled or leadership changes.

Skeleton:

```rust
use crate::{
    client::ReplicationClient,
    config::ReplicationConfig,
    error::Result,
    truncate::reconcile_and_truncate,
};
use tansu_storage::{ReplicationStorage, Topition};
use tokio_util::sync::CancellationToken;
use tracing::{debug, error, instrument};

#[derive(Clone, Debug)]
pub struct FollowerTask<S, C> {
    pub storage: S,
    pub client: C,
    pub cfg: ReplicationConfig,
    pub topition: Topition,
    pub cancellation: CancellationToken,
}

impl<S, C> FollowerTask<S, C>
where
    S: ReplicationStorage + Clone + Send + Sync + 'static,
    C: ReplicationClient + Clone + Send + Sync + 'static,
{
    #[instrument(skip_all, fields(topic = self.topition.topic(), partition = self.topition.partition()))]
    pub async fn run(self) -> Result<()> {
        loop {
            tokio::select! {
                _ = self.cancellation.cancelled() => {
                    debug!("follower replication cancelled");
                    return Ok(());
                }

                result = self.replicate_once() => {
                    if let Err(err) = result {
                        error!(?err, "follower replication iteration failed");
                        tokio::time::sleep(self.cfg.follower_backoff).await;
                    }
                }
            }
        }
    }

    async fn replicate_once(&self) -> Result<()> {
        let state = self.storage.partition_replica_state(&self.topition).await?;

        if state.leader_id == self.cfg.node_id {
            tokio::time::sleep(self.cfg.follower_backoff).await;
            return Ok(());
        }

        reconcile_and_truncate(
            &self.storage,
            &self.client,
            &self.topition,
            state.leader_id,
            state.leader_epoch,
        )
        .await?;

        let local_leo = self.storage.log_end_offset(&self.topition).await?;

        let fetched = self
            .client
            .follower_fetch(
                state.leader_id,
                self.cfg.node_id,
                &self.topition,
                state.leader_epoch,
                local_leo,
                self.cfg.replica_fetch_min_bytes,
                self.cfg.replica_fetch_max_bytes,
                self.cfg.replica_fetch_max_wait,
            )
            .await?;

        if !fetched.records.is_empty() {
            self.storage
                .append_as_follower(&self.topition, state.leader_epoch, fetched.records)
                .await?;
        }

        self.storage
            .update_local_high_watermark(
                &self.topition,
                state.leader_epoch,
                fetched.high_watermark,
            )
            .await?;

        Ok(())
    }
}
```

### `tansu-replication/src/truncate.rs`

```rust
use crate::{client::ReplicationClient, error::Result};
use tansu_storage::{ReplicationStorage, Topition};
use tracing::{debug, instrument};

#[instrument(skip(storage, client))]
pub async fn reconcile_and_truncate<S, C>(
    storage: &S,
    client: &C,
    topition: &Topition,
    leader_id: i32,
    leader_epoch: i32,
) -> Result<()>
where
    S: ReplicationStorage + Send + Sync,
    C: ReplicationClient + Send + Sync,
{
    let local_epoch = storage.latest_leader_epoch(topition).await?;
    let local_leo = storage.log_end_offset(topition).await?;

    let leader_end = client
        .offset_for_leader_epoch(leader_id, topition, local_epoch)
        .await?;

    let truncate_to = leader_end.end_offset.min(local_leo);

    if truncate_to < local_leo {
        debug!(
            ?topition,
            local_epoch,
            local_leo,
            leader_epoch,
            leader_end_offset = leader_end.end_offset,
            truncate_to,
            "truncating divergent follower log"
        );

        storage.truncate_to(topition, truncate_to).await?;
    }

    storage.ensure_leader_epoch(topition, leader_epoch).await?;

    Ok(())
}
```

This is the essential “old leader rejoins as follower” safety mechanism.

### `tansu-replication/src/leader.rs`

Leader actor responsibilities:

1. Accept local leader appends.
2. Track follower fetch positions.
3. Update ISR.
4. Advance high watermark.
5. Wake pending `acks=all` produce waiters.
6. Fence stale epoch requests.

```rust
use crate::{
    config::ReplicationConfig,
    high_watermark::compute_high_watermark,
    isr::recompute_isr,
};
use std::time::SystemTime;
use tansu_storage::{ReplicationStorage, Topition};
use tracing::instrument;

#[derive(Clone, Debug)]
pub struct LeaderStateMachine<S> {
    pub storage: S,
    pub cfg: ReplicationConfig,
}

impl<S> LeaderStateMachine<S>
where
    S: ReplicationStorage + Clone + Send + Sync + 'static,
{
    #[instrument(skip(self))]
    pub async fn record_follower_progress(
        &self,
        topition: &Topition,
        follower_id: i32,
        log_end_offset: i64,
    ) -> tansu_storage::Result<i64> {
        let now = SystemTime::now();

        let mut state = self.storage.partition_replica_state(topition).await?;

        self.storage
            .update_replica_progress(topition, follower_id, log_end_offset, now)
            .await?;

        state = self.storage.partition_replica_state(topition).await?;

        let next_isr = recompute_isr(now, &self.cfg, &state);

        if next_isr != state.isr {
            self.storage
                .update_isr(
                    topition,
                    state.leader_epoch,
                    state.partition_epoch,
                    next_isr,
                )
                .await?;
        }

        state = self.storage.partition_replica_state(topition).await?;

        let next_hw = compute_high_watermark(
            state.high_watermark,
            state.leader_id,
            state.leader_log_end_offset,
            &state.isr,
            &state.progress,
        );

        if next_hw > state.high_watermark {
            self.storage
                .update_high_watermark(topition, state.leader_epoch, next_hw)
                .await?;
        }

        Ok(next_hw)
    }
}
```

```rust
use tansu_sans_io::{
    ApiKey, ErrorCode, OffsetForLeaderEpochRequest, OffsetForLeaderEpochResponse,
    offset_for_leader_epoch_response::{
        OffsetForLeaderEpochResponsePartition,
        OffsetForLeaderEpochResponseTopic,
    },

use crate::{Error, ReplicationStorage, Result, Topition};

#[derive(Clone, Copy, Debug, Default, Eq, Hash, Ord, PartialEq, PartialOrd)]
pub struct OffsetForLeaderEpochService;

impl ApiKey for OffsetForLeaderEpochService {
    const KEY: i16 = OffsetForLeaderEpochRequest::KEY;
}

impl<G> Service<G, OffsetForLeaderEpochRequest> for OffsetForLeaderEpochService
where
    G: ReplicationStorage,
{
    type Response = OffsetForLeaderEpochResponse;
    type Error = Error;

    async fn serve(
        &self,
        ctx: Context<G>,
        req: OffsetForLeaderEpochRequest,
    ) -> Result<Self::Response, Self::Error> {
        let mut topics = Vec::new();

        for topic in req.topics.unwrap_or_default() {
            let topic_name = topic.topic.clone();
            let mut partitions = Vec::new();

            for partition in topic.partitions.unwrap_or_default() {
                let tp = Topition::new(&topic_name, partition.partition);

                let end = ctx
                    .state()
                    .end_offset_for_leader_epoch(&tp, partition.leader_epoch)
                    .await;

                let response_partition = match end {
                    Ok(end) => OffsetForLeaderEpochResponsePartition::default()
                        .partition(partition.partition)
                        .error_code(ErrorCode::None.into())
                        .leader_epoch(Some(end.leader_epoch))
                        .end_offset(end.end_offset),

                    Err(Error::Api(code)) => OffsetForLeaderEpochResponsePartition::default()
                        .partition(partition.partition)
                        .error_code(code.into())
                        .leader_epoch(Some(-1))
                        .end_offset(-1),

                    Err(_) => OffsetForLeaderEpochResponsePartition::default()
                        .partition(partition.partition)
                        .error_code(ErrorCode::UnknownServerError.into())
                        .leader_epoch(Some(-1))
                        .end_offset(-1),
                };

                partitions.push(response_partition);
            }

            topics.push(
                OffsetForLeaderEpochResponseTopic::default()
                    .topic(topic_name)
                    .partitions(Some(partitions)),
            );
        }

        Ok(OffsetForLeaderEpochResponse::default()
            .throttle_time_ms(Some(0))
            .topics(Some(topics)))
```

# Produce path changes

## Modify `tansu-storage/src/service/produce.rs`

Current behavior delegates directly to `Storage::produce`.  In replicated mode:

1. Resolve partition state.
2. If local broker is not leader, return `NotLeaderOrFollower`.
3. If request has stale `current_leader_epoch`, return `FencedLeaderEpoch`.
4. Append batch as leader.
5. For `acks=0`, do not wait.
6. For `acks=1`, return after local leader append.
7. For `acks=-1` / all, wait until high watermark has reached the append’s last offset, subject to timeout.
8. If ISR size is below `min.insync.replicas`, return `NotEnoughReplicas`.
9. If append succeeds but replication does not reach ISR before timeout, return `NotEnoughReplicasAfterAppend` or `RequestTimedOut`, depending on exact Kafka version/behavior target.

Add helper:

```rust
async fn await_high_watermark<G>(
    storage: &G,
    topition: &Topition,
    leader_epoch: i32,
    target_offset: i64,
    timeout: Duration,
) -> Result<()>
where
    G: ReplicationStorage,
{
    let start = tokio::time::Instant::now();

    loop {
        let stage = storage.offset_stage(topition).await?;

        if stage.high_watermark() >= target_offset {
            return Ok(());
        }

        if start.elapsed() >= timeout {
            return Err(Error::Api(ErrorCode::RequestTimedOut));
        }

        tokio::time::sleep(Duration::from_millis(5)).await;
    }
}
```

In partition produce:

```rust
let state = ctx.state().partition_replica_state(&tp).await?;

if state.leader_id != ctx.state().node().await? {
    return self.error(partition.index, ErrorCode::NotLeaderOrFollower)
        .current_leader(Some(/* leader id/epoch */));
}

if req.acks == -1 && state.isr.len() < min_insync_replicas {
    return self.error(partition.index, ErrorCode::NotEnoughReplicas);
}

let append = ctx
    .state()
    .append_as_leader(
        transaction_id,
        &tp,
        state.leader_epoch,
        batch,
    )
    .await?;

match req.acks {
    0 | 1 => {}
    -1 => {
        await_high_watermark(
            ctx.state(),
            &tp,
            state.leader_epoch,
            append.last_offset + 1,
            Duration::from_millis(req.timeout_ms as u64),
        )
        .await?;
    }
    _ => return self.error(partition.index, ErrorCode::InvalidRequiredAcks),
}
```

Steering: Kafka offsets are “next offset” oriented in several places. If append returns `last_offset`, then the record is committed when `high_watermark >= last_offset + 1`.

# Fetch path changes

## Modify `tansu-storage/src/service/fetch.rs`

Current `FetchService` treats all fetches similarly and calls `Storage::fetch`.  Phase 11 needs to split:

```rust
let is_follower_fetch = req.replica_id.is_some_and(|id| id >= 0);
```

Pseudo-flow:

```rust
if is_follower_fetch {
    self.follower_fetch(ctx, req).await
} else {
    self.client_fetch(ctx, req).await
}
```

### Client fetch rules

For normal consumer fetch:

1. If this broker is not leader for the partition, return `NotLeaderOrFollower` and `current_leader`.
2. Return records only up to high watermark for `READ_UNCOMMITTED`.
3. Return records only up to LSO for `READ_COMMITTED`.
4. Include real `high_watermark`, `last_stable_offset`, and `log_start_offset`.

### Follower fetch rules

For follower fetch:

1. Validate `replica_id` is an assigned replica.
2. Validate `current_leader_epoch`.
3. Update follower progress based on `fetch_offset`.
4. Return records starting at `fetch_offset`, possibly beyond high watermark, because followers must replicate uncommitted leader log.
5. Return leader high watermark so follower can update local HW.
6. Recompute ISR and high watermark after progress update.

Important: follower fetch reads leader LEO; client fetch reads committed/HW boundary.

Skeleton:

```rust
async fn fetch_partition<G>(
    &self,
    ctx: &Context<G>,
    req_replica_id: Option<i32>,
    // existing args...
) -> Result<PartitionData>
where
    G: ReplicationStorage,
{
    let is_follower_fetch = req_replica_id.is_some_and(|id| id >= 0);

    if is_follower_fetch {
        self.fetch_for_follower(ctx, req_replica_id.unwrap(), topic, fetch_partition).await
    } else {
        self.fetch_for_client(ctx, topic, fetch_partition).await
    }
}
```

Follower fetch:

```rust
async fn fetch_for_follower<G>(
    &self,
    ctx: &Context<G>,
    follower_id: i32,
    topic: &str,
    fetch_partition: &FetchPartition,
) -> Result<PartitionData>
where
    G: ReplicationStorage,
{
    let tp = Topition::new(topic, fetch_partition.partition);
    let state = ctx.state().partition_replica_state(&tp).await?;

    if !state.replicas.contains(&follower_id) {
        return Ok(error_partition(fetch_partition.partition, ErrorCode::ReplicaNotAvailable));
    }

    if let Some(epoch) = fetch_partition.current_leader_epoch {
        if epoch < state.leader_epoch {
            return Ok(error_partition(fetch_partition.partition, ErrorCode::FencedLeaderEpoch));
        }

        if epoch > state.leader_epoch {
            return Ok(error_partition(fetch_partition.partition, ErrorCode::UnknownLeaderEpoch));
        }
    }

    ctx.state()
        .update_replica_progress(
            &tp,
            follower_id,
            fetch_partition.fetch_offset,
            SystemTime::now(),
        )
        .await?;

    let result = ctx
        .state()
        .fetch_as_follower(
            follower_id,
            &tp,
            state.leader_epoch,
            fetch_partition.fetch_offset,
            1,
            fetch_partition.partition_max_bytes as u32,
            Duration::from_millis(500),
        )
        .await?;

    Ok(PartitionData::default()
        .partition_index(fetch_partition.partition)
        .error_code(ErrorCode::None.into())
        .high_watermark(result.high_watermark)
        .last_stable_offset(Some(result.high_watermark))
        .log_start_offset(Some(state.log_start_offset))
        .records(Some(Frame { batches: result.records })))
}
```

Steering: update progress before and after fetch carefully. The request’s fetch offset tells you what the follower already has. The returned batches tell you what it may have after a successful append. The leader should update progress again on the next follower fetch, when the follower asks from the new LEO. This mirrors Kafka’s pull-based replication model.

# Leader epoch handling

Create durable leader epoch checkpoint file/table per partition.

Rules:

1. When a broker becomes leader for `(topic, partition)` at epoch `E`, append checkpoint:

```text
(E, current_log_end_offset)
```

2. Never rewrite old checkpoints except during truncation recovery.
3. `OffsetForLeaderEpoch(E)` returns the end offset for that epoch.
4. Followers use it to find the first divergent offset and truncate.

Example checkpoints:

```text
epoch 5 starts at offset 0
epoch 6 starts at offset 120
epoch 7 starts at offset 150
```

End offsets:

```text
end(epoch 5) = 120
end(epoch 6) = 150
end(epoch 7) = current LEO
```

Implementation:

```rust
pub fn end_offset_for_epoch(
    checkpoints: &[LeaderEpochCheckpoint],
    requested_epoch: i32,
    current_log_end_offset: i64,
) -> EpochEndOffset {
    for window in checkpoints.windows(2) {
        let current = &window[0];
        let next = &window[1];

        if current.epoch == requested_epoch {
            return EpochEndOffset {
                leader_epoch: current.epoch,
                end_offset: next.start_offset,
            };
        }
    }

    if let Some(last) = checkpoints.last() {
        if requested_epoch >= last.epoch {
            return EpochEndOffset {
                leader_epoch: last.epoch,
                end_offset: current_log_end_offset,
            };
        }
    }

    EpochEndOffset {
        leader_epoch: -1,
        end_offset: -1,
    }
}
```

Test:

```rust
#[test]
fn epoch_end_offsets_are_derived_from_next_epoch_start() {
    let checkpoints = vec![
        LeaderEpochCheckpoint { epoch: 5, start_offset: 0 },
        LeaderEpochCheckpoint { epoch: 6, start_offset: 120 },
        LeaderEpochCheckpoint { epoch: 7, start_offset: 150 },
    ];

    assert_eq!(
        EpochEndOffset { leader_epoch: 5, end_offset: 120 },
        end_offset_for_epoch(&checkpoints, 5, 200)
    );

    assert_eq!(
        EpochEndOffset { leader_epoch: 7, end_offset: 200 },
        end_offset_for_epoch(&checkpoints, 7, 200)
    );
}
```

# Leader/follower role transitions

## Become leader

When applying new partition state where local broker is leader:

1. Stop follower task for this partition.
2. Load local log.
3. Append leader epoch checkpoint using current LEO.
4. Initialize leader progress:

   * leader progress = local LEO
   * follower progress = last known fetch offsets
5. Set ISR:

   * if clean election: controller-provided ISR
   * if no controller yet: leader-only ISR until followers catch up
6. Start leader state machine.
7. Allow produce.

Pseudo-code:

```rust
async fn become_leader<S>(
    storage: &S,
    state: PartitionReplicaState,
) -> Result<()>
where
    S: ReplicationStorage,
{
    let leo = storage.log_end_offset(&state.topition).await?;

    storage
        .append_leader_epoch_checkpoint(
            &state.topition,
            state.leader_epoch,
            leo,
        )
        .await?;

    storage.apply_partition_replica_state(state).await?;

    Ok(())
}
```

## Become follower

When applying new partition state where local broker is follower:

1. Fence local produce path immediately.
2. Cancel leader waiters.
3. Stop leader state machine.
4. Start follower task.
5. Reconcile epoch with leader.
6. Truncate divergent tail.
7. Fetch from leader.

```rust
async fn become_follower<S, C>(
    storage: S,
    client: C,
    cfg: ReplicationConfig,
    topition: Topition,
    cancellation: CancellationToken,
) -> Result<()>
where
    S: ReplicationStorage + Clone + Send + Sync + 'static,
    C: ReplicationClient + Clone + Send + Sync + 'static,
{
    let task = FollowerTask {
        storage,
        client,
        cfg,
        topition,
        cancellation,
    };

    tokio::spawn(async move {
        let _ = task.run().await;
    });

    Ok(())
}
```

# Truncation on leader change

This is the most failure-prone part of Phase 11.

Scenario:

```text
broker 1 leader epoch 10
broker 2 follower
broker 3 follower

broker 1 accepts offsets 100..110 but crashes before replication
broker 2 becomes leader epoch 11 at offset 100
broker 1 restarts as follower with divergent tail 100..110
broker 1 must truncate to 100 before fetching from broker 2
```

Required algorithm:

1. Local broker loads latest local epoch and LEO.
2. Local broker sends `OffsetForLeaderEpoch` to new leader.
3. New leader returns the end offset for that epoch.
4. Local broker truncates to returned end offset if local LEO is greater.
5. Local broker deletes index entries, producer sequence state, transaction markers, and leader epoch checkpoints beyond truncated offset.
6. Local broker starts follower fetch from truncated offset.

Test:

```rust
#[tokio::test]
async fn old_leader_truncates_unreplicated_tail_after_rejoin() {
    let cluster = TestCluster::three_brokers().await;

    cluster.create_topic("orders", 1, 3).await;
    cluster.produce_to_leader("orders", 0, 0..100).await;
    cluster.wait_for_hw("orders", 0, 100).await;

    cluster.isolate_follower(2).await;
    cluster.produce_to_leader_without_full_replication("orders", 0, 100..110).await;

    cluster.kill_broker(1).await;
    cluster.elect_leader("orders", 0, 2).await;

    cluster.restart_broker(1).await;
    cluster.wait_until_follower_caught_up(1, "orders", 0).await;

    assert_eq!(100, cluster.log_end_offset(1, "orders", 0).await);
    assert_eq!(100, cluster.high_watermark("orders", 0).await);
    assert!(!cluster.contains_offsets(1, "orders", 0, 100..110).await);
}
```

Steering: never let a rejoining old leader serve produce or client fetch before role reconciliation completes. It must first learn that it is follower and truncate.

# ISR tracking

ISR is not simply “replicas that are online.” ISR means replicas that are sufficiently caught up.

Recommended rules:

```text
A replica can be in ISR if:
- it is assigned as a replica,
- it is not fenced,
- it has fetched recently,
- its LEO is >= current high watermark,
- its lag is within configured lag time/byte thresholds.
```

Add config:

```text
replica.lag.time.max.ms
replica.fetch.max.bytes
replica.fetch.wait.max.ms
min.insync.replicas
unclean.leader.election.enable
```

ISR state transitions:

```text
Follower joins ISR:
    follower LEO >= leader HW
    AND follower has fetched within lag timeout

Follower leaves ISR:
    no fetch progress for replica.lag.time.max.ms
    OR follower is fenced/offline
    OR follower log is below required committed position

Leader stays in ISR:
    always, while it is current leader
```

When ISR changes:

1. Persist ISR update.
2. Increment partition epoch.
3. Notify/wake produce waiters.
4. Update metadata response.
5. Expose through `DescribeTopicPartitions`, `Metadata`, and `ListPartitionReassignments`.

# High watermark advancement

High watermark advancement should happen after:

1. Leader append.
2. Follower fetch progress update.
3. ISR change.
4. Leader transition.
5. Recovery/restart.

Formula:

```text
HW = min(LEO of every replica in ISR)
```

But apply guardrails:

```text
new_hw >= old_hw during stable leader epoch
new_hw <= leader_leo
new_hw <= each ISR replica LEO
```

When a follower fetches, the leader should update that follower’s known LEO from the follower’s `fetch_offset`. If follower asks for offset `N`, that means it has replicated through `N - 1`.

Then:

```rust
let follower_leo = fetch_partition.fetch_offset;
storage.update_replica_progress(&tp, follower_id, follower_leo, now).await?;
leader_state_machine.maybe_advance_high_watermark(&tp).await?;
```

After returning records, do **not** assume the follower has appended them until it comes back with a higher fetch offset. This prevents advancing HW based on bytes sent but not durably stored by the follower.

# Error semantics

Add helper methods to centralize errors:

```rust
fn check_leader_epoch(
    requested: Option<i32>,
    actual: i32,
) -> Result<(), ErrorCode> {
    match requested {
        Some(epoch) if epoch < actual => Err(ErrorCode::FencedLeaderEpoch),
        Some(epoch) if epoch > actual => Err(ErrorCode::UnknownLeaderEpoch),
        _ => Ok(()),
    }
}
```

Use consistently in:

```text
Produce
Fetch
OffsetForLeaderEpoch
AlterPartition
LeaderAndIsr
```

Recommended error mapping:

| Condition                                      | Error                                               |
| ---------------------------------------------- | --------------------------------------------------- |
| Produce to follower                            | `NotLeaderOrFollower`                               |
| Fetch from non-leader as client                | `NotLeaderOrFollower`                               |
| Follower fetch from non-leader                 | `NotLeaderOrFollower`                               |
| Unknown partition                              | `UnknownTopicOrPartition`                           |
| Stale leader epoch                             | `FencedLeaderEpoch`                                 |
| Future leader epoch                            | `UnknownLeaderEpoch`                                |
| Replica not assigned                           | `ReplicaNotAvailable`                               |
| ISR below min for `acks=all`                   | `NotEnoughReplicas`                                 |
| Append succeeded but ISR replication timed out | `NotEnoughReplicasAfterAppend` or `RequestTimedOut` |
| Invalid acks                                   | `InvalidRequiredAcks`                               |
