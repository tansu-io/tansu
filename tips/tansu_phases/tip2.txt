My take: **Tansu is far enough along that the roadmap should not start with “build Kafka.”** It should start with **making Tansu’s advertised compatibility exact, then closing the semantic gaps in the order real Kafka clients discover them**. The win is to keep Tansu’s Rust/stateless/pluggable-storage architecture, but make every externally visible behavior look like Kafka.

Tansu already has a serious base: the repo describes it as a “stateless Kafka-compatible broker with pluggable storage,” with PostgreSQL, SQLite/libSQL, S3, and memory storage engines; it also includes broker, topic CLI, cat CLI, and proxy in a single statically linked binary. ([GitHub][1]) The docs also call it a drop-in Kafka replacement with PostgreSQL, SQLite, S3, or memory storage, written in safe async Rust. ([Tansu][2]) The protocol layer is not toy-level either: `tansu-sans-io` is explicitly a Kafka protocol implementation that operates on bytes, generated from Kafka message descriptors and backed by generated message metadata used by the encoder/decoder. 

The central gap is this: **Tansu has broad protocol knowledge, but the routed broker service surface and the production semantics are narrower than full Kafka.** The broker routes a concrete set of storage APIs such as Produce, Fetch, Metadata, CreateTopics, DeleteTopics, DescribeConfigs, ListOffsets, FindCoordinator, InitProducerId, some transaction APIs, ACL/SCRAM description/alteration, and telemetry subscription.  It separately routes classic group coordinator APIs such as JoinGroup, SyncGroup, Heartbeat, LeaveGroup, OffsetCommit, and OffsetFetch.  SASL handshake/authenticate are present, and the auth layer supports PLAIN/SCRAM-style flows backed by storage.   That is an excellent base, but not yet “replace every Kafka backend stack overnight.”

I would run this as a **16-stage program**.

## The critical ordering

The first three stages are non-negotiable. **Do not add more features before ApiVersions truthfulness and the differential lab.** Kafka clients are negotiation-heavy; if Tansu advertises the wrong API/version/feature, clients will take paths that fail later in confusing ways. The current `ApiVersionsService` design is clean, but it needs a semantic support matrix layered on top of descriptor metadata. 

Stages 4–10 are the real Kafka core: routed API surface, storage/log semantics, produce, fetch, admin, groups, and transactions. This is where Tansu becomes a backend-stack Kafka replacement rather than a strong Kafka-compatible broker for simpler workloads.

Stages 11–14 close the production and modern-client gaps: compaction, security, multi-broker metadata, and Kafka 4.2-era APIs. Stages 15–16 are what make it credible to users: ecosystem certification, performance proof, and migration tooling.

## The biggest blockers I would track as “drop-in release blockers”

The top blocker is **semantic API coverage**, not Rust code volume. Tansu’s protocol generator and codec are a strength, but the broker must route and semantically implement the APIs that clients actually negotiate.

The second blocker is **storage-engine parity**. Tansu’s pluggable storage model is exciting because it avoids Kafka’s broker-local replicated log, but every storage engine must still provide Kafka’s externally visible ordering, offset, high-watermark, compaction, retention, and transaction guarantees.

The third blocker is **consumer groups under real-world churn**. Tansu has a real group coordinator today, but the bar for drop-in parity is brutal: rolling deploys, static membership, cooperative rebalancing, lost heartbeats, delayed joins, offset commits during rebalance, and modern ConsumerGroupHeartbeat behavior.

The fourth blocker is **exactly-once semantics**. Idempotent-producer groundwork exists, but “Kafka replacement overnight” requires end-to-end transactional producer, transactional offset commits, read_committed fetches, abort visibility, fencing, recovery, and Kafka Streams EOS.

The fifth blocker is **ecosystem certification**. A backend stack does not just use Produce and Fetch. It uses AdminClient, Connect, Streams, MirrorMaker, CLI tools, Schema Registry conventions, security configs, metrics, and operational runbooks.

## What I would explicitly *not* do

I would **not** rebuild Kafka’s internals just to imitate Kafka. Tansu’s differentiated architecture is stateless brokers plus pluggable storage. Keep that. The goal is not internal mimicry; the goal is **externally observable Kafka behavior**.

I would also **not** let schema registry, Iceberg, Delta, lake sinks, or proxy features outrun Kafka-core parity. Those are excellent differentiators, but the path to “drop-in replacement” is first: clients, protocol negotiation, admin, storage semantics, groups, transactions, security, operations.

The short version of the roadmap is: **truthful compatibility → differential tests → complete routed API surface → storage/log invariants → produce/fetch/groups/transactions → admin/security/controller façade → modern Kafka 4.2 features → ecosystem certification → performance/ops/migration.**
