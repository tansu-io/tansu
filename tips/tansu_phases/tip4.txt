I’d turn the scratch-build plan into **27 Tansu-specific stages**. Tansu is not starting at milestone 1; it already has a Rust Kafka protocol stack, broker/proxy binaries, storage engines, service routing, SASL plumbing, a storage abstraction, tests around produce/fetch/groups/transactions, and promising baseline performance. The hard part is no longer “can we speak Kafka?” It is: **can we make every important Kafka client, CLI, framework, and operational workflow behave as though Kafka is still there?**

The big strategic point: **Tansu’s internals are intentionally not Kafka’s internals.** Its docs say all brokers are stateless/equal, there is no ISR, every broker can act as leader/coordinator, and state changes are persisted optimistically through the storage engine rather than through Kafka’s broker replication/controller model. That can absolutely be a strength, but it means the roadmap must focus on **externally visible Kafka semantics**, not copying Kafka’s JVM architecture. ([Tansu][1])

## Where Tansu already is

Tansu already describes itself as a stateless Kafka-compatible broker with PostgreSQL, SQLite/libSQL, S3, and memory storage engines, plus schema-backed topics that can write to Iceberg or Delta Lake. ([GitHub][2]) The repo is very Rust-heavy, with its latest GitHub release shown as `v0.6.0` on March 13, 2026, and GitHub reports Rust as 99.4% of the codebase. ([GitHub][2])

The existing crate layout is already close to the right shape for a serious Kafka replacement: `tansu-sans-io` for protocol, `tansu-service` for routing/layers, `tansu-broker`, `tansu-storage`, `tansu-auth`, `tansu-proxy`, `tansu-schema`, `tansu-otel`, `tansu-perf`, CLI crates, and more. ([GitHub][2]) The protocol implementation is generated from Kafka JSON protocol definitions into Rust types, and Tansu’s service layer deserializes Kafka frames, routes them, and serializes responses back through composable services. ([Tansu][3])

The broker already routes a meaningful set of Kafka APIs: produce/fetch/list-offsets/metadata, topic admin, configs, ACL create/describe, SCRAM credential APIs, init-producer-id, transaction-adjacent APIs, group APIs, consumer group describe, telemetry subscription, and topic-partition description routes. ([GitHub][4]) It also has coordinator routes for JoinGroup, SyncGroup, Heartbeat, LeaveGroup, OffsetCommit, and OffsetFetch. ([GitHub][5]) SASL handshake/authenticate services are wired into broker routing, and the auth crate handles PLAIN/SCRAM-style flows through `rsasl`. ([GitHub][6])

The storage abstraction is broad: the `Storage` trait includes broker registration, topic creation/deletion, produce/fetch, offsets, metadata, SCRAM credentials, group updates, producer initialization, transaction operations, maintenance, cluster/node/listener lookup, and more. ([GitHub][7]) PostgreSQL storage already documents batch insertion through PostgreSQL `COPY`, delete retention, compact retention, per-topic-partition table partitioning, cascading deletes, prepared-statement caching, and multiple brokers sharing one PostgreSQL database. ([Tansu][8]) SQLite supports delete/compact cleanup policies and is positioned as a single-broker engine, while S3 supports multiple brokers sharing a bucket through conditional writes. ([Tansu][9])

Performance is not a toy baseline either. Tansu’s own performance writeup reports a null-storage protocol/broker benchmark around 600k–650k records/sec, roughly 585–642 MB/sec, with sub-millisecond average latency in that null-storage setup and 13 MB RSS under load. That is not durable-storage parity yet, but it is exactly the kind of low-overhead foundation you want before pushing real storage hot paths. ([Tansu][10])

## The current gap, stated bluntly

Tansu is already a **credible Kafka-compatible Rust broker platform**. It is not yet defensibly a **100% overnight Kafka replacement**.

The reason is that Kafka compatibility is much larger than Produce/Fetch/Metadata. The current Apache Kafka 4.2 protocol docs list many APIs beyond the classic producer/consumer path: ACL deletion, log-dir APIs, partition reassignment, quota APIs, transactions inspection, new consumer group APIs, share group APIs, streams group APIs, Raft voter APIs, and more. ([Apache Kafka][11]) Some of these are essential for applications, some for admin tooling, some for Connect/Streams, and some only for Kafka’s own internals. A “drop-in replacement” has to decide which of those are implemented, emulated, rejected compatibly, or intentionally profiled out.

So the plan should not be “implement Kafka from scratch.” The plan should be:

> **Turn Tansu into a compatibility-certified Kafka product, using its stateless Rust/storage-engine architecture, while making every externally visible Kafka behavior boringly indistinguishable from Kafka for real backend stacks.**

## The 27-stage Tansu-specific roadmap

### Phase 1 — Define the compatibility target and make gaps measurable

| Stage | Work                                                                                                                                                                                                                                                                                                      | Exit gate                                                                                                           |
| ----: | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
|     1 | **Create the Tansu Kafka Compatibility Contract.** Pick target lines: for example Kafka 3.9 LTS-ish clients, Kafka 4.1/4.2 protocol, Java client, librdkafka, KafkaJS, Sarama, kafka-python, Kafka CLI tools, Kafka Connect, MirrorMaker 2, and common stream processors.                                 | A public matrix exists: API key/version, client library, feature, expected result, and storage-engine profile.      |
|     2 | **Split “100%” into product profiles.** Suggested profiles: `dev-single` SQLite, `prod-postgres`, `prod-s3-object`, `proxy`, and `lake-sink`. Do not let SQLite single-broker limitations block production parity claims for PostgreSQL/S3, and do not let lake-sink behavior confuse Kafka-log behavior. | Every feature is marked supported/unsupported/emulated per profile.                                                 |
|     3 | **Build a Kafka-in-the-loop differential test harness.** For every API, run the same client operation against Apache Kafka and Tansu, compare response shape, errors, metadata, offsets, group states, retry behavior, timing boundaries, and disconnect behavior.                                        | CI can answer: “Tansu behaves the same as Kafka for this client/version/test.”                                      |
|     4 | **Generate the API coverage dashboard from code.** Use `tansu-sans-io` metadata plus broker route registration to auto-produce a supported/missing/partial table. This phase comes first because it prevents the team from confusing “implemented route” with “Kafka-compatible behavior.” That distinction matters a lot.

|     6 | **ApiVersions correctness.** Kafka clients use ApiVersions negotiation to choose request versions; Kafka’s docs explicitly describe this compatibility mechanism and how unsupported versions should be handled. |     7 | **Frame, connection, and pipelining semantics.** Kafka’s protocol docs state TCP framing, request ordering, max request size disconnects, and response ordering expectations. ([Apache Kafka][11]) Tansu’s `BytesFrameService` is the right hook; harden it.                     | Fuzzed malformed frames, oversized requests, disconnects, idle timeouts, SASL pre-auth behavior, and pipelined requests match Kafka-compatible expectations.             |

|    11 | **Controller/admin identity emulation.** Kafka tools expect a controller, broker registrations, cluster ID, node IDs, listener metadata, and sometimes KRaft-ish details. ## The critical ordering

The dangerous temptation is to jump straight to “implement missing APIs.” I would not do that first. The right order is:

1. **Compatibility contract and differential harness.** Otherwise, you will ship many APIs that are technically implemented but semantically wrong.
2. **Wire/protocol/server behavior.** Clients fail early if ApiVersions, SASL, frame handling, or error responses are off.
3. **Metadata projection.** Tansu’s stateless architecture is a differentiator, but clients and tools need Kafka-shaped metadata.
4. **Produce/fetch/offsets.** This is the money path.
5. **Groups.** Most backend stacks use consumer groups; broken rebalance behavior kills confidence fast.
6. **Idempotence/transactions.** Required for serious Kafka Streams, Connect, and exactly-once workloads.
7. **Security/admin/quotas.** Required for enterprise replacement.
8. **Ecosystem certification.** The “drop-in” claim is won here, not in unit tests.
9. **Chaos/perf/ops/migration.** This is what lets someone replace Kafka in production rather than in a demo.

## The biggest Tansu-specific engineering decisions

First, decide whether **PostgreSQL or S3 is the flagship production durability profile**. SQLite is brilliant for dev/small deployments, but Tansu’s own docs position SQLite as single-broker, while PostgreSQL and S3 support multiple brokers sharing storage. ([Tansu][8]) The “Kafka replacement overnight” story should probably lead with PostgreSQL for transactional consistency and operational familiarity, then S3/object storage for cost/tiered-style workloads.

Second, make **replication factor semantics** boring. Kafka users will create topics with replication factor 3. Tansu can either emulate that through external storage durability profiles or reject/normalize it. But it must not surprise Terraform, Helm charts, Kafka Connect, or AdminClient workflows.

Third, treat **transactions as a first-class gate**, not a late nice-to-have. Tansu already has transaction tests in the broker test tree and transaction-related storage/protocol pieces. ([GitHub][15]) But Kafka’s EOS semantics are where most replacements get exposed.

Fourth, make **compatibility observable**. Every release should publish a matrix: Kafka client versions, protocol APIs, admin tools, Connect, Streams, MirrorMaker, Schema Registry, storage profiles, and known caveats. That matrix is what lets a team say, “yes, we can swap this in tonight.”

## My recommended milestone naming

I would package the journey as four public releases:

**Tansu Compatibility Preview**
Stages 1–8. The pitch: “we know exactly what works, and wire compatibility is certified.”

**Tansu Core Kafka Replacement Beta**
Stages 9–18. The pitch: “normal producers, consumers, topics, offsets, retention, compaction, and consumer groups are production-testable.”

**Tansu EOS/Admin/Security Beta**
Stages 19–25. The pitch: “Kafka Streams, Connect, idempotence, transactions, ACLs, SCRAM/TLS, quotas, and admin workflows are ready for serious migrations.”

**Tansu Drop-In GA**
Stages 26–27. The pitch: “certified against real clients and tools, chaos-tested, benchmarked, packaged, documented, and migration-ready.”

Bottom line: **Tansu already has the right bones.** The route now is not a generic 18-step Kafka rebuild. It is a rigorous compatibility-certification program plus targeted implementation of the missing externally visible Kafka semantics. Do that, and the anti-JVM story becomes much stronger: not “Rust Kafka-ish broker,” but **a Rust Kafka replacement with a public proof matrix.**
